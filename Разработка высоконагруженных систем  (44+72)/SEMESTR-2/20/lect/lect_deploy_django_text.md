## Слайд 1: Django в production — другая реальность

Добрый день. Сегодня мы поговорим о том, что происходит с вашим Django-проектом, когда он покидает уютный localhost и выходит в большой страшный интернет.

Когда вы пишете `python manage.py runserver`, Django поднимает однопоточный сервер, который сами авторы фреймворка называют «для разработки». Это как кататься на самокате по двору — весело, удобно, но выезжать на нём на трассу вы не станете. На продакшене `runserver` — это картонный болид на Формуле-1. Он не умеет обрабатывать параллельные запросы, не раздаёт статику эффективно и падает под любой осмысленной нагрузкой. Десять одновременных пользователей — и он начинает задыхаться.

Production — это совсем другой стек технологий. Вместо `runserver` — WSGI-сервер Gunicorn или ASGI-сервер Uvicorn, которые умеют запускать несколько процессов и обрабатывать десятки запросов параллельно. Перед ними — reverse proxy Nginx, который принимает HTTP-запросы из интернета, раздаёт статические файлы, терминирует SSL-шифрование. База данных — уже не SQLite-файлик, а PostgreSQL в отдельном контейнере. Секреты — не в `settings.py`, а в переменных окружения. И CI/CD пайплайн, который деплоит новую версию за 30 секунд после `git push`, без ручного SSH на сервер.

Зачем всё это? Потому что код — это ещё не продукт. Код, который работает только у вас на ноутбуке, не имеет никакой ценности для пользователей. Ценность появляется, когда пользователь открывает браузер, вводит URL — и получает ответ. Быстро, стабильно, безопасно. Вот об этом мы сегодня.

В этой лекции мы пройдём весь путь: от голого Django до оптимизированного продакшена с реальными бенчмарками. Будет теория, будут цифры, будут ошибки — и будет понимание, почему всё работает именно так. Поехали.

## Слайд 2: Архитектура production Django

Давайте сразу посмотрим на архитектуру. В продакшене Django никогда не стоит один. Это как оркестр: каждый инструмент играет свою партию, и без дирижёра — хаос.

Классическая архитектура — четыре слоя. Первый — браузер пользователя делает HTTP-запрос. Второй — Nginx на порту 80 или 443 принимает этот запрос. Если это статический файл — CSS, JavaScript, картинка — Nginx отдаёт его сам, мгновенно, без участия Python. Если это динамический запрос — проксирует его дальше, на третий слой: Gunicorn на порту 8000, который запускает Python-код Django. Django обрабатывает бизнес-логику, лезет в четвёртый слой — PostgreSQL — за данными, формирует ответ и возвращает его обратно по цепочке.

Почему нельзя просто поставить Gunicorn и открыть его в интернет? Потому что Gunicorn — узкий специалист. Он умеет быстро обрабатывать Python-код. Но он не умеет раздавать файлы эффективно — для каждого файла он запустит Python, это в сто раз медленнее, чем Nginx, который работает с файлами на уровне операционной системы. Gunicorn не понимает SSL — не умеет расшифровывать HTTPS. Он не защищает от slowloris-атак — когда злоумышленник открывает тысячу медленных соединений и держит их, съедая все воркеры.

Nginx — это «охранник на входе». Он быстрый, легковесный, обрабатывает тысячи соединений в одном процессе благодаря event-driven архитектуре. Gunicorn — «специалист внутри», который занимается только бизнес-логикой. Вместе они — мощная пара.

Запомните эту цепочку: Браузер → Nginx → Gunicorn → Django → PostgreSQL. Вы будете встречать её в 90% Django-проектов в production.

## Слайд 3: Dev vs Production — таблица различий

Теперь давайте зафиксируем разницу между разработкой и продакшеном. Это важно, потому что многие начинающие разработчики думают: «Ну, у меня же работает на localhost, значит, можно деплоить». Нет, нельзя. Вот почему.

Сервер. В dev — `runserver`. В prod — Gunicorn или Uvicorn. `runserver` — однопоточный, без параллелизма, без воркеров. Gunicorn запускает несколько процессов, каждый обрабатывает запросы независимо.

База данных. В dev — SQLite, файлик на диске. В prod — PostgreSQL или MySQL, полноценная СУБД с транзакциями, индексами, пулом соединений.

Статика. В dev — Django сам раздаёт CSS и JavaScript через `django.contrib.staticfiles`. В prod — это делает Nginx, потому что он раздаёт файлы в сто раз быстрее.

DEBUG. Вот здесь самое опасное. Когда `DEBUG=True`, Django при ошибке показывает полный traceback с переменными. Включая пароли от базы данных, секретные ключи, содержимое запросов. На проде это как оставить ключи от квартиры в замке с надписью «добро пожаловать». Любой может зайти на несуществующий URL, получить 404 с полным дампом настроек — и всё, ваша база данных уже чья-то.

`ALLOWED_HOSTS`. На dev вы ставите `*` — принимать запросы с любым доменом. На prod — только конкретный домен. Это защита от Host header attacks: злоумышленник подставляет свой домен в заголовок Host, и Django генерирует ссылки с этим доменом — например, в письмах сброса пароля.

HTTPS, логи, перезапуск — всё отличается. На dev — ручной запуск, консольные логи, никакого шифрования. На prod — автоматический перезапуск через systemd, логи в файлах, обязательный HTTPS через Let's Encrypt.

## Слайд 4: settings.py для production

Посмотрим, как выглядит settings.py для прода. Главное правило: никаких секретов в коде. Всё читается из переменных окружения.

`SECRET_KEY` — ключ шифрования сессий и CSRF-токенов. В dev вы его хардкодите в settings.py. В prod он лежит в переменной окружения `SECRET_KEY`, и если кто-то получит доступ к вашему репозиторию на GitHub — ключ не утечёт. Потому что в коде только `os.environ.get('SECRET_KEY', 'change-me')`.

`DEBUG` — строка из переменной окружения, сравнивается с `'True'`. По умолчанию — `False`. Это значит: если вы забудете выставить переменную — debug будет выключен. Безопасный дефолт. Это важный принцип: дефолтные значения должны быть безопасными.

`ALLOWED_HOSTS` — список доменов через запятую. `os.environ.get('ALLOWED_HOSTS', '').split(',')`. Если переменная не задана — пустой список, Django откажет всем запросам. Опять безопасный дефолт.

`dj_database_url` — библиотека, которая парсит строку подключения `DATABASE_URL` в формат Django. Одна переменная `postgresql://postgres:postgres@localhost:5432/shop` вместо пяти отдельных параметров: engine, name, user, password, host. Компактно и стандартно.

`conn_max_age=600` — не закрывать соединение к базе данных 10 минут. По умолчанию Django открывает новое соединение на каждый запрос и закрывает после ответа. Это TCP handshake + аутентификация — лишние 5-10 миллисекунд на каждый запрос. При 200 запросах в секунду — это уже ощутимо. `conn_max_age=600` говорит: «Переиспользуй соединение, не закрывай его 10 минут». Но с async-сервером это бомба замедленного действия — поговорим об этом через несколько слайдов.

`STATIC_ROOT` — папка, куда `collectstatic` соберёт все статические файлы из всех приложений. Nginx будет раздавать их напрямую, минуя Django.

## Слайд 5: Порядок деплоя Django (ручной)

Ручной деплой — это чек-лист из восьми шагов. Запомните последовательность, она одинаковая для девяноста процентов проектов.

Шаг первый — SSH на сервер. `ssh root@81.90.182.174`. Вы подключаетесь к удалённой машине по защищённому каналу. Если это первый раз — вас спросят про fingerprint, говорите yes.

Шаг второй — установка зависимостей. Python, pip, virtualenv, git. На чистом Ubuntu это `apt install python3 python3-pip python3-venv git`.

Шаг третий — клонирование репозитория. `git clone` в `/opt/project`. Почему `/opt`? Это стандартная директория для дополнительного ПО в Linux. Не `/home`, не `/var` — именно `/opt`.

Шаг четвёртый — виртуальное окружение. `python3 -m venv venv && pip install -r requirements.txt`. Изоляция зависимостей. На одном сервере может быть несколько проектов с разными версиями библиотек.

Шаг пятый — переменные окружения. Создаёте `.env` файл или экспортируете переменные. `SECRET_KEY`, `DATABASE_URL`, `DEBUG=False`, `ALLOWED_HOSTS`.

Шаг шестой — миграции. `python manage.py migrate`. Создаёт таблицы в базе данных. Если забудете — получите 500 на первом же запросе.

Шаг седьмой — статика. `python manage.py collectstatic`. Собирает CSS, JS, картинки в одну папку для Nginx.

Шаг восьмой — запуск. Через systemd или Docker.

Каждый из этих шагов можно автоматизировать. Но логика не меняется. И если что-то не работает — идите по чек-листу сверху вниз. В восьмидесяти процентах случаев проблема в забытом шаге. Обычно — в миграциях или в переменных окружения.

## Слайд 6: Systemd — автозапуск без Docker

Если вы не используете Docker, Django нужно запускать через systemd — это менеджер служб Linux. Без него вы закроете SSH-сессию — и приложение умрёт. Просто потому что процесс привязан к терминалу. Нет терминала — нет процесса.

Systemd решает эту проблему. Вы описываете «сервис» — unit-файл с расширением `.service`. Давайте разберём его по строкам.

Секция `[Unit]`. `Description=Django Shop API` — человекочитаемое описание. `After=network.target` — запускать после того, как поднимется сеть. Логично: Django нужна сеть, чтобы подключиться к базе и принимать запросы.

Секция `[Service]`. `User=deploy` — запускать от имени пользователя deploy, не root. Принцип минимальных привилегий. Если приложение взломают — злоумышленник получит права deploy, а не root. `WorkingDirectory` — рабочая директория. `ExecStart` — команда запуска: gunicorn с привязкой к `127.0.0.1:8000` и четырьмя воркерами. Почему `127.0.0.1`, а не `0.0.0.0`? Потому что снаружи — только через Nginx. Gunicorn не должен торчать в интернет напрямую.

`Restart=always` — при падении перезапускать автоматически. `RestartSec=5` — с паузой пять секунд, чтобы не уйти в бесконечный цикл перезапусков, если приложение падает сразу при старте.

Секция `[Install]`. `WantedBy=multi-user.target` — включить сервис при загрузке системы.

Команды, которые вы будете использовать каждый день: `systemctl start shop` — запустить, `systemctl enable shop` — автозапуск при перезагрузке, `systemctl status shop` — проверить статус, `journalctl -u shop -f` — логи в реальном времени. Последняя команда — ваш лучший друг при дебаге.

## Слайд 7: Порядок деплоя с CI/CD

Ручной деплой — это хорошо для понимания. Вы должны уметь задеплоить руками. Но делать это каждый раз при каждом коммите — это как мыть посуду в ресторане без посудомойки. Можно, но зачем, если есть автоматизация?

CI/CD — Continuous Integration / Continuous Deployment. Непрерывная интеграция и непрерывный деплой. Идея простая: вы пушите код в master — и через тридцать секунд новая версия уже на сервере. Без SSH-подключения, без ручных команд, без человеческого фактора.

Как это работает? `git push` → GitHub получает код → запускается GitHub Actions — это встроенная CI/CD система GitHub → Action подключается к вашему VPS по SSH → выполняет `git pull` и `docker compose up` → готово, новая версия задеплоена.

Что для этого нужно? Три вещи. Первое — GitHub Secrets. Это зашифрованные переменные: IP сервера, логин, пароль. GitHub хранит их в зашифрованном виде и не показывает в логах. Даже если кто-то форкнет ваш репозиторий — секреты не скопируются.

Второе — workflow файл. Это YAML-файл `.github/workflows/deploy.yml`, который описывает, что делать при определённом событии. В нашем случае — при push в master.

Третье — SSH-доступ. GitHub Actions нужно уметь подключаться к вашему VPS. Для этого мы используем готовый action `appleboy/ssh-action`.

Немного сленга. «Пайплайн» — это цепочка действий CI/CD. «Воркфлоу» — файл с описанием пайплайна. «Секреты» — зашифрованные переменные в GitHub. «Триггер» — событие, которое запускает пайплайн: push, pull request, расписание, ручной запуск.

## Слайд 8: GitHub Actions — deploy.yml

Давайте разберём файл деплоя по строкам.

`name: Deploy to VPS` — человекочитаемое имя. Будет видно во вкладке Actions на GitHub.

`on: push: branches: [master]` — триггер. Запускается только при push в ветку master. Не при создании pull request, не при комментарии, не при создании issue — только при мерже в master. Это важно: вы не хотите деплоить каждый коммит в фиче-ветку.

`jobs: deploy: runs-on: ubuntu-latest` — задача будет выполняться на виртуальной машине с Ubuntu, которую GitHub предоставляет бесплатно. Каждый запуск — чистая машина, без остатков от предыдущих запусков.

`steps` — шаги задачи. У нас один шаг: `Deploy via SSH`. Используем action `appleboy/ssh-action@v1` — это готовый action от разработчика appleboy, который умеет подключаться по SSH и выполнять команды.

Параметры: `host`, `username`, `password` — берутся из секретов через синтаксис `${{ secrets.VPS_HOST }}`. Фигурные скобки внутри фигурных — это шаблонизатор GitHub Actions.

`script` — что выполнить на сервере. Три команды: `cd /opt/shop` — перейти в директорию проекта. `git pull origin master` — подтянуть последние изменения. `docker compose -f docker-compose.prod.yml up --build -d` — пересобрать и перезапустить контейнеры в фоне.

Этот подход называется «серверная сборка»: код собирается прямо на VPS. Для продакшена крупных проектов лучше собирать Docker-образ на CI, пушить в Docker Registry и на VPS только делать pull. Но для учебных и небольших проектов — серверная сборка более чем достаточна.

## Слайд 9: GitHub Secrets — безопасность CI/CD

Секреты — это фундамент безопасности CI/CD. Без них вам пришлось бы хранить пароли от серверов прямо в коде. А код хранится в Git, Git хранится на GitHub, и даже в приватном репозитории — это плохая идея. Помните: Git помнит всё. Даже если вы удалите файл с паролем — он останется в истории коммитов навсегда.

GitHub Secrets шифруются и хранятся отдельно от кода. В логах Actions они заменяются на три звёздочки. Даже если кто-то форкнет ваш репозиторий — секреты не скопируются. Даже collaborator с правом чтения не увидит значения секретов.

Настройка: Settings → Secrets and variables → Actions → New repository secret. Три секрета: `VPS_HOST` — IP-адрес сервера, `VPS_USER` — имя SSH-пользователя, `VPS_PASSWORD` — пароль.

Но будьте внимательны. Если написать `echo ${{ secrets.VPS_PASSWORD }}`, GitHub замаскирует вывод. А если `echo ${{ secrets.VPS_PASSWORD }} | base64`, то в логах появится base64-закодированная версия — и GitHub может её НЕ замаскировать, потому что не распознает. Декодировать base64 — дело двух секунд. Поэтому будьте аккуратны с трансформациями секретов в пайплайнах.

Продвинутый подход — SSH-ключи вместо пароля. Генерируете пару ключей, публичный кладёте на сервер в `~/.ssh/authorized_keys`, приватный — в секрет GitHub. Это безопаснее и стандартнее для продакшена. Но для учебных целей пароль проще и нагляднее.

## Слайд 10: Docker при деплое — зачем?

Можно деплоить без Docker. Virtualenv, systemd, Nginx — классический стек. Работает. Но есть проблемы.

Проблема номер один: «У меня Python 3.12, а на сервере 3.9». Вы используете фичи Python 3.12 — match-case, например. А на сервере Ubuntu 20.04 с Python 3.8. Приехали.

Проблема номер два: «Я поставил libpq-dev, а оно конфликтует с другим проектом на этом же сервере». На одном сервере два проекта, оба хотят разные версии системных библиотек. Добро пожаловать в dependency hell.

Проблема номер три — классика жанра: «На моём маке всё работало!». macOS, Linux, Windows — разные операционные системы, разные версии библиотек, разное поведение. Код, который работает на маке, может не работать на Ubuntu. И наоборот.

Docker решает все три проблемы одним ударом. Вы описываете окружение в Dockerfile: какая ОС, какой Python, какие системные пакеты. Собираете образ. И он одинаково работает везде — на маке, на линуксе, на VPS, на Kubernetes.

Для Django-проекта Docker даёт четыре вещи. Изоляция — PostgreSQL, Django, Redis живут в отдельных контейнерах, не мешают друг другу. Воспроизводимость — `docker compose up` и всё работает, у любого разработчика, на любой машине. Масштабирование — легко добавить воркеры или запустить несколько реплик. Откат — если новая версия сломала что-то, вернуться к предыдущему образу — дело одной команды.

На сленге: «контейнер» — изолированный процесс с собственной файловой системой. «Образ» — шаблон для создания контейнера, как класс и экземпляр класса. «Билдить» — собирать образ. «Поднять контейнер» — запустить.

## Слайд 11: Dockerfile для Django

Давайте разберём Dockerfile для Django-проекта построчно. Каждая строка — это инструкция, и порядок имеет значение.

`FROM python:3.14-slim`. Базовый образ. `slim` — минимальный вариант: Python + минимум системных утилит. Весит примерно 150 мегабайт. Для сравнения, полный образ `python:3.14` — около гигабайта. В нём компиляторы, документация, инструменты разработки — всё, что в продакшене не нужно.

`WORKDIR /app`. Рабочая директория внутри контейнера. Все последующие команды выполняются в `/app`. Аналог `cd /app`, но если директории нет — создастся автоматически.

`RUN apt-get update && apt-get install -y --no-install-recommends libpq-dev && rm -rf /var/lib/apt/lists/*`. Эта строка устанавливает системную библиотеку `libpq-dev`, которая нужна для `psycopg2` — драйвера PostgreSQL для Python. Без неё `pip install psycopg2` упадёт с ошибкой компиляции. `--no-install-recommends` — не ставить рекомендуемые пакеты, только обязательные. `rm -rf /var/lib/apt/lists/*` — удалить кэш apt после установки. Это уменьшает размер образа.

Дальше — ключевой трюк. `COPY requirements.txt .` — копируем ТОЛЬКО файл зависимостей. `RUN pip install --no-cache-dir -r requirements.txt` — устанавливаем зависимости. Почему отдельно? Docker кэширует слои. Если вы изменили код, но зависимости не трогали — Docker не будет заново устанавливать пакеты, возьмёт из кэша. Это экономит минуты при каждой сборке. Если бы мы сначала скопировали весь код (`COPY . .`) и потом делали `pip install` — любое изменение в любом файле инвалидировало бы кэш, и pip install запускался бы заново.

`COPY . .` — теперь копируем весь код.

`RUN python manage.py collectstatic --noinput 2>/dev/null || true`. Собираем статику. `--noinput` — не спрашивать подтверждение. `2>/dev/null` — подавить предупреждения. `|| true` — не падать, если что-то пошло не так. На этапе сборки образа базы данных ещё нет, и некоторые приложения могут ругаться.

`EXPOSE 8000` — документация для человека: контейнер слушает порт 8000. Фактически порт не открывается — это делает `docker compose`.

`CMD` — команда по умолчанию при запуске контейнера. Gunicorn с тремя воркерами на порту 8000.

## Слайд 12: docker-compose.prod.yml

Docker Compose позволяет описать несколько сервисов в одном файле и запустить их одной командой. Давайте разберём наш production-файл.

Два сервиса: `db` и `web`. Docker автоматически создаёт сеть между ними — не нужно вручную настраивать networking. Сервис `db` доступен из `web` по имени `db`. Docker поднимает DNS внутри сети: имя сервиса → IP контейнера.

Сервис `db` — PostgreSQL 17. `image: postgres:17` — берём готовый образ из Docker Hub. `restart: unless-stopped` — перезапускать при падении и после перезагрузки сервера. Единственное исключение — если вы явно остановили контейнер командой `docker stop`.

Переменные окружения: `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD`. Образ PostgreSQL при первом запуске читает эти переменные и создаёт базу данных и пользователя. `${DB_PASSWORD:-postgres}` — синтаксис Docker Compose: взять переменную окружения `DB_PASSWORD`, если не задана — использовать `postgres` как дефолт.

`command: postgres -c max_connections=300` — переопределяем команду запуска PostgreSQL, увеличивая лимит соединений с дефолтных 100 до 300. Зачем — поймёте, когда будем разбирать бенчмарки.

`volumes: pg_data:/var/lib/postgresql/data` — именованный volume. Данные базы сохраняются на диске хоста и переживают пересоздание контейнера.

Сервис `web` — наше Django-приложение. `build: .` — собрать образ из Dockerfile в текущей директории. `ports: "8080:8000"` — порт 8080 на хосте проксируется на порт 8000 в контейнере. `depends_on: db` — запускать web после db.

`command` — переопределяем CMD из Dockerfile. Сначала `migrate` — создать таблицы, потом `load_data.py` — загрузить тестовые данные, потом Gunicorn с восемью воркерами и четырьмя потоками.

## Слайд 13: Разбор docker-compose по строкам

Остановимся подробнее на нюансах docker-compose.

`restart: unless-stopped`. Есть четыре политики перезапуска: `no` — не перезапускать, `always` — всегда, `on-failure` — только при ошибке, `unless-stopped` — всегда, кроме случая ручной остановки. Для прода — `unless-stopped` оптимален. При перезагрузке сервера контейнеры поднимутся автоматически.

`${DB_PASSWORD:-postgres}`. Синтаксис `:-` означает: если переменная не задана или пуста — использовать дефолт. Есть ещё `-` без двоеточия — только если не задана. Разница тонкая, но есть.

`max_connections=300`. По умолчанию PostgreSQL разрешает 100 одновременных соединений. Когда у вас 8 воркеров Gunicorn по 4 потока — это 32 соединения. Плюс соединения для миграций, мониторинга, бэкапов. 100 может быть мало. 300 — с запасом.

Именованный volume `pg_data`. Контейнеры эфемерны: `docker compose down` — и всё содержимое контейнера исчезает. Но данные базы должны жить. Volume хранится на диске хоста. `docker compose down` — volume жив. `docker compose down -v` — volume удаляется. Осторожно с флагом `-v`!

`depends_on: db`. Важный нюанс: `depends_on` гарантирует порядок СТАРТА контейнеров, но НЕ ждёт, пока PostgreSQL будет готов принимать соединения. PostgreSQL может стартовать за 2-3 секунды, а Django попытается подключиться через 100 миллисекунд. Получит ошибку, контейнер упадёт, `restart: unless-stopped` перезапустит его, и со второй попытки всё заработает. Это нормально. Для более элегантного решения есть `healthcheck` и `depends_on: condition: service_healthy`, но для учебного проекта — избыточно.

## Слайд 14: Docker volumes — где живут данные

Давайте подробнее про volumes. Это одна из ключевых концепций Docker, которую часто недопонимают.

Контейнер — это процесс с изолированной файловой системой. Когда контейнер удаляется — файловая система удаляется вместе с ним. Для кода это нормально: код всегда можно пересобрать из Git. Но данные базы — это другое. Вы же не хотите терять заказы пользователей при каждом обновлении приложения.

Volumes — это механизм монтирования внешнего хранилища в контейнер. Три типа.

Именованные volumes. `pg_data:/var/lib/postgresql/data`. Docker создаёт директорию в `/var/lib/docker/volumes/` на хосте и монтирует её внутрь контейнера. Управление — через Docker. При `docker compose down` — сохраняется. При `docker compose down -v` — удаляется. Для баз данных — только этот тип.

Bind mount. `./data:/app/data`. Монтирует конкретную папку хоста в контейнер. Удобно для разработки: меняете файл на хосте — изменения сразу видны в контейнере. Но для продакшена не рекомендуется — привязка к конкретному пути на хосте.

Anonymous volumes. Создаются автоматически, без имени. Привязаны к жизненному циклу контейнера. Удаляется контейнер — удаляется volume. Используются редко.

Правило простое: данные базы — именованные volumes. Код в dev — bind mount. В prod — код внутри образа, volumes только для данных.

## Слайд 15: Первый запуск на VPS

Теперь практика. У вас есть VPS — виртуальный приватный сервер. Пустой, чистый, с Ubuntu. Давайте поднимем Django за пять минут.

Подключаемся под root: `ssh root@81.90.182.174`. На чистом сервере — первым делом ставим Docker. Один лайнер: `curl -fsSL https://get.docker.com | sh`. Это официальный скрипт от Docker Inc. Скачивает и устанавливает Docker Engine и Docker Compose. Для учебного сервера — идеально. Для банковского — лучше ручная установка из официального репозитория, чтобы контролировать каждый пакет.

Дальше — безопасность. Не запускайте приложение от root. Создаём пользователя `deploy`: `adduser --disabled-password deploy`. Добавляем его в группу docker: `usermod -aG docker deploy`. Теперь deploy может работать с Docker без `sudo`.

Создаём директорию проекта: `mkdir -p /opt/shop && chown deploy:deploy /opt/shop`. Передаём владение пользователю deploy.

Переключаемся на deploy: `su - deploy`. Клонируем репозиторий: `git clone https://github.com/user/shop.git .`. Точка в конце — клонировать в текущую директорию, а не создавать вложенную папку.

И запускаем: `docker compose -f docker-compose.prod.yml up --build -d`. `--build` — пересобрать образы. `-d` — detached mode, в фоне. Docker скачает PostgreSQL, соберёт образ Django, запустит оба контейнера.

Проверяем: `curl http://localhost:8080/api/health`. Если видите `{"status": "ok"}` — поздравляю, ваш Django работает на VPS.

## Слайд 16: Nginx — зачем он нужен Django

У вас Django работает на порту 8080. Можно открыть этот порт в firewall и дать пользователям стучаться напрямую. Работать будет. Но это примерно как выпустить школьника на боксёрский ринг — технически можно, практически — плохая идея.

Gunicorn обрабатывает Python-код. Это его единственная задача, и он делает её хорошо. Но он НЕ умеет много других вещей.

Раздавать статику. CSS, JavaScript, картинки. Gunicorn для каждого файла запустит Python-процесс — это в сто раз медленнее, чем Nginx, который отдаёт файлы на уровне системных вызовов, без интерпретатора.

Терминировать SSL. HTTPS — это шифрование. Расшифровка каждого запроса нагружает CPU. Nginx оптимизирован для этого, Gunicorn — нет.

Буферизовать запросы. Когда клиент с медленным интернетом загружает файл — он отправляет данные по чуть-чуть. Gunicorn будет держать воркер занятым всё время загрузки. Nginx принимает весь запрос, буферизует его, и только потом передаёт Gunicorn мгновенно. Воркер занят миллисекунды вместо секунд.

Ограничивать rate. Если кто-то шлёт тысячу запросов в секунду — Nginx может заблокировать его по IP. Без этого — DDoS кладёт ваш сервис.

Кэшировать ответы. Повторные запросы к одному и тому же URL можно отдавать из кэша Nginx, вообще не трогая Django.

Nginx — это «охранник на входе». Event-driven, обрабатывает тысячи соединений в одном процессе. Gunicorn — «специалист внутри», который занимается только бизнес-логикой. Без Nginx Gunicorn торчит голым в интернет. Любой скрипт-кидди положит ваш сервис slowloris-атакой за минуту.

## Слайд 17: Конфиг Nginx для Django

Давайте разберём конфиг Nginx для нашего проекта.

Блок `server` — описание виртуального хоста. На одном Nginx может быть десятки сайтов, каждый в своём `server`-блоке.

`listen 80` — слушать на порту 80 (HTTP). После настройки HTTPS Certbot добавит `listen 443 ssl`.

`server_name django.h1n.ru` — домен. Nginx сопоставляет заголовок Host из HTTP-запроса с `server_name` и выбирает нужный `server`-блок.

`location /` — обработка всех запросов. `proxy_pass http://127.0.0.1:8080` — перенаправить на Gunicorn. Порт 8080, потому что в docker-compose мы маппим `8080:8000`.

`proxy_set_header Host $host` — передать оригинальный домен. Без этого Django увидит `Host: 127.0.0.1` вместо `Host: django.h1n.ru` и вернёт ошибку 400, потому что `127.0.0.1` нет в `ALLOWED_HOSTS`.

`X-Real-IP` — передать IP клиента. Без этого в логах Django все запросы будут от `127.0.0.1`. А вам нужно знать, кто стучится — для аналитики, для бана, для дебага.

`X-Forwarded-For` — цепочка прокси. Если перед Nginx ещё один прокси (CDN, load balancer) — в этом заголовке будет вся цепочка IP.

`X-Forwarded-Proto` — протокол. HTTP или HTTPS. Django использует это для генерации правильных ссылок.

`location /static/` — отдельное правило для статики. `alias /opt/shop/staticfiles/` — Nginx отдаёт файлы напрямую из этой папки. Django вообще не участвует. Быстро и эффективно.

## Слайд 18: Nginx — активация и проверка

Конфиг написали. Теперь его нужно активировать. В Nginx есть элегантная система двух папок.

`/etc/nginx/sites-available/` — здесь лежат все конфиги. Активные и неактивные.

`/etc/nginx/sites-enabled/` — здесь лежат симлинки на активные конфиги. Nginx читает только эту папку.

Создаём конфиг: `sudo nano /etc/nginx/sites-available/django.h1n.ru`.

Активируем: `sudo ln -s /etc/nginx/sites-available/django.h1n.ru /etc/nginx/sites-enabled/`. Создаём символическую ссылку. Хотите отключить сайт — удалите симлинк. Хотите включить обратно — создайте заново. Сам конфиг не трогается.

И вот теперь — священное правило, которое нельзя нарушать: `sudo nginx -t`. Это проверка синтаксиса конфига. ОБЯЗАТЕЛЬНО перед каждым reload или restart.

Почему? Потому что если в конфиге ошибка — `systemctl reload nginx` уронит весь Nginx. Не только ваш сайт — ВСЕ сайты на этом сервере перестанут работать. Если на сервере десять сайтов — десять команд разработчиков будут вам благодарны. В кавычках.

`nginx -t` проверяет конфиг без перезапуска. Если ошибка — покажет строку и описание. Исправляете — проверяете снова — и только потом `sudo systemctl reload nginx`.

`reload` — мягкий перезапуск. Nginx перечитывает конфиг без разрыва существующих соединений. `restart` — жёсткий. Для применения конфига — всегда `reload`.

## Слайд 19: HTTPS — Let's Encrypt + Certbot

HTTP — открытый текст. Буквально. Всё, что передаётся между браузером и сервером — логины, пароли, куки, персональные данные — видно любому, кто может перехватить трафик. На WiFi в кафе — это все за соседними столиками. В корпоративной сети — это админ. У провайдера — это провайдер.

HTTPS — это HTTP, обёрнутый в TLS-шифрование. Перехватить можно, прочитать — нет. Let's Encrypt — некоммерческий центр сертификации, который выдаёт SSL-сертификаты бесплатно. Certbot — утилита для автоматического получения и обновления этих сертификатов.

Две команды. `apt install certbot python3-certbot-nginx -y` — установить Certbot с плагином для Nginx. `certbot --nginx -d django.h1n.ru` — получить сертификат для домена.

Certbot автоматически делает четыре вещи. Первое — связывается с Let's Encrypt и доказывает, что вы контролируете домен. Для этого он либо создаёт файл в корне сайта и просит Let's Encrypt проверить, либо использует DNS-запись. Второе — получает сертификат и сохраняет его на сервере. Третье — изменяет конфиг Nginx: добавляет `listen 443 ssl`, пути к сертификатам, настраивает редирект с HTTP на HTTPS. Четвёртое — добавляет cron-задачу для автоматического обновления сертификатов каждые 90 дней.

После этого: `https://django.h1n.ru` — зелёный замочек в браузере. Без этого Chrome показывает «Не безопасно», а Google понижает сайт в поисковой выдаче. HTTPS — не опция, а требование.

## Слайд 20: Полная схема production Django

Давайте соберём всё вместе и посмотрим на полную архитектуру.

Пользователь вводит `django.h1n.ru` в браузере. Браузер обращается к DNS — системе доменных имён — и получает IP-адрес `81.90.182.174`. Это A-запись в DNS: имя → IP.

Запрос приходит на Nginx на порту 80. Nginx видит, что это HTTP, и делает редирект на HTTPS — порт 443. Браузер повторяет запрос по HTTPS. Nginx расшифровывает SSL.

Дальше — маршрутизация. Если URL начинается с `/static/` — Nginx отдаёт файл напрямую из папки `/opt/shop/staticfiles/`. CSS, JavaScript, картинки — мгновенно, без участия Python.

Если URL динамический — `/`, `/api/products`, `/admin/` — Nginx проксирует запрос на Docker-контейнер с Gunicorn, порт 8080. Gunicorn внутри контейнера слушает на 8000, Docker маппит 8080→8000.

Gunicorn — 8 воркеров, 4 потока в каждом. Итого 32 параллельных запроса. Один из свободных потоков берёт запрос и выполняет Django-код. Django через ORM обращается к PostgreSQL в соседнем контейнере. PostgreSQL с `max_connections=300` обрабатывает запрос и возвращает данные.

Ответ идёт обратно по цепочке: PostgreSQL → Django → Gunicorn → Nginx → браузер. Четыре слоя. И каждый можно масштабировать независимо. Нужно больше трафика? Добавьте второй Nginx — load balancer. Нужно больше Python-мощности? Горизонтальное масштабирование — запускаете второй Gunicorn на другом сервере. Нужно больше данных? Read-реплика PostgreSQL.

Сленг: «стек» — вся эта цепочка. «Ботлнек» — узкое место. «Скейлить» — масштабировать. «Горизонтально» — больше серверов. «Вертикально» — мощнее сервер.

## Слайд 21: Бенчмарки — зачем измерять

Мы настроили стек. Django работает, Nginx проксирует, PostgreSQL хранит данные. Вопрос: а насколько быстро? Сколько запросов выдержит?

«Premature optimization is the root of all evil» — сказал Дональд Кнут. Преждевременная оптимизация — корень всех зол. Но оптимизация без измерений — это вообще гадание на кофейной гуще. Вы не знаете, что тормозит. Вы не знаете, помогло ли ваше изменение. Сначала — измеряем. Потом — оптимизируем. Потом — измеряем снова.

Два инструмента для нагрузочного тестирования.

`wrk` — написан на C, минимальный overhead, умеет создавать тысячи соединений. `wrk -t4 -c200 -d30s URL` — 4 потока, 200 одновременных соединений, 30 секунд. Это стресс-тест: 200 параллельных подключений — серьёзная нагрузка.

`ab` — Apache Benchmark. Проще в использовании, предустановлен на многих системах. `ab -n 10000 -c 100 URL` — 10 000 запросов, 100 параллельных. Более предсказуемый тест: фиксированное количество запросов.

Что смотреть в результатах? Четыре метрики. Requests per second — пропускная способность, главная метрика. Latency — время ответа, то, что чувствует пользователь. Timeouts — сколько запросов не дождались ответа. Errors — сколько запросов получили ошибку 5xx.

Идеал: высокий Req/sec, низкий Latency, ноль Timeouts, ноль Errors. В реальности — ищем баланс.

## Слайд 22: Baseline — Gunicorn 3 воркера

Начинаем с baseline — базового замера. Дефолтный Gunicorn, 3 синхронных воркера, PostgreSQL с дефолтными настройками. Ничего не тюнили — запустили и тестируем.

Результат wrk: 200 соединений, 30 секунд. 50 запросов в секунду. Средняя задержка — 1.06 секунды. И главное — 1531 таймаут из 1658 запросов. Девяносто два процента запросов не дождались ответа. Это катастрофа.

Почему так плохо? Давайте посчитаем. Три синхронных воркера — это три параллельных запроса. Не четыре, не десять — ровно три. Все остальные 197 соединений из 200 стоят в очереди и ждут. Каждый воркер обрабатывает один запрос за раз: принял, сходил в базу, сериализовал ответ, отправил — и только потом берёт следующий.

А при `conn_max_age=0` — дефолтное значение — каждый запрос открывает новое соединение к PostgreSQL. Это TCP handshake + аутентификация — плюс 5-10 миллисекунд. При 50 запросах в секунду это 50 новых соединений в секунду. PostgreSQL тоже не бесконечный.

Запомните эту цифру: 50 req/s. Это наш «замер до оптимизации». Точка отсчёта. Дальше будем улучшать и сравнивать.

## Слайд 23: Попытка 1 — uvicorn (async)

Первая идея оптимизации. Мы слышали, что «async — это быстро». Django Ninja поддерживает асинхронные views. Uvicorn — ASGI-сервер, который обрабатывает множество соединений в одном процессе через event loop. Давайте переключимся!

Меняем команду на `uvicorn config.asgi:application --host 0.0.0.0 --port 8000 --workers 4`. Четыре асинхронных воркера вместо трёх синхронных. И ставим `conn_max_age=600` — переиспользуем соединения к базе.

Результат: 55 req/s. Чуть лучше, чем 50. Но 1418 таймаутов. И — сюрприз — 1415 ответов с кодом не 2xx. То есть 1415 ошибок 500!

Смотрим в логи PostgreSQL: `FATAL: sorry, too many clients already`. Слишком много клиентов. PostgreSQL по умолчанию принимает 100 соединений, и мы их исчерпали.

Почему? Потому что `conn_max_age=600` с async-воркерами — это бомба замедленного действия. В синхронном Gunicorn один воркер = одно соединение к базе. Три воркера — три соединения. Стабильно.

В асинхронном Uvicorn event loop создаёт сотни потоков через `sync_to_async` — потому что Django ORM синхронный, и uvicorn оборачивает его в thread pool. Каждый поток создаёт соединение к PostgreSQL и держит его 10 минут — `conn_max_age=600`. Потоков — двести. Соединений — двести. PostgreSQL разрешает сто. Результат — ошибка.

Урок номер один: не все «оптимизации» улучшают ситуацию. Интуиция обманывает. Измеряйте.

## Слайд 24: Попытка 2 — uvicorn + conn_max_age=0

Окей, поняли проблему. Слишком много соединений. Исправляем: `conn_max_age=0` — закрывать соединение сразу после запроса. И на всякий случай `max_connections=300` в PostgreSQL — расширяем лимит.

Результат: 34 req/s. Ошибки пропали — ноль. Но 864 таймаута. И главное — 34 запроса в секунду. Это ХУЖЕ, чем дефолтный Gunicorn с 50!

Как так? Мы же на async! Это же должно быть быстрее! Ну, нет. И вот почему.

Django ORM — синхронный. Когда вы пишете `VideoCard.objects.all()`, это синхронный вызов. Uvicorn не может его выполнить в event loop — он блокирующий. Поэтому Uvicorn оборачивает его в `sync_to_async`, что означает: запустить в отдельном потоке из thread pool. Это добавляет overhead: переключение контекста между event loop и thread pool. Для каждого ORM-вызова.

В синхронном Gunicorn этого overhead нет. Воркер просто выполняет Python-код последовательно, без переключений. Прямолинейно и эффективно.

Вывод: uvicorn быстрее только для `async def` views, где вы используете асинхронные библиотеки — aiohttp, databases, aioredis. Для обычного Django ORM — gunicorn выигрывает. Это не баг, это архитектура.

## Слайд 25: conn_max_age — sync vs async

Давайте разберём, почему один и тот же параметр ведёт себя по-разному в sync и async.

Gunicorn с `conn_max_age=600`. У нас три синхронных воркера. Каждый воркер — отдельный процесс. Каждый процесс обрабатывает один запрос за раз. Открывает соединение к БД, делает запрос, получает ответ. Соединение не закрывается — переиспользуется для следующего запроса этим же воркером. Через 10 минут бездействия — закрывается. Итого: три воркера — три соединения. Стабильно, предсказуемо, безопасно.

Uvicorn с `conn_max_age=600`. Event loop принимает запрос и отправляет ORM-вызов в thread pool. Thread pool создаёт поток, поток открывает соединение к БД. `conn_max_age=600` — соединение держится 10 минут. Но поток из thread pool не равен воркеру — потоков может быть сотни. 200 одновременных запросов — 200 потоков — 200 соединений к БД. Каждое держится 10 минут. PostgreSQL с лимитом 100 — переполнение.

Правило простое. `conn_max_age=600` — только с синхронным Gunicorn. С async — либо `conn_max_age=0` (закрывать сразу), либо использовать пул соединений: pgbouncer, django-db-connection-pool. Пул ограничивает количество одновременных соединений и ставит остальные в очередь.

Это одна из тех ловушек, которые не описаны в туториалах. Вы читаете «поставьте conn_max_age=600 для ускорения» — и ставите. А потом ваш async-сервер валит PostgreSQL. Теперь вы знаете — почему.

## Слайд 26: Финальная оптимизация — Gunicorn 8w + 4t

Вернёмся к Gunicorn. Но настроим его правильно.

`--workers 8` — восемь воркеров. Формула: `2 × CPU_cores + 1`. На четырёхъядерном VPS: `2 × 4 + 1 = 9`, округляем до 8 для ровного счёта. Каждый воркер — отдельный процесс с собственным интерпретатором Python.

`--threads 4` — четыре потока в каждом воркере. Потоки работают внутри одного процесса, но благодаря тому, что ORM-запросы — это I/O (ждём ответа от БД), GIL отпускается, и потоки работают параллельно. 

8 воркеров × 4 потока = 32 параллельных запроса. Вместо трёх. В десять раз больше параллелизма.

`conn_max_age=600` — безопасно с sync Gunicorn. Максимум 32 соединения к БД — по одному на поток. PostgreSQL с `max_connections=300` — запас в десять раз.

Важно: больше воркеров не значит лучше. 20 воркеров на двухъядерном VPS будут конкурировать за CPU, делать context switch, и работать медленнее, чем 5 воркеров. Каждый воркер — это отдельный процесс Python, который занимает память и CPU. Формула `2 × CPU + 1` — это не с потолка, это эмпирически проверенный баланс.

С потоками аналогично. 2-4 потока для I/O-bound задач (запросы к БД) — оптимально. Больше — overhead на переключение перевешивает выигрыш.

## Слайд 28: Итоговое сравнение конфигураций

Давайте сведём все результаты в таблицу и сделаем выводы.

Gunicorn 3 воркера — 50 req/s, 1.06 секунды latency, 1531 таймаут. Baseline. Ужасно.

## Слайд 30: Чек-лист production Django

Перед выкаткой Django на прод — пройдите по этому списку. Распечатайте и повесьте над монитором.

Безопасность. `DEBUG = False` — без вариантов. `SECRET_KEY` из переменной окружения — не в коде. `ALLOWED_HOSTS` — конкретный домен, не звёздочка. HTTPS через Certbot — обязательно.

Производительность. Gunicorn с правильным числом воркеров — `2 × CPU + 1`. `conn_max_age=600` для sync. `collectstatic` выполнен, Nginx раздаёт статику. PostgreSQL `max_connections` больше или равно числу воркеров умножить на два — с запасом.

Надёжность. `restart: unless-stopped` в Docker или systemd с `Restart=always`. CI/CD пайплайн — GitHub Actions. Мониторинг — хотя бы `curl /health` по крону. Бэкапы базы данных — ежедневно.

Production — это не «код работает». Production — это «код работает надёжно, быстро, безопасно и обновляется автоматически». Разница — как между прототипом и серийным автомобилем. Прототип может ехать. Серийный автомобиль проходит краш-тесты, имеет гарантию, сервисное обслуживание и страховку.

Ваш чек-лист — это ваш краш-тест перед выездом на дорогу.

## Слайд 31: Инструменты профилирования Django

Мы оптимизировали Gunicorn — это «внешняя» оптимизация. Мы увеличили пропускную способность сервера: больше воркеров, больше потоков, переиспользование соединений. Но что если тормозит не сервер, а сам код?

N+1 запросы, когда Django делает 50 SQL-запросов вместо одного. Медленные ORM-вызовы без индексов. Неоптимальная сериализация, которая перебирает тысячи объектов в цикле. Тяжёлые шаблоны с вложенными циклами. Эти проблемы не решить количеством воркеров — нужны инструменты профилирования.

Шесть инструментов, которые должен знать каждый Django-разработчик. Django Silk — перехватывает каждый HTTP-запрос, записывает SQL, время, профиль Python. Работает с JSON API. Django Debug Toolbar — боковая панель в HTML-страницах с SQL, шаблонами, кэшем. django-querycount — просто пишет в логи количество SQL на каждый запрос. runprofileserver — запускает dev-сервер с cProfile, сохраняет профили в файлы. line_profiler — построчное профилирование Python-функций. django-prometheus — экспорт метрик для Grafana.

Ключевой принцип: сначала профилируйте, потом оптимизируйте. Без замеров вы чините то, что не сломано, и игнорируете реальное узкое место. Я видел, как разработчики неделю оптимизировали сериализацию, а тормозил — запрос к БД без индекса. Профилирование показало бы это за пять минут.

## Слайд 32: Django Silk — профилирование API

Django Silk — мой любимый инструмент. Он перехватывает каждый HTTP-запрос и записывает в базу данных: время выполнения, все SQL-запросы, профиль Python-кода, тело запроса и ответа.

Главное преимущество Silk: он работает с JSON API. Django Ninja, DRF — всё покрывает. Debug Toolbar не может — ему нужен HTML-ответ с тегом `</body>`, чтобы встроить свою панель. Если ваш API возвращает JSON — Debug Toolbar бесполезен. Silk — нет.

Установка: `pip install django-silk`. Добавляете в `INSTALLED_APPS`, подключаете middleware — обязательно первым в списке, чтобы перехватывать запрос до всех остальных middleware. Включаете `SILKY_PYTHON_PROFILER = True` — это профилирование Python-кода, не только SQL. `SILKY_META = True` — замеряет overhead самого Silk, чтобы вы знали, сколько он добавляет.

Добавляете URL-маршрут: `path("silk/", include("silk.urls"))`. Выполняете `python manage.py migrate` — Silk создаёт свои таблицы.

После этого — делаете пару запросов к API и заходите на `http://127.0.0.1:8000/silk/`. Панель покажет все запросы, отсортированные по времени. Кликаете на медленный — видите каждый SQL-запрос, сколько он занял, есть ли дубликаты. Это как рентген для вашего приложения.

## Слайд 33: Silk — что смотреть в панели

Панель Silk — это три раздела, каждый отвечает на свой вопрос.

Раздел Requests — «Какие запросы медленные?». Таблица со всеми HTTP-запросами. Колонка Time — общее время. Если больше 100 миллисекунд — повод разобраться. Num. Queries — количество SQL-запросов. Больше 5-10 — возможна N+1 проблема. Time on queries — если это время примерно равно общему Time — значит, тормозит база, а не Python-код. Если Time сильно больше Time on queries — тормозит Python: сериализация, валидация, вычисления.

Раздел SQL — «Какие SQL-запросы выполняются?». Текст каждого запроса, время выполнения. И самое ценное — дубликаты. Когда один и тот же `SELECT * FROM video_cards WHERE id = 5` выполняется 50 раз — это красный флаг. Это N+1.

Раздел Profiling — «Какие Python-функции медленные?». Граф вызовов — какая функция сколько CPU съела. Помогает найти тормоза, не связанные с базой: тяжёлая сериализация, регулярные выражения, обработка файлов.

Практический пример. `GET /api/products` — 1 SQL-запрос, 0.8 миллисекунды. Отлично, ничего не трогаем. `GET /api/orders` — 51 SQL-запрос, 120 миллисекунд. 51 запрос? Это точно N+1. Один запрос на список заказов + по одному на каждый товар в заказе. 50 заказов — 51 запрос. Нужен `select_related`.

## Слайд 34: N+1 проблема — главный враг производительности

N+1 — это самый частый баг производительности в Django ORM. И самый коварный — он не вызывает ошибок. Код работает правильно, просто медленно. И чем больше данных — тем медленнее. Silk находит его моментально.

Как выглядит N+1 в Silk? `GET /api/orders — 51 queries, 120ms`. Первый запрос: `SELECT * FROM orders` — получить все заказы. Потом для каждого заказа: `SELECT * FROM video_cards WHERE id = 1`, `WHERE id = 2`, `WHERE id = 3`... И так 50 раз. Django ORM по умолчанию ленивый — он не подгружает связанные объекты, пока вы к ним не обратитесь. Когда вы в шаблоне или сериализаторе пишете `order.product.name` — Django делает отдельный запрос к базе. Для каждого заказа.

50 заказов — 51 запрос. 1000 заказов — 1001 запрос. 10 000 — десять тысяч запросов к базе на один HTTP-запрос. Линейный рост. При 100 миллисекундах на запрос — это 100 секунд на ответ. Не совместимо с жизнью.

Исправление — одна строчка. `Order.objects.select_related('product').all()`. `select_related` делает SQL JOIN: один запрос возвращает и заказы, и товары. Вместо 51 запроса — 2. Вместо 120 миллисекунд — 3. В сорок раз быстрее от одной строчки кода.

Для ManyToMany-связей — `prefetch_related('items')`. Он делает два запроса: один за заказами, один за товарами — и связывает их в Python. Тоже два запроса вместо N+1.

Silk покажет разницу мгновенно. До — 51 запрос. После — 2. Это самая эффективная оптимизация, которую вы когда-либо сделаете.

## Слайд 35: Silk — настройки для разных окружений

Silk полезен на dev, но опасен на prod. Он добавляет 5-10 миллисекунд на каждый запрос и пишет в базу данных. При 200 запросах в секунду — это 200 записей в секунду в таблицу Silk. Ваша база быстро разрастётся.

Правильная конфигурация — включать Silk только в debug-режиме.

```python
if DEBUG:
    INSTALLED_APPS += ['silk']
    MIDDLEWARE.insert(0, 'silk.middleware.SilkyMiddleware')
```

Полезные настройки для контроля. `SILKY_INTERCEPT_PERCENT = 100` — записывать 100% запросов. Для нагрузочного тестирования поставьте 10 — записывать каждый десятый. `SILKY_MAX_REQUEST_BODY_SIZE = 0` и `SILKY_MAX_RESPONSE_BODY_SIZE = 0` — не записывать тела запросов и ответов, экономит место. `SILKY_AUTHENTICATION = True` и `SILKY_AUTHORISATION = True` — требовать авторизацию для доступа к панели, только staff-пользователи.

Silk забивает базу данных. Периодически чистите: `python manage.py silk_clear_request_log`. Или настройте `SILKY_MAX_RECORDED_REQUESTS = 1000` — хранить только последнюю тысячу запросов.

И помните: на проде Silk должен быть выключен. Без исключений. Даже если «только посмотреть». Потому что «только посмотреть» превращается в «забыл выключить» — и вот у вас база на 10 гигабайт из логов Silk.

## Слайд 36: Django Debug Toolbar — профилирование HTML-страниц

Django Debug Toolbar — второй инструмент в арсенале. В отличие от Silk, который работает тихо в фоне и пишет в базу, Debug Toolbar встраивается прямо в HTML-страницу — боковая панель справа.

Установка стандартная: `pip install django-debug-toolbar`. В `INSTALLED_APPS` добавляете `debug_toolbar`, middleware — `DebugToolbarMiddleware`, и обязательно — `INTERNAL_IPS = ['127.0.0.1']`. Toolbar показывается только для IP из этого списка.

В urls.py подключаете URL-маршруты Debug Toolbar — но только при `DEBUG=True`.

Открываете HTML-страницу — справа появляется панель с разделами. SQL — все SQL-запросы с кнопкой EXPLAIN для каждого. Time — CPU time и total time. Templates — какие шаблоны рендерились и какие переменные контекста в них переданы. Cache — hit и miss кэша. Signals — Django-сигналы и обработчики. Static files — загруженные CSS и JS.

Ограничение, о котором нужно знать: Debug Toolbar работает только с HTML-ответами. Он встраивает свой JavaScript перед `</body>`. Если ваш API возвращает JSON — панель не появится. Для JSON API — используйте Silk. Для HTML-страниц — Debug Toolbar. Вместе — покрывают 100% случаев.

## Слайд 37: Debug Toolbar — SQL и EXPLAIN

Главный раздел Debug Toolbar — SQL. Кликаете на него и видите все запросы, которые выполнил Django при обработке страницы. Для каждого — текст запроса и время.

Но главная фишка — кнопка EXPLAIN. Нажимаете — и видите план выполнения PostgreSQL. Это как рентгеновский снимок запроса: как база данных решила его выполнить.

`Seq Scan on video_cards` — последовательное сканирование. PostgreSQL перебирает все строки таблицы одну за другой. На 5 строках — нормально, быстрее индекса. На 100 000 строках — катастрофа. Нужен индекс.

`Index Scan` — PostgreSQL использует индекс. Быстро. Если видите Index Scan — всё хорошо.

Ещё Debug Toolbar подсвечивает две важные категории. Similar — запросы с одинаковой структурой, но разными параметрами. `SELECT * FROM products WHERE id = 1`, `SELECT * FROM products WHERE id = 2`... Это маркер N+1 проблемы. Duplicate — полностью идентичные запросы. Если один и тот же SELECT выполняется дважды — это точно баг. Может, вы вызываете функцию дважды, или шаблон запрашивает одни и те же данные в разных местах.

Когда панель не появляется — четыре причины и других нет. Ответ не HTML. IP не в `INTERNAL_IPS`. `DEBUG = False`. В шаблоне нет закрывающего `</body>`. Проверяйте по этому списку.

## Слайд 38: django-querycount и runprofileserver

Два лёгких инструмента, когда Silk и Debug Toolbar — overkill. Иногда вам не нужна полная панель с историей — нужно просто быстро глянуть: «сколько запросов делает этот endpoint?»

django-querycount — middleware, который пишет в логи одну строку на каждый HTTP-запрос: `[QueryCount] 3 queries in 2.1ms`. Всё. Без UI, без базы данных, без настройки. Pip install, добавили middleware — работает. Идеально для быстрой проверки во время разработки. Открыли страницу — глянули в консоль — увидели 47 запросов — пошли разбираться.

runprofileserver — из пакета django-extensions. Запускает dev-сервер с встроенным cProfile. Каждый HTTP-запрос сохраняется как `.prof` файл в указанную папку. Потом открываете его в snakeviz — интерактивная визуализация в браузере. Flamegraph: какие функции сколько CPU заняли, кто кого вызывал.

`python manage.py runprofileserver --use-cprofile --prof-path=/tmp/profiles/`. Делаете запрос. В папке появляется файл. `snakeviz /tmp/profiles/GET.api.products.prof` — открывается браузер с графиком.

Полезно для тяжёлых вычислений: сериализация больших объёмов данных, обработка файлов, сложные алгоритмы. Когда SQL быстрый, а Python тормозит — runprofileserver покажет, какая именно строчка виновата.

## Слайд 39: cProfile, line_profiler, django-prometheus

Ещё три инструмента для разных задач.

cProfile — встроенный профилировщик Python. Не нужно ничего устанавливать — он есть в стандартной библиотеке. `cProfile.run('my_function()', sort='cumulative')` — и вы видите таблицу: каждая функция, количество вызовов, суммарное время. Грубый, но бесплатный и всегда доступен. Используйте, когда нужно быстро понять, что тормозит, без установки дополнительных пакетов.

line_profiler — построчное профилирование. Это следующий уровень детализации. Вы знаете, что функция тормозит — но какая именно строка? Декоратор `@profile` и команда `kernprof -l -v script.py`. Результат: каждая строка функции с временем выполнения. `products = VideoCard.objects.all()` — 0.3мс. `data = serialize(products)` — 12.5мс. Bottleneck найден — сериализация. Незаменим, когда нужно найти конкретную строку.

django-prometheus — это уже не отладка, а мониторинг. Эндпоинт `/metrics` отдаёт данные в формате Prometheus: количество запросов, латенси по перцентилям, ошибки, SQL-запросы. Подключаете Prometheus для сбора метрик, Grafana для визуализации — и видите графики в реальном времени. Dashboards с latency, RPS, error rate. Алерты, когда метрики выходят за пределы. Это инструмент для production — 24/7 мониторинг.

## Слайд 40: Silk vs Debug Toolbar — когда что использовать

Два главных инструмента — Silk и Debug Toolbar. Давайте чётко разграничим.

JSON API — Silk. Debug Toolbar не работает с JSON-ответами. Если ваш проект — это Django Ninja или DRF с мобильным приложением — Silk единственный выбор.

HTML-страницы — оба. Но Debug Toolbar удобнее: панель прямо на странице, не нужно переключаться в отдельный интерфейс.

История запросов — Silk. Он хранит все запросы в базе. Вы можете посмотреть, что было час назад, сравнить до и после оптимизации. Debug Toolbar показывает только текущую страницу.

SQL EXPLAIN — Debug Toolbar. Кнопка EXPLAIN для каждого запроса, план выполнения PostgreSQL. У Silk этого нет.

Профилирование Python — Silk. cProfile граф вызовов. Debug Toolbar не умеет.

Шаблоны и контекст — Debug Toolbar. Какие шаблоны рендерились, какие переменные передали. Silk это не покрывает.

Моя рекомендация: используйте оба одновременно. Silk — для API-эндпоинтов и истории. Debug Toolbar — для HTML-страниц, EXPLAIN и шаблонов. Вместе они покрывают 95% задач профилирования Django.

И на проде — оба выключены. Через `if DEBUG`. Silk пишет в базу на каждый запрос. Debug Toolbar показывает внутренности приложения. На проде это и тормоз, и дыра в безопасности.

## Слайд 41: Как использовать связку инструментов правильно

Завершающий слайд. Каждый инструмент закрывает свой этап жизненного цикла приложения. Не нужно выбирать один — нужно использовать правильный инструмент в правильный момент.

Цепочка на практике. Этап первый — разработка. Debug Toolbar и Silk. Вы пишете код, открываете страницу, смотрите в панель. Нашли N+1 — добавили `select_related`. Нашли медленный запрос — добавили индекс. Это ваш ежедневный рабочий цикл.

Этап второй — локальная оптимизация. Вы нашли функцию, которая тормозит, но не понятно почему. line_profiler или cProfile — построчный анализ. Нашли строку — оптимизировали — проверили снова.

Этап третий — деплой. Добавляете django-prometheus. Эндпоинт `/metrics` экспортирует метрики. Silk и Debug Toolbar — выключены.

Этап четвёртый — мониторинг в production. Grafana читает метрики из Prometheus и рисует дашборды: графики latency, RPS, error rate. Вы видите в реальном времени, как себя чувствует приложение.

Этап пятый — инцидент. Prometheus Alertmanager отправляет алерт в Telegram или Slack: «p95 latency больше 500мс, проверьте». Вы открываете Grafana, видите скачок — и знаете, когда он начался, что изменилось, куда копать.

На dev — профилируете и чините. На prod — мониторите и реагируете. Silk и Debug Toolbar на проде отключены. Prometheus и Grafana на dev не нужны. Каждый инструмент на своём месте — и вы спите спокойно.

## Слайд 42: Пулы соединений к PostgreSQL (PgBouncer)

Представьте: вы подняли Gunicorn с восемью воркерами. Каждый воркер открывает пул соединений к Postgres. Плюс админка, плюс миграции, плюс бэкап — и внезапно база отвечает «sorry, too many connections». Это не баг Django — это арифметика: лимит `max_connections` у PostgreSQL конечен.

PgBouncer — это маленький демон между приложением и базой. Приложение думает, что у него куча соединений; на самом деле PgBouncer держит десяток реальных сессий к Postgres и переиспользует их между запросами. Для Django в проде это почти стандарт de facto на своём железе.

Нюанс с `conn_max_age`: если за PgBouncer вы держите долгоживущие коннекты из Django, иногда получается двойной пул и странные таймауты. На практике за балансировщиком часто ставят `conn_max_age=0` и доверяют пулу на стороне PgBouncer или облака. Главное — осознанно, а не «скопировали из туториала и удивляемся».

## Слайд 43: Кэш Redis + Django

Кэш — это когда вы один раз потратились на тяжёлый запрос, а следующие сто раз отдаёте результат из памяти. Redis рядом с приложением — классика: django-redis или встроенный Redis backend в новых версиях Django.

Кэшировать имеет смысл каталоги, справочники, редко меняющиеся JSON для API. Не кэшируйте слепо всё подряд: персональная корзина в общем ключе — это либо утечка данных, либо чужая корзина у пользователя. И всегда продумайте инвалидацию: обновили товар — сбросили ключ или префикс. Иначе фронт показывает прошлогодний прайс, а поддержка объясняет, что «у нас всё обновилось, просто кэш».

## Слайд 44: Фоновые задачи — Celery и RQ

Пользователь нажал «Скачать отчёт за год». Если вы строите отчёт прямо в view — HTTP-соединение висит минуту, воркер Gunicorn занят, очередь запросов растёт. Правильный паттерн: положить задачу в очередь, сразу ответить «отчёт готовится», воркер в фоне соберёт PDF и положит в S3 или пришлёт письмо.

Celery — тяжёлая артиллерия с кучей возможностей. RQ — проще, на Redis. Суть одна: HTTP должен быть коротким; тяжёлое — в асинхрон. Плюс вы получаете ретраи и мониторинг очереди: если очередь растёт как снежный ком — значит, воркеров мало или задача зависла.

## Слайд 45: Django 6 — встроенный Tasks API

С шестой ветки Django в ядре появился официальный слой для «очереди задач» — не замена Celery одной кнопкой, а **единый контракт**: вы помечаете функцию декоратором `@task`, ставите в очередь через `.enqueue()`, получаете объект результата со статусом. Приоритеты, имена очередей, отложенный запуск `run_after` — всё это описано в одном API, а не в десяти разных стилях из разных библиотек.

Подвох в том, что **движок выполнения Django в коробке не даёт**. По умолчанию стоит ImmediateBackend — задача выполняется прямо сейчас, в том же запросе. То есть из коробки это не фон, а удобная обёртка. Для тестов есть DummyBackend — задачи копятся, но не бегут. Настоящий фон — это уже пакеты вроде очереди в PostgreSQL, мост на Celery, Huey, пул потоков — каждый реализует интерфейс бэкенда.

Зачем тогда это студенту? Во-первых, в новых проектах на Django 6 имеет смысл **сразу писать задачи через `@task`**, а бэкенд менять по мере роста: сначала immediate на dev, потом database или celery на проде. Во-вторых, аргументы в очередь идут через JSON — меньше сюрпризов с несериализуемыми объектами. В-третьих, документация одна — `ref/tasks` и `topics/tasks`, а не разрозненные README.

Итого: Django 6 не отменяет Celery и RQ, а **стандартизирует вход** в мир фоновых задач. Воркер по-прежнему ваш или community.

## Слайд 46: Rate limiting в Nginx

Не каждый запрос к вашему API честный. Боты, случайные скрипты, DDoS низкой интенсивности — Nginx может отрезать грубым limit_req: столько-то запросов в секунду с IP, с burst на короткие всплески. Это не замена OAuth и не защита от умного атакующего, но снимает 80% «случайного» перегруза.

Отдельно настраивают лимиты для публичного API и для админки — у админки обычно жёстче и меньше клиентов. Главное — не задушить легитимный мобильный клиент с тысячей пользователей за одним NAT; иногда лимитируют по API-ключу в приложении, а не только по IP.

## Слайд 47: Ошибки в production — Sentry

Метрики говорят: error rate вырос. А *где* именно упало — в Grafana не всегда видно. Sentry ловит исключение, собирает stack trace, версию релиза, breadcrumbs — что пользователь нажал до падения. После выката новой версии первые часы критичны: один алерт в Slack «новый тип ошибки» экономит ночь саппорта.

Интеграция в Django — несколько строк и DSN в секрете. На проде Sentry оставляют включённым (в отличие от Silk), потому что это не про отладку каждого SQL, а про агрегированные ошибки без утечки секретов в ответах клиенту.

## Слайд 48: Логи, метрики, трейсы

Три кита observability. Логи — что случилось в конкретный момент; если логи структурированы JSON и есть correlation id, можно проследить один запрос через сервисы. Метрики — Prometheus, тренды, алерты. Трейсы — OpenTelemetry: видно, что запрос провёл 200 мс в Django, 180 из них в одном SQL.

На одном VPS полный Jaeger не обязателен, но понимание этой тройки помогает при инциденте: сначала метрика «latency выросла», потом лог «timeout к Redis», потом по желанию трейс «узкое место — внешний API». Минимальный здравый набор: нормальные логи, уже есть метрики, плюс Sentry на исключения.

## Слайд 49: Чек-лист после деплоя и куда расти

Короткий ритуал после каждого выката: smoke curl на health, один короткий прогон wrk или ab против baseline — вдруг регрессия в два раза. В Grafana глянуть ошибки и p95. Проверить место на диске и число соединений к БД — банально, но ночью диск забитый логами никого не радует.

Развитие темы дальше — не «ещё один слайд», а целые курсы: read replicas, CDN для статики, blue-green чтобы не ловить 502 при перезапуске контейнера, chaos на staging. Но суть лекции одна: нагрузку можно измерять, ограничивать, смещать в фон и наблюдать.
