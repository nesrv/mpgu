Слайд 1. HTMX — что это

HTMX — библиотека ~14 KB: расширяет HTML атрибутами, не фреймворком.

  • Частичные обновления страницы без тяжёлого SPA
  • Гипермедиа — сервер отдаёт HTML-фрагменты → вставка в DOM
  • Слоган: HTML over the wire вместо JSON + клиентский рендер

Зачем: меньше JS, проще стек (Django, FastAPI, шаблоны).

Контракт: (1) атрибуты → что запрашивать; (2) заголовки ответа → редирект, события, target/swap.

HTMX — «середина»: динамика без SPA.

// Без HTMX: вручную fetch + вставка в DOM (каждый раз одно и то же)
fetch('/items/partial/')
  .then(r => r.text())
  .then(html => {
    document.getElementById('list').innerHTML = html; // риск XSS, если сервер недоверенный
  });
<!-- HTMX: тот же сценарий — атрибуты вместо JS -->
<button hx-get="/items/partial/"
        hx-target="#list"
        hx-swap="innerHTML">Обновить</button>
<div id="list"></div>

Слайд 3. Как это работает (модель)

  1. Клик / ввод / отправка формы
  2. HTMX → HTTP к URL из атрибута (+ заголовки HX-* с клиента)
  3. Сервер → тело (HTML) + при необходимости заголовки HX-*
  4. Вставка в DOM (hx-target + hx-swap) или действия по заголовкам

Ключ: бэкенд — источник правды.

# Исходящий запрос (фрагмент; Host и др. опущены)
GET /search/?q=phone HTTP/1.1
HX-Request: true          # признак HTMX — на бэкенде отдаём partial
HX-Target: #results        # куда на странице вставят ответ (информативно)
<!-- Тело ответа 200 OK — не целая страница, а кусок HTML -->
<ul>
  <li>Товар A</li>
  <li>Товар B</li>
</ul>

Ответ сервера: <span>42</span>

# urls: path("fragments/counter/", counter_fragment)
def counter_fragment(request):
    # Счётчик в сессии (пример; в проде — своя модель/логика)
    n = int(request.session.get("n", 0)) + 1
    request.session["n"] = n
    return HttpResponse(f"<span>{n}</span>")  # только фрагмент, не layout

Слайд 7. Форма + ответ сервера

  • <form hx-post="/save" hx-target="#result"> — фрагмент в ответе
  • hx-boost="true" — ссылки и формы через AJAX внутри контейнера
  • Файлы: hx-encoding="multipart/form-data" + enctype + file input
<!-- Ответ сервера подменяет #result (ошибка или «OK») -->
<form hx-post="/contact/" hx-target="#result" hx-swap="innerHTML">
  {% csrf_token %}
  <input name="email" type="email" required>
  <button type="submit">Отправить</button>
</form>
<div id="result"></div>
def contact(request):
    if request.method == "POST":
        if not request.POST.get("email"):
            # Снова форма + ошибка (тот же шаблон)
            return render(request, "form.html", {"err": "Нужен email"})
        return HttpResponse('<p class="ok">Отправлено</p>')
        # Успех + редирект: resp["HX-Redirect"] = "/thanks/"
    return render(request, "contact_full.html")  # первая загрузка — полная страница

Слайд 8. CSRF + partial во view

  • HX-Request: true — отличить HTMX от обычного запроса (partial vs full page)
  • CSRF: {% csrf_token %} + htmx:configRequest → X-CSRFToken
// Все HTMX-запросы (POST/PUT/DELETE) получают CSRF — иначе Django 403
document.body.addEventListener('htmx:configRequest', function (e) {
  e.detail.headers['X-CSRFToken'] = '{{ csrf_token }}'; // из Django-шаблона
});
def item_list(request):
    ctx = {"items": Item.objects.all()}
    if request.headers.get("HX-Request"):
        # Запрос из HTMX — только строки таблицы (без base.html)
        return render(request, "items/_rows.html", ctx)
    # Обычный заход — полная страница (SEO, No-JS)
    return render(request, "items/list.html", ctx)

Слайд 9. HX-Redirect и HX-Trigger

ЗаголовокДействие
HX-RedirectПолный переход (как location)
HX-Refreshtrue — перезагрузка страницы
HX-TriggerJSON событий на document/body
HX-Retarget / HX-ReswapПереопределить target/swap
HX-Push-Url / HX-Replace-UrlИстория с сервера
# После успешного POST — полный переход (как форма с redirect)
resp = HttpResponse(status=204)  # тело можно пустое
resp["HX-Redirect"] = "/orders/"
return resp

# Плюс уведомление на клиенте (слушатель на кастомное событие toast)
r = HttpResponse("<div>OK</div>")
r["HX-Trigger"] = '{"toast": "Заказ создан"}'  # JSON имён событий
return r

Слайд 10. hx-select + ограничения

  • Не заменяет тяжёлый клиентский state (редакторы, канвас) — там SPA
  • Много запросов → debounce (hx-trigger), кэш, hx-sync
  • SEO: без JS ссылки должны работать, если важна индексация
<!-- Сервер отдал HTML с layout; берём из ответа только #main -->
<button hx-get="/full-page/"
        hx-select="#main"
        hx-target="#main"
        hx-swap="outerHTML">Подгрузить контент</button>

Слайд 12. Итоги

Выбирайте HTMX: списки, формы, фильтры, пагинация, модалки; сильный бэкенд; MVP.

SPA: офлайн, очень сложный UI, большой клиентский state.

Резюме: гипермедиа + двусторонний контракт (атрибуты + заголовки) = мало JS.

Слайд 13. Out-of-band swap (oob)

Один ответ → основной target + доп. узлы: hx-swap-oob="true" + совпадающий id.

<!-- Основной swap — в hx-target (например tbody); oob — второй кусок -->
<tr><td>Молоко</td><td>2</td></tr>
<!-- Элемент с тем же id на странице заменится дополнительно -->
<div id="cart-badge" hx-swap-oob="true">3</div>

Счётчики, хедер, тосты — без второго запроса.

Слайд 16. afterSwap (скролл чата)

  • htmx:beforeRequest / afterRequest
  • htmx:afterSwap — после вставки DOM
  • htmx:responseError — 4xx/5xx
// После подгрузки сообщений — прокрутить чат вниз
document.body.addEventListener('htmx:afterSwap', function (e) {
  if (e.detail.target.id === 'chat') {
    e.detail.target.scrollTop = e.detail.target.scrollHeight;
  }
});

Слайд 17. Тест partial (pytest)

  • Метод, CSRF, права — как у обычных форм
  • HX-Target, HX-Trigger-Name с клиента — не доверять для авторизации
  • XSS: экранирование; не вставлять сырой пользовательский HTML
# Имитация HTMX-запроса — ожидаем partial, не полную страницу
def test_partial(client):
    r = client.get("/items/", HTTP_HX_REQUEST="true")
    assert r.status_code == 200
    body = r.content.decode()
    assert "tr" in body or "tbody" in body  # фрагмент таблицы
