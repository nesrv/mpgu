Управление нагрузкой Django в production

Django — другая реальность

  • python manage.py runserver — только для разработки
  • Production — WSGI/ASGI-сервер, reverse proxy, CI/CD
  • От голого Django до оптимизированного продакшена с бенчмарками

Архитектура production Django

Браузер → Nginx (порт 80/443) → Gunicorn (порт 8000) → Django → PostgreSQL
  • Nginx — принимает HTTP, раздаёт статику, терминирует SSL
  • Gunicorn — WSGI-сервер, запускает воркеры Django
  • Django — бизнес-логика, ORM, шаблоны
  • PostgreSQL — база данных, а не SQLite

Gunicorn не умеет раздавать файлы, SSL, защищаться от slowloris. Nginx — «охранник на входе».

settings.py для production

import os
import dj_database_url

SECRET_KEY = os.environ.get('SECRET_KEY', 'change-me')
DEBUG = os.environ.get('DEBUG', 'False') == 'True'
ALLOWED_HOSTS = os.environ.get('ALLOWED_HOSTS', '').split(',')

DATABASES = {
    'default': dj_database_url.config(
        default='postgresql://postgres:postgres@localhost:5432/shop',
        conn_max_age=600,
    )
}

STATIC_URL = '/static/'
STATIC_ROOT = os.path.join(BASE_DIR, 'staticfiles')
  • dj_database_url — парсит строку подключения в формат Django
  • conn_max_age=600 — не закрывать соединение к БД 10 минут
  • STATIC_ROOT — папка для collectstatic, Nginx раздаёт напрямую

Порядок деплоя с CI/CD

git push → GitHub Actions → SSH на VPS → git pull → docker compose up

Пушишь в master — через 30 секунд новая версия на сервере.

Что нужно:

  • GitHub Secrets — пароли от сервера в зашифрованном виде
  • Workflow файл — .github/workflows/deploy.yml
  • SSH-доступ — GitHub Actions подключается к VPS

Сленг: «пайплайн» — цепочка CI/CD. «Воркфлоу» — файл пайплайна. «Триггер» — событие запуска.

GitHub Actions — deploy.yml

name: Deploy to VPS

on:
  push:
    branches: [master]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          password: ${{ secrets.VPS_PASSWORD }}
          script: |
            cd /opt/shop
            git pull origin master
            docker compose -f docker-compose.prod.yml up --build -d

appleboy/ssh-action — готовый action для SSH. Секреты из GitHub Settings → Secrets.

GitHub Secrets — безопасность CI/CD

Секреты GitHub шифруются и не показываются в логах. Даже при fork — не копируются.

ИмяЗначениеЗачем
VPS_HOST81.90.182.174IP сервера
VPS_USERdeploySSH-пользователь
VPS_PASSWORD***SSH-пароль

⚠️ echo ${{ secrets.VPS_PASSWORD }} — GitHub замаскирует. Но echo ... | base64 — может и не замаскировать декодированную версию!

Продвинутый подход — SSH-ключи вместо пароля.

Docker при деплое — зачем?

Проблемы без Docker:

  • «У меня Python 3.12, а на сервере 3.9»
  • «libpq-dev конфликтует с другим проектом»
  • «На моём маке всё работало!»

Docker для Django:

  • Изоляция — PostgreSQL, Django, Redis в отдельных контейнерах
  • Воспроизводимость — docker compose up и всё работает
  • Масштабирование — легко добавить воркеры, реплики
  • Откат — вернуться к предыдущему образу за секунду

Dockerfile для Django

FROM python:3.14-slim

WORKDIR /app

RUN apt-get update && apt-get install -y --no-install-recommends \
    libpq-dev && \
    rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

RUN python manage.py collectstatic --noinput 2>/dev/null || true

EXPOSE 8000

CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000", "--workers", "3"]
  • python:3.14-slim — ~150 МБ vs ~1 ГБ для полного
  • COPY requirements.txt отдельно — кэширование слоёв Docker

docker-compose.prod.yml

services:
  db:
    image: postgres:17
    restart: unless-stopped
    environment:
      POSTGRES_DB: shop
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: ${DB_PASSWORD:-postgres}
    command: postgres -c max_connections=300
    volumes:
      - pg_data:/var/lib/postgresql/data

  web:
    build: .
    restart: unless-stopped
    ports:
      - "8080:8000"
    depends_on:
      - db
    environment:
      DATABASE_URL: postgresql://postgres:${DB_PASSWORD:-postgres}@db:5432/shop
      DEBUG: "False"
    command: >
      sh -c "python manage.py migrate &&
             python load_data.py &&
             gunicorn config.wsgi:application --bind 0.0.0.0:8000 --workers 8 --threads 4"
volumes:
  pg_data:

Два сервиса, одна сеть. Docker DNS: сервис db доступен по имени db.

Разбор docker-compose по строкам

ПараметрЧто делает
restart: unless-stoppedПерезапуск при падении и после ребута
${DB_PASSWORD:-postgres}Переменная окружения с дефолтом
max_connections=300Увеличение лимита PostgreSQL (дефолт — 100)
pg_data:/var/lib/…Именованный volume — данные переживают down
build: .Собрать образ из Dockerfile
ports: "8080:8000"Хост 8080 → контейнер 8000
depends_on: dbЗапускать web после db (не ждёт готовности!)

depends_on гарантирует порядок старта, но НЕ готовность БД. restart: unless-stopped спасёт.

Docker volumes — где живут данные

volumes:
  - pg_data:/var/lib/postgresql/data

Три типа volumes:

  • Именованные (pg_data:) — управляются Docker, переживают пересоздание
  • Bind mount (./data:/app/data) — монтируют папку хоста
  • Anonymous — привязаны к контейнеру, удаляются с ним

docker compose down — volume сохраняется. down -v — удаляется! ⚠️

Для баз данных — только именованные. Для кода в dev — bind mount.

Nginx — зачем он нужен Django

Gunicorn обрабатывает Python-код. Но он НЕ умеет:

  • Раздавать статику — в 100× медленнее через Python
  • Терминировать SSL — HTTPS расшифровка нагружает CPU
  • Буферизовать запросы — медленный клиент займёт воркер
  • Ограничивать rate — защита от DDoS
  • Кэшировать ответы — без обращения к Django

Nginx — «охранник на входе» (event-driven, тысячи соединений). Gunicorn — «специалист внутри».

Конфиг Nginx для Django

server {
    listen 80;
    server_name django.h1n.ru;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location /static/ {
        alias /opt/shop/staticfiles/;
    }
}
  • proxy_pass — перенаправить запрос на Gunicorn
  • proxy_set_header Host — иначе Django вернёт 400
  • location /static/ — Nginx отдаёт статику напрямую

Nginx — активация и проверка

# Создать конфиг
sudo nano /etc/nginx/sites-available/django.h1n.ru

# Включить (симлинк)
sudo ln -s /etc/nginx/sites-available/django.h1n.ru /etc/nginx/sites-enabled/

# Проверить синтаксис (ОБЯЗАТЕЛЬНО!)
sudo nginx -t

# Применить без остановки
sudo systemctl reload nginx

sites-available / sites-enabled: все конфиги в sites-available, активные — симлинки в sites-enabled.

⚠️ nginx -t перед каждым reload. Без исключений.

HTTPS — Let's Encrypt + Certbot

apt install certbot python3-certbot-nginx -y
certbot --nginx -d django.h1n.ru

Certbot автоматически:

  • Получит сертификат (проверит, что домен → этот сервер)
  • Изменит конфиг Nginx (listen 443 ssl)
  • Настроит редирект HTTP → HTTPS
  • Добавит cron для автообновления (каждые 90 дней)

После: https://django.h1n.ru — зелёный замочек 🔒

Полная схема production Django

Браузер → django.h1n.ru
         ↓
   DNS → 81.90.182.174 (A-запись)
         ↓
   Nginx (порт 80 → 443, SSL)
    ├── /static/ → файлы напрямую
    └── / → proxy_pass
         ↓
   Docker: Gunicorn (порт 8080)
    ├── 8 воркеров × 4 потока = 32 параллельных запроса
    └── Django ORM
         ↓
   Docker: PostgreSQL (порт 5432)
    └── max_connections=300

4 слоя: DNS → Nginx → Gunicorn → PostgreSQL. Каждый масштабируется независимо.

Бенчмарки — зачем измерять

«Premature optimization is the root of all evil» — Д. Кнут. Но оптимизация без измерений — гадание.

# wrk: 4 потока, 200 соединений, 30 секунд
wrk -t4 -c200 -d30s http://81.90.182.174:8080/api/products

# ab: 10000 запросов, 100 параллельных
ab -n 10000 -c 100 http://81.90.182.174:8080/api/products

Что смотреть в результатах:

  • Requests/sec — пропускная способность (главная метрика)
  • Latency — время ответа (что чувствует пользователь)
  • Timeouts — сколько запросов не дождались ответа
  • Errors — сколько получили 5xx

Baseline — Gunicorn 3 воркера

Стартовая конфигурация: дефолтный Gunicorn, 3 синхронных воркера, PostgreSQL с дефолтами.

wrk -t4 -c200 -d30s http://81.90.182.174:8080/api/products

Requests/sec:     50.08
Avg Latency:      1.06s
Timeouts:         1531 из 1658
Errors:           0

50 req/s при 200 соединениях. 92% — таймауты. Катастрофа.

3 синхронных воркера = 3 параллельных запроса. Остальные 197 в очереди. Каждый запрос к PostgreSQL — новое подключение.

Попытка 1 — uvicorn (async)

command: uvicorn config.asgi:application --host 0.0.0.0 --port 8000 --workers 4

Результат с conn_max_age=600:

Requests/sec:     55.34
Timeouts:         1418
Non-2xx responses: 1415  ← ОШИБКИ 500!

Стало хуже! 1415 запросов вернули 500:

FATAL: sorry, too many clients already

conn_max_age=600 + async = бомба. Каждый async-поток держит соединение 10 минут. Потоков — сотни → PostgreSQL переполнен.

Урок: не все «оптимизации» улучшают ситуацию. Измеряйте.

Попытка 2 — uvicorn + conn_max_age=0

# settings.py
DATABASES = {
    'default': dj_database_url.config(
        default='postgresql://postgres:postgres@localhost:5432/shop',
        conn_max_age=0,  # закрывать сразу
    )
}
Requests/sec:     34.00
Timeouts:         864
Errors:           0

Ошибки пропали, но 34 req/s — хуже, чем Gunicorn с 50!

Uvicorn оборачивает синхронные ORM-вызовы в sync_to_async — overhead переключения event loop ↔ thread pool.

Вывод: uvicorn быстрее только для async def views.

conn_max_age — sync vs async

Gunicorn (sync), conn_max_age=600:

Воркер 1 → соединение к БД (держит 10 мин, переиспользует)
Воркер 2 → соединение к БД (держит 10 мин, переиспользует)
Воркер 3 → соединение к БД (держит 10 мин, переиспользует)
= 3 соединения (стабильно)

Uvicorn (async), conn_max_age=600:

async task 1   → соединение (не отпускает 10 мин)
async task 2   → соединение (не отпускает 10 мин)
...
async task 200 → соединение (не отпускает 10 мин)
= 200 соединений → PostgreSQL: "too many clients"

Правило: conn_max_age=600 — только с sync Gunicorn. С async — conn_max_age=0 или пул (pgbouncer).

Финальная оптимизация — Gunicorn 8w + 4t

command: gunicorn config.wsgi:application \
    --bind 0.0.0.0:8000 \
    --workers 8 \
    --threads 4
conn_max_age=600  # безопасно с sync gunicorn

8 воркеров × 4 потока = 32 параллельных запроса

Формула workers: 2 × CPU_cores + 1. На 4-ядерном VPS: 2 × 4 + 1 = 9 ≈ 8

Потоки — 2-4 для I/O-bound задач (ORM-запросы к БД)

⚠️ Больше воркеров ≠ лучше. 20 воркеров на 2-ядерном VPS конкурируют за CPU.

Django Silk — профилирование API

Silk перехватывает каждый HTTP-запрос и записывает: время, SQL, профиль Python-кода.

INSTALLED_APPS = [..., 'silk']

MIDDLEWARE = [
    'silk.middleware.SilkyMiddleware',   # первым в списке!
    'django.middleware.security.SecurityMiddleware',
    ...
]

SILKY_PYTHON_PROFILER = True
SILKY_MAX_RECORDED_REQUESTS = 1000
SILKY_META = True   # замерять overhead самого Silk
# urls.py
path("silk/", include("silk.urls", namespace="silk")),

Главное: работает с JSON API (Django Ninja, DRF). Debug Toolbar не может.

Silk — что смотреть в панели

Requests — «Какие запросы медленные?»

КолонкаНа что смотреть
TimeОбщее время > 100мс — повод разобраться
Num. QueriesБольше 5-10 — возможна N+1 проблема
Time on queriesЕсли ≈ Time — тормозит БД, не Python

SQL — «Какие SQL-запросы?»

  • Текст каждого запроса + время
  • Дубликаты — один и тот же SELECT 50 раз

Profiling — «Какие Python-функции медленные?»

  • Граф вызовов — какая функция сколько CPU съела
  • Тормоза не связанные с БД (сериализация, валидация)

N+1 проблема — главный враг

Как выглядит в Silk:

GET /api/orders — 51 queries, 120ms

SELECT * FROM orders                          -- 1 запрос
SELECT * FROM video_cards WHERE id = 1        -- N запросов
SELECT * FROM video_cards WHERE id = 2
SELECT * FROM video_cards WHERE id = 3
... (ещё 47 штук)

Исправление:

# Плохо — N+1:
orders = Order.objects.all()

# Хорошо — 2 запроса вместо 51:
orders = Order.objects.select_related('product').all()

# Для ManyToMany:
orders = Order.objects.prefetch_related('items').all()

Было 51 запрос → стало 2. Время: 120мс → 3мс. В 40 раз быстрее от одной строчки.

Silk — настройки для разных окружений

# settings.py — включать только в DEBUG
if DEBUG:
    INSTALLED_APPS += ['silk']
    MIDDLEWARE.insert(0, 'silk.middleware.SilkyMiddleware')
    SILKY_PYTHON_PROFILER = True
    SILKY_PYTHON_PROFILER_BINARY = True
    SILKY_MAX_RECORDED_REQUESTS = 1000
SILKY_INTERCEPT_PERCENT = 100       # 100% запросов (или 10 для sampling)
SILKY_MAX_REQUEST_BODY_SIZE = 0     # не записывать тело запроса
SILKY_MAX_RESPONSE_BODY_SIZE = 0    # не записывать тело ответа
SILKY_AUTHENTICATION = True         # требовать авторизацию
SILKY_AUTHORISATION = True          # только для staff
# Очистка данных (Silk забивает БД):
python manage.py silk_clear_request_log

Debug Toolbar — SQL и EXPLAIN

2 queries in 1.5ms

SELECT ⋯ FROM "video_cards"               0.8ms
SELECT ⋯ FROM "django_session" WHERE ⋯    0.3ms

Кнопка EXPLAIN для каждого запроса:

Seq Scan on video_cards  (cost=0.00..1.05 rows=5 width=...)
  • Seq Scan на 5 строках — нормально
  • Seq Scan на 100 000 строках — нужен индекс
  • Index Scan — PostgreSQL использует индекс ✓

Similar — одинаковая структура, разные параметры (N+1). Duplicate — идентичные запросы (баг).

cProfile — встроенный профилировщик:

import cProfile

@profile
def list_products(request):
    products = VideoCard.objects.all()  # 0.3ms
    data = serialize(products)           # 12.5ms  ← bottleneck!
    return data

Декоратор @profile + kernprof -l -v script.py

Пулы соединений — PgBouncer

  • Много воркеров Gunicorn → много соединений к Postgres → лимит max_connections
  • PgBouncer — пул между приложением и БД; мало реальных коннектов к PostgreSQL
  • Режим для Django чаще transaction
  • conn_max_age согласовать с пулом (иногда 0 за балансировщиком)

Кэш Redis + Django

  • Тяжёлые повторяющиеся запросы → cache.set, {% cache %}
  • Инвалидация при обновлении данных — иначе устаревший контент
  • Осторожно с персональными данными в общем кэше
CACHES = {
  'default': {
    'BACKEND': 'django.core.cache.backends.redis.RedisCache',
    'LOCATION': 'redis://127.0.0.1:6379/1',
  }
}

Фоновые задачи — Celery / RQ

  • Письма, отчёты, тяжёлая обработка — не в HTTP-запросе
  • Очередь (Redis/RabbitMQ) + воркеры → стабильная latency для пользователя
  • Мониторить длину очереди и failed jobs

Django 6 — Tasks API

  • @task + .enqueue() — единый контракт очереди в ядре (с 6.0)
  • TaskResult — статусы, priority, queue_name, run_after
  • Встроенно: ImmediateBackend (сразу в запросе) и DummyBackend (тесты)
  • Продакшен: бэкенды community — django-tasks-database, django-tasks-celery, Huey…
  • Аргументы enqueue — только JSON-сериализуемые
from django.tasks import task

@task()
def notify(user_id: int) -> str:
    return "ok"

notify.enqueue(42)  # TaskResult
