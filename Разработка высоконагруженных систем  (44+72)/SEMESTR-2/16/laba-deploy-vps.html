1.4. Настройка Nginx для вашего домена

Nginx использует схему sites-available / sites-enabled:

  • sites-available — все заготовки конфигов (могут быть выключены),
  • sites-enabled — только те, что активны (симлинки на файлы из sites-available).

В конфиге задаётся:

  • listen 80 — слушать порт 80 (HTTP),
  • server_name — по какому домену Nginx будет отвечать,
  • root — каталог с файлами сайта,
  • index — файл по умолчанию при заходе в каталог,
  • try_files — искать файл по URI, при отсутствии — 404.

Команда ln -sf создаёт символическую ссылку: «включить» конфиг, не удаляя оригинал. Перед перезагрузкой обязательно проверяем конфиг командой nginx -t, чтобы не сломать работающий Nginx.

Про tee: cat | sudo tee файл > /dev/null — heredoc передаётся в tee, он пишет в файл с правами root. Так обходим проблему: sudo cat > файл не работает, потому что редирект выполняет shell без sudo. > /dev/null отключает лишний вывод в терминал.

# Создаём конфиг Nginx (sudo) cat << 'EOF' | sudo tee /etc/nginx/sites-available/alekseeva > /dev/null server { listen 80; server_name alekseeva.h1n.ru; root /var/www/alekseeva; index index.html; location / { try_files $uri $uri/ =404; } } EOF sudo ln -sf /etc/nginx/sites-available/alekseeva /etc/nginx/sites-enabled/ sudo nginx -t && sudo systemctl reload nginx

systemctl reload nginx применяет новый конфиг без полной остановки сервиса — текущие соединения не обрываются, а новые запросы идут по новым правилам.

2.4. Systemd-сервис

Systemd — менеджер служб в Linux. С ним приложение:

  • запускается при старте сервера (автозапуск),
  • перезапускается при падении,
  • управляется командами systemctl.

Секции unit-файла:

  • [Unit] — метаданные: описание и зависимости (например, After=network.target — ждать поднятия сети).
  • [Service] — как запускать процесс: от какого пользователя, из какой папки, какая команда (ExecStart), политика перезапуска (Restart=always, RestartSec=5).
  • [Install] — когда включать сервис: WantedBy=multi-user.target — вместе с обычным multi-user режимом (после загрузки).

Слушаем 127.0.0.1, а не 0.0.0.0: приложение доступно только локально, снаружи — только через Nginx (порт 80).

Команды: daemon-reload — подхватить новый unit; enable — включить автозапуск; start — запустить сейчас.

# Создаём unit-файл (sudo) cat << 'EOF' | sudo tee /etc/systemd/system/alekseeva-api.service > /dev/null [Unit] Description=Alekseeva FastAPI App After=network.target [Service] User=root WorkingDirectory=/opt/alekseeva-api ExecStart=/opt/alekseeva-api/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8010 Restart=always RestartSec=5 [Install] WantedBy=multi-user.target EOF sudo systemctl daemon-reload sudo systemctl enable alekseeva-api sudo systemctl start alekseeva-api sudo systemctl status alekseeva-api

Убедитесь, что в выводе sudo systemctl status alekseeva-api указано active (running).

2.5. Nginx для FastAPI

Nginx выступает как reverse proxy: принимает запросы на порт 80 и передаёт их FastAPI (порт 8010). Клиент общается только с Nginx.

Директивы в конфиге:

  • listen 80 — слушать HTTP на 80-м порту.
  • server_name — для какого домена отвечать.
  • location / — правило для всех путей: всё перенаправляем на бэкенд.
  • proxy_pass — куда отдавать запрос: FastAPI на 127.0.0.1:8010.
  • proxy_set_header — передать бэкенду заголовки клиента: Host (домен), X-Real-IP (IP клиента), X-Forwarded-For, X-Forwarded-Proto (HTTP/HTTPS). FastAPI видит исходного клиента, а не Nginx.

Команды: nginx -t — проверка синтаксиса; systemctl reload nginx — применить конфиг без остановки сервиса.

# Обновляем конфиг (sudo) cat << 'EOF' | sudo tee /etc/nginx/sites-available/alekseeva > /dev/null server { listen 80; server_name alekseeva.h1n.ru; location / { proxy_pass http://127.0.0.1:8010; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; proxy_set_header X-Forwarded-Proto $scheme; } } EOF sudo nginx -t && sudo systemctl reload nginx

  2. Локально: git remote add origin git@github.com:ВАШ_ЛОГИН/alekseeva-api.git
  3. На VPS: ssh-keygen -t ed25519 -f ~/.ssh/github_deploy -N "" → GitHub → репозиторий → Settings → Deploy keys, вставьте cat ~/.ssh/github_deploy.pub

Создайте файл .github/workflows/deploy.yml:

name: Deploy to Server on: push: branches: [main] jobs: deploy: runs-on: ubuntu-latest steps: - name: Deploy via SSH uses: appleboy/ssh-action@v1.0.0 with: host: 81.90.182.174 username: root # или логин студента password: ${{ secrets.SSH_PASSWORD }} script: | cd /opt/alekseeva-api git pull origin main source venv/bin/activate pip install -r requirements.txt systemctl restart alekseeva-api

Settings → Secrets and variables → Actions → New secret → Name: SSH_PASSWORD, Value: пароль от сервера (root или логин студента).
