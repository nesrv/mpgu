Что такое деплой?

Деплой (Deploy) — размещение приложения на сервере, чтобы оно стало доступно пользователям через интернет.

Локальный компьютер
Ваш код
main.py
→SSH
Сервер (VPS)
Приложение
работает
←HTTP
Пользователи
Браузеры

VPS (Virtual Private Server) — виртуальный сервер в дата-центре с публичным IP-адресом.

Общий алгоритм деплоя

1 Подключение к серверу (SSH)
↓
2 Установка зависимостей (Python, pip, venv)
↓
3 Загрузка кода (git clone или копирование)
↓
4 Настройка окружения (venv, pip install)
↓
5 Настройка автозапуска (systemd)
↓
6 Настройка веб-сервера (Nginx)
↓
7 Настройка DNS (привязка домена)
↓
8 Настройка HTTPS (Certbot)

SSH — подключение к серверу

SSH (Secure Shell) — защищённый протокол для удалённого управления сервером.

ssh root@81.90.182.174
Часть командыЗначение
sshПрограмма для подключения
rootИмя пользователя
@Разделитель

После подключения вы управляете сервером через командную строку (терминал).

Создание SSH-сессии в MobaXterm

Пошаговая инструкция:

  1. Нажмите Session → SSH
  2. Заполните поля: Remote host: 81.90.182.174, Username: root, Port: 22
  3. Нажмите OK
  4. Введите пароль при первом подключении
  5. Нажмите Yes чтобы сохранить пароль

Сохранение сессии:

  • Сессия появится в левой панели
  • Двойной клик для быстрого подключения

Передача файлов (SFTP)

SFTP — безопасная передача файлов через SSH.

В MobaXterm: левая панель показывает файлы сервера, Drag & Drop для загрузки.

В командной строке:

# Копирование файла на сервер
scp main.py root@81.90.182.174:/opt/myapi/

# Копирование папки на сервер
scp -r myproject/ root@81.90.182.174:/opt/

# Копирование файла с сервера
scp root@81.90.182.174:/opt/myapi/main.py ./

Альтернативы: WinSCP, FileZilla

Права доступа (chmod)

В Linux каждый файл имеет права доступа для трёх категорий:

КатегорияОбозначение
Владелец (user)u
Группа (group)g
Остальные (others)o
ПравоБукваЧисло
Чтение (read)r4
Запись (write)w2
Выполнение (execute)x1

chmod — примеры

# Сделать скрипт исполняемым
chmod +x deploy.sh

# Числовой формат: rwx = 4+2+1 = 7
chmod 755 deploy.sh    # rwxr-xr-x
chmod 644 main.py      # rw-r--r--
chmod 600 secrets.env  # rw------- (только владелец)

Расшифровка 755:

  • 7 (владелец) = 4+2+1 = rwx
  • 5 (группа) = 4+0+1 = r-x
  • 5 (остальные) = 4+0+1 = r-x

Systemd — автозапуск сервисов

Systemd — система управления сервисами в Linux.

Проблема: Если запустить uvicorn в терминале и закрыть SSH — приложение остановится.

Решение: Создать systemd-сервис.

systemd
init-система
→управляет
uvicorn
приложение
  • Автозапуск при загрузке ОС
  • Перезапуск при падении
  • Логирование (journalctl)

Создание systemd-сервиса

# /etc/systemd/system/myapi.service

[Unit]
Description=My FastAPI App
After=network.target

[Service]
User=root
WorkingDirectory=/opt/myapi
ExecStart=/opt/myapi/venv/bin/uvicorn main:app --host 127.0.0.1 --port 8010
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
СекцияОписание
[Unit]Метаданные и зависимости
[Service]Как запускать
[Install]Когда запускать

Команды systemctl

systemctl daemon-reload    # Перечитать конфиги
systemctl enable myapi     # Автозапуск при загрузке
systemctl start myapi      # Запустить
systemctl stop myapi       # Остановить
systemctl restart myapi    # Перезапустить
systemctl status myapi     # Статус

Просмотр логов:

journalctl -u myapi -f       # В реальном времени
journalctl -u myapi -n 50    # Последние 50 строк

Nginx — зачем нужен?

Пользователь
Интернет
→
VPS — Nginx (:80, :443)
→ FastAPI (:8010)
→ Django (:8000)
→ Статика (/var/www/)

Nginx выполняет: Reverse proxy, SSL/TLS терминация, балансировка нагрузки, раздача статики, кеширование

Конфигурация Nginx

# /etc/nginx/sites-available/mysite

server {
    listen 80;
    server_name mysite.ru;
    
    location / {
        proxy_pass http://127.0.0.1:8010;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

server { } — блок виртуального хоста; location / { } — правила; proxy_pass — адрес бэкенда

Nginx — включение сайта

# Создать символическую ссылку
ln -sf /etc/nginx/sites-available/mysite /etc/nginx/sites-enabled/

# Проверить синтаксис
nginx -t

# Перезагрузить Nginx
systemctl reload nginx

Важно: Всегда проверяйте nginx -t перед перезагрузкой!

DNS — привязка домена

DNS (Domain Name System) — преобразует домен в IP-адрес.

mysite.ru → DNS-сервер → 81.90.182.174

ТипИмяЗначениеTTL
A@81.90.182.174300
Awww81.90.182.174300

@ — корневой домен; www — поддомен; TTL — время жизни записи

HTTPS с Certbot

Certbot — автоматическое получение SSL-сертификатов от Let's Encrypt.

apt install -y certbot python3-certbot-nginx
certbot --nginx -d mysite.ru -d www.mysite.ru

Что делает Certbot: проверяет владение доменом → получает сертификат → настраивает Nginx → редирект HTTP→HTTPS → автообновление

Скрипт деплоя

#!/bin/bash
# deploy.sh
set -e
echo "=== Pulling latest code ==="
cd /opt/myapi
git pull origin main
echo "=== Installing dependencies ==="
source venv/bin/activate
pip install -r requirements.txt
echo "=== Restarting service ==="
systemctl restart myapi
echo "=== Deploy completed! ==="
chmod +x deploy.sh
./deploy.sh
