# Текст лекции: Деплой FastAPI на VPS

*Презентация для студентов. Лектор: синьор, кандидат наук. Стиль: академический + сленг разработчиков, тонкий юмор. ~4–5 минут на слайд.*

---

## Слайд 1: Что такое деплой?

Итак, начнём с базы. **Деплой** — это когда вы выносите приложение из своего ноутбука на сервер и делаете его доступным миру. Пока код живёт только у вас в `main.py`, пользователи его не видят. Деплой — это мост между вашим кодовым скиллом и реальными людьми в браузерах.

Схема простая: у вас локальный компьютер, вы заливаете код на сервер по SSH, а сервер по HTTP раздаёт его пользователям. Промежуточное звено — **VPS**, виртуальный сервер в дата-центре с собственным IP.

Сленг: «задеплоить» — выкатить приложение на сервер. «Продакшен», «прод» — боевое окружение, где реальные пользователи. «Запушить на прод» — задеплоить обновление. «Локалка» — ваш компьютер или dev-окружение. Это уже не ваш ПК и не «магический облачный сервис», а конкретная Linux-машина, на которую вы заходите по SSH и настраиваете всё руками. Для джунов это самый полезный опыт: понимать, как всё устроено под капотом. Без этого знание Docker и Kubernetes будет скорее религией, чем инженерией.

## Слайд 2: Общий алгоритм деплоя

Деплой — это не один клик. Это последовательность шагов, и лучше их держать в голове как чек-лист. Сначала — подключение к серверу по SSH. Потом — установка зависимостей: Python, pip, venv. Дальше — загрузка кода: `git clone` или копирование. Затем — настройка окружения: виртуальное окружение и `pip install`. Пятый шаг — автозапуск через systemd, чтобы приложение не умирало при закрытии терминала. Шестой — Nginx как входная точка для HTTP. Седьмой — DNS, привязка домена к IP. И наконец — HTTPS с Certbot.

Каждый шаг можно упростить или заменить инструментами, но логика всегда одна: доступ → код → окружение → запуск → веб-сервер → домен → шифрование. Если пропустить хоть один — либо не заработает, либо будет небезопасно.

«Прогнать деплой» — выполнить весь процесс выкатки. «Поднять» — запустить сервис или приложение. «Поднять стек» — развернуть всю цепочку (nginx, app, база и т.д.).

## Слайд 3: SSH — подключение к серверу

SSH — это «удалённый рабочий стол» в виде терминала. Secure Shell — защищённый протокол, через него вы подключаетесь к серверу и работаете в его командной строке. Формат команды: `ssh root@81.90.182.174`. `root` — пользователь, `81.90.182.174` — IP сервера. После подключения вы — как будто сидите за консолью этого сервера.

Важно понимать разницу: `root` на проде — это бог сервера, без ограничений. Для учебных и пет-проектов окей; в реальной жизни лучше заводить отдельных пользователей и ограничивать права. Но для старта — root упрощает жизнь.

«Зайти по эсэшэ», «по SSH» — подключиться к серверу. «Рут» — root, суперпользователь (произносится «рут», не «роот»). «Подняться на сервер» — подключиться по SSH.

## Слайд 4: SSH-клиенты для Windows

В Linux и macOS SSH уже есть в терминале. В старых версиях Windows его не было. Сейчас в десятке и одиннадцатке — есть, но не все об этом знают. Если нужен нормальный клиент: **PuTTY** — классика, минимализм, весит копейки. **MobaXterm** — швейцарский нож: терминал, SFTP, сессии, всё в одном окне. **Windows Terminal** — встроенный в Windows 10/11. **Termius** — красивый, облачный, синхронизирует настройки. **Git Bash** — если уже ставили Git, SSH идёт в комплекте.

Для студентов логичнее всего MobaXterm: видно и терминал, и файлы, не нужно прыгать между программами.

Пару слов про сленг — чтобы вы понимали коллег. Если слышите «стукнуться по путику» или «замазка слетела» — речь о сессии в PuTTY. «Мобаикс» — самое распространённое сокращение MobaXterm. «Скинь сессию из Мобаикса» — значит, экспортируйте сохранённое подключение. «Моба» — ещё короче, в очень неформальной переписке. Если хотите звучать как опытный разработчик — говорите «МобаИкс» или полностью «МобаИксте́рм».

## Слайд 5: PuTTY — базовый SSH-клиент

PuTTY — это «открыл, ввёл IP, нажал Open». В окне конфигурации: Host Name — IP сервера, Port — 22, Connection type — SSH. Жмёте Open, вводите логин и пароль. Важно: пароль при вводе не отображается, ни звёздочками, ни точками — просто молчит. Это нормально, так и должно быть. Несколько раз случайно «заблокировал» себя, думая, что клавиатура сломалась.

Совет: если часто подключаетесь к одному серверу — сохраняйте сессию. PuTTY это умеет, MobaXterm — тем более.

«Путь», «путик» — PuTTY. «Конфигнуть путик» — настроить сохранённое подключение.

---

## Слайд 6: MobaXterm — продвинутый клиент

MobaXterm — это PuTTY на стероидах. Встроенный SFTP — видите файлы сервера в боковой панели, можно тащить-бросать. Несколько вкладок и сессий, сохранение паролей. X11 forwarding — если на сервере есть GUI-приложение, оно откроется у вас локально. Плюс grep, awk, vim и прочие утилиты уже в комплекте.

Один раз настроил — и работаешь как с локальной папкой, только она на сервере. Для джунов это сильно снижает порог входа.

«Мобаикс», «моба» — MobaXterm. «Завести сессию в мобаиксе» — сохранить SSH-подключение.

---

## Слайд 7: Создание SSH-сессии в MobaXterm

В MobaXterm: Session → SSH. Remote host — IP, Username — root, Port — 22. OK. При первом подключении спросит пароль и предложит сохранить — можно согласиться, если это ваш учебный сервер. Сессия появится в левой панели, дальше подключаетесь двойным кликом. Никаких лишних настроек, всё интуитивно.

Если пароль не сохраняете — каждый раз будете вводить. Для проды это даже правильно с точки зрения безопасности.

«Закликать», «засессить» — создать и сохранить сессию. «Сессия слетела» — подключение оборвалось.

## Слайд 8: Передача файлов (SFTP)

SFTP — это «файлы по SSH». В MobaXterm левая панель уже показывает файловую систему сервера, можно просто перетащить файл. В командной строке — `scp`. Чтобы скопировать файл на сервер: `scp main.py root@81.90.182.174:/opt/myapi/`. Папку — с флагом `-r`. С сервера к себе — поменять источник и назначение местами.

Альтернативы: WinSCP, FileZilla. Но если MobaXterm уже открыт — зачем лишнее окно?

Сленг: «залить» и «скинуть» — скопировать файлы на сервер или с сервера. «Залить код» — перенести проект. «Скопировать по эсцэпэ», «scp-нуть» — использовать команду `scp`. «Драг-энд-дроп» — перетащить мышкой.

## Слайд 9: Структура Linux-сервера

В Linux всё — файлы и папки. `/etc/` — конфиги: nginx, systemd и т.д. `/opt/` — приложения, сюда логично класть наш проект. `/var/` — логи, веб-корни, данные. `/home/` — домашние директории пользователей. `/tmp/` — временное.

Для приложения лучше `/opt/` или домашняя папка пользователя. В `/tmp/` при перезагрузке всё может исчезнуть — не для продового кода.

Произношение путей: «этка» — /etc/, «опт» — /opt/, «вар» — /var/. «Хоум» — /home/. «Тэмп» — /tmp/. В переписке часто пишут «etc», «opt» без слэшей.

## Слайд 10: Права доступа (chmod)

В Linux у каждого файла есть владелец, группа и «остальные». У каждой категории — права: чтение (r, 4), запись (w, 2), выполнение (x, 1). Складываете цифры и получаете число: 7 = rwx, 5 = r-x, 4 = r-- и т.д. Три числа — владелец, группа, остальные.

По умолчанию скрипты могут быть не исполняемыми. Тогда либо `chmod +x deploy.sh`, либо явно `chmod 755`. Это база, без неё сервер будет выдавать «Permission denied» на ровном месте.

«Права», «пермишены» — права доступа. «Чмоднуть» — выполнить chmod. «Рутануть» — запустить от root или получить root-права. «Пол-литра пять» — 755, стандартные права на исполняемый файл.

---

## Слайд 11: chmod — примеры

`chmod +x` — сделать исполняемым. `chmod 755` — владелец всё, остальные читают и выполняют. `chmod 644` — владелец пишет, остальные только читают. `chmod 600` — только владелец, для секретов вроде `.env` самое то.

Расшифровка 755: 7 = 4+2+1 = rwx, 5 = 4+1 = r-x. То есть владелец может всё, остальные — читать и запускать, но не менять. Типичная схема для исполняемых файлов и директорий.

«Шестьсот» (600) — только владелец, для секретов. «Шесть-четыре-четыре» (644) — обычный файл.

## Слайд 12: FastAPI — минимальное приложение

FastAPI — отличный выбор для API. Минимальное приложение: импорт, экземпляр, пара эндпоинтов. Запуск — `uvicorn main:app --host 0.0.0.0 --port 8010`. `0.0.0.0` означает «слушать все интерфейсы», не только localhost — иначе снаружи не подключиться. Порт — любой свободный, 8010 — просто пример.

На локалке можно и `127.0.0.1`, но на сервере для доступа извне нужен именно `0.0.0.0`. Это частая ошибка у новичков: приложение запущено, но с хоста не открывается.

«Апишка», «апи» — API. «Поднять uvicorn», «запустить uvicorn» — стартовать сервер. «Порт висит», «слушает порт» — приложение приняло соединения на этом порту. «Зеро-зеро» — 0.0.0.0.

## Слайд 13: Systemd — автозапуск сервисов

Если запустить uvicorn в терминале и закрыть SSH — процесс умрёт. Systemd решает эту проблему: это система управления сервисами в Linux. Она запускает ваш процесс, перезапускает при падении, пишет логи в journalctl. Вы создаёте unit-файл, включаете сервис — и приложение живёт независимо от вашей сессии.

Для production это must have. Без systemd (или аналога) приложение — одноразовое: перезагрузили сервер или закрыли терминал — и всё.

«Поднять сервис» — запустить. «Демон» — фоновый процесс, сервис. «Юнит», «юнит-файл» — конфиг systemd.

---

## Слайд 14: Создание systemd-сервиса

Unit-файл кладём в `/etc/systemd/system/`, например `myapi.service`. Три секции: `[Unit]` — описание и зависимости, `[Service]` — как запускать: пользователь, рабочая директория, команда ExecStart. `[Install]` — когда запускать, обычно `multi-user.target` — после загрузки сети.

Важно: `Restart=always` — при падении процесс перезапустится. `RestartSec=5` — пауза перед перезапуском. После правок — `systemctl daemon-reload`, потом `enable` и `start`.

«Релоаднуть демона» — daemon-reload. «Включить в автозагрузку» — enable.

---

## Слайд 15: Команды systemctl

`daemon-reload` — перечитать конфиги после изменений. `enable` — включить автозапуск. `start`, `stop`, `restart` — управление. `status` — посмотреть, жив ли сервис. Логи — `journalctl -u myapi -f` в реальном времени, `-n 50` — последние 50 строк.

При любых проблемах первым делом — `systemctl status` и `journalctl`. Там почти всегда видно, почему не поднялось.

«Рестартануть» — restart. «Упал», «лег» — сервис перестал работать. «Поднять» — запустить снова. «Журнал», «журналы» — логи. «Журctl», «журналctl» — journalctl. «Повесить -эф» — `journalctl -u myapi -f`, смотреть логи в реальном времени.

---

## Слайд 16: Nginx — зачем нужен?

Nginx — входная точка: принимает запросы на 80 и 443 порты, раздаёт статику и проксирует остальное на бэкенд. Один сервер может обслуживать несколько приложений: FastAPI на одном порту, Django на другом, статика из папки. Nginx ещё занимается SSL, балансировкой, кешированием.

Приложение слушает localhost — снаружи к нему напрямую не обратиться. Nginx — «лицо» сервера, приложение — «внутренний орган». Так и безопаснее, и гибче.

«Энжинкс», «энж», «энжин» — Nginx. «Прокси», «прокся» — reverse proxy. «Проксировать на порт» — направлять трафик на бэкенд.

---

## Слайд 17: Конфигурация Nginx

Конфиг — блок `server` с `listen` и `server_name`. В `location /` — `proxy_pass` на адрес бэкенда. Заголовки `proxy_set_header` передают реальный IP и протокол, чтобы приложение понимало, что запрос пришёл через HTTPS.

Файл кладём в `sites-available`, включаем симлинком в `sites-enabled`. После правок — `nginx -t` (проверка синтаксиса), потом `systemctl reload nginx`. Без `-t` можно случайно сломать конфиг и получить 502 на всём сайте.

«Конфиг» — конфигурационный файл. «Энжикс -тэ» — nginx -t. «Пятисотка», «502» — Bad Gateway, обычно значит, что Nginx не может достучаться до бэкенда.

---

## Слайд 18: Nginx — включение сайта

Два каталога: `sites-available` — все конфиги, `sites-enabled` — активные. Включаем симлинком: `ln -sf ... sites-available/mysite ... sites-enabled/`. Дальше `nginx -t` и `reload`. Никогда не перезагружайте Nginx без проверки — одна опечатка может положить весь веб-сервер.

«Симлинк», «символическая ссылка» — ln -s. «Релоаднуть энж» — systemctl reload nginx.

---

## Слайд 19: DNS — привязка домена

DNS переводит домен в IP. A-запись: имя `@` — корневой домен, `www` — поддомен. Значение — IP вашего VPS. TTL — сколько кешировать запись. После создания DNS может обновляться до 48 часов, обычно — быстрее.

Без DNS придётся давать пользователям голый IP. С доменом и HTTPS — нормальный продукт.

«Днс», «дээнэс» — DNS. «Резолвится» — домен преобразуется в IP. «Пропинговаться» — проверить доступность по ping. «А-запись», «эйка» — A record. «С propagation», «пропагируется» — когда DNS обновляется по всему миру.

---

## Слайд 20: HTTPS с Certbot

Certbot — автоматическое получение SSL от Let's Encrypt. Установка: `apt install certbot python3-certbot-nginx`. Запуск: `certbot --nginx -d mysite.ru -d www.mysite.ru`. Certbot проверит домен, получит сертификат, настроит Nginx, добавит редирект HTTP→HTTPS и автообновление. Всё бесплатно и за пару минут.

Без HTTPS браузеры будут ругаться, а поисковики — понижать в выдаче. Сейчас HTTPS — стандарт, не опция.

«Сертик», «сертификат» — SSL-сертификат. «Летс энкрипт» — Let's Encrypt. «Зелёный замочек» — индикатор HTTPS в браузере. «Сербот» — Certbot.

## Слайд 21: Скрипт деплоя

Вместо ручного `git pull`, `pip install`, `systemctl restart` — один скрипт `deploy.sh`. `set -e` — остановиться при первой ошибке. Потом pull, активация venv, pip install, restart, проверка статуса. Запуск: `chmod +x deploy.sh`, `./deploy.sh`. При добавлении новых шагов — дописываете в скрипт.

Со временем это вырастет в CI/CD, но для старта скрипт уже сильно экономит время и нервы.

«Деплой-скрипт», «деплойник» — скрипт выкатки. «Прогнать деплой» — выполнить выкатку. «Задеплоить», «выкатить» — обновить приложение на сервере. «Пуллить» — git pull.

## Слайд 22: Полная схема деплоя

Итоговая картинка: Nginx на 80/443 принимает трафик, проксирует на Uvicorn на 8010, Uvicorn крутит FastAPI из `/opt/myapi/`. Nginx — из `sites-available`, Uvicorn — через systemd. Всё связано, ничего лишнего. Если что-то падает — идёте по цепочке и ищете слабое звено.

«Стек», «деплой-стек» — вся связка (nginx, app, база). «Крутится» — работает, запущено.

## Слайд 23: Чеклист деплоя

Десять пунктов для проверки: SSH, наличие кода, venv, зависимости, systemd unit, статус сервиса, порт, Nginx, ответ приложения. Перед сдачей проекта или перед демо — прогнать этот список. Сэкономит часы отладки.

«Прогнать чеклист», «проверить по чеклисту» — пройтись по списку. «Демо», «демка» — демонстрация проекта.

## Слайд 24: Полезные команды

Логи приложения — `journalctl -u myapi -f`. Логи Nginx — `tail -f /var/log/nginx/error.log`. Порты — `ss -tlnp`. Диск — `df -h`, память — `free -h`. Проверка сайта — `curl -I https://mysite.ru/`. При проблемах — журналы и порты покажут, где застряло.

«Тейлить логи» — tail -f. «Эсэс», «сс» — ss (замена netstat). «Дф -аш», «дээф» — df -h. «Фри» — free. «Керл» — curl. «Починить», «пофиксить» — исправить проблему.

## Слайд 25: Модальные редакторы

Модальный редактор — несколько режимов: Normal для навигации, Insert для ввода, Visual для выделения, Command для команд. Vim, Neovim, Helix. Vim есть на любом Linux-сервере — если нет GUI, а править конфиг надо, вариантов немного. Навык инвестиция: один раз разобрался — пользуешься годами.

«Вим», «вима» — Vim (произносится «вим», не «вай-ай-эм»). «Вимер» — тот, кто пользуется Vim. «Выйти из вима» — шутка-мем, потому что новички не знают :q. «Нювим», «нвим» — Neovim.

---

## Слайд 26: Neovim — быстрый старт

`apt install neovim`, `nvim filename`. Минимум: `i` — ввод, Esc — выход из ввода, `:wq` — сохранить и выйти, `:q!` — выйти без сохранения. Write, quit, insert — мнемоника простая. Остальное — по мере надобности.

«Вай-кью», «вкью» — :wq. «Кью-фак», «кью банг» — :q!. «Ай» — i, войти в режим ввода. «Эскейпнуться» — нажать Esc, выйти из Insert.

## Слайд 27: Навигация в Neovim

hjkl — влево, вниз, вверх, вправо. w, b — по словам. 0, $ — начало и конец строки. gg, G — начало и конец файла. Стрелки тоже работают, но hjkl быстрее и не требуют переноса руки. Сначала кажется странным, потом входит в мышечную память.

«Эйч-джей-кей-эль», «джи-кей» — hjkl. «Дабл-джи» — gg, в начало файла. «Большая джи» — G, в конец. «Вим-гольф» — минимизация нажатий при редактировании, спорт вимеров.

## Слайд 28: Редактирование в Neovim

i, a, o, O, A — разные варианты входа в Insert. x, dd, dw, d$ — удаление. u — отмена, Ctrl+r — повторить. Этого хватает для правки конфигов и логов. Остальное — когда понадобится.

«Дэ-дэ» — dd, удалить строку. «Дэ-дабл-ю» — dw, удалить слово. «Андо», «ундо» — u, отмена. «Ридо» — Ctrl+r, повторить.

## Слайд 29: Практика Neovim за 15 минут

Откройте `nvim`, наберите `:Tutor` — встроенный туториал. Потом попробуйте отредактировать конфиг Nginx или посмотреть лог. Есть Vim Adventures и OpenVim для тех, кому нравится учиться через игру. Минимум для деплоя: открыть, ввести `i`, поправить, Esc, `:wq`. Остальное — бонусом.

«Тьютор» — :Tutor. «Войти, поправить, вай-кью» — минимальный скилл для правки конфигов на сервере. «Вимадвенчерс», «опенвим» — игры и туториалы для обучения.
