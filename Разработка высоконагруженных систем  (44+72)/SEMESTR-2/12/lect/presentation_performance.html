Настройка производительности FastAPI и Django. Деплой и тестирование на PaaS.

Архитектурные метрики

  • RPS / QPS — запросы в секунду
  • Peak RPS — пиковая нагрузка
  • Read/Write ratio — соотношение чтения и записи
  • Payload size (avg / p95) — размер полезной нагрузки (средний / 95‑й перцентиль)

Масштабируемость

  • Horizontal scalability (можно ли масштабироваться без боли)
  • Statefulness (stateless vs stateful) — состояние (без состояния vs с хранением состояния)
  • Fan-out — сколько RPC вызывает один запрос

Детали по ⭐ FastAPI

  • Очень высокие показатели throughput и низкая задержка — часто в 3–5× быстрее, чем Django/DRF
  • Асинхронность встроена по умолчанию → эффективное использование событийного цикла
  • Отлично подходит для микросервисов, real-time API, WebSockets, высоких нагрузок

На VPS 2 ядра, 4 ГБ FastAPI обычно устойчиво обрабатывает десятки тысяч запросов в секунду при простых API.

Детали по 🐍 Django + DRF

  • Built-in Django ORM, но DRF значительно увеличивает overhead из-за сериализации и middleware
  • Производительность ощутимо ниже FastAPI: типично в 2–5× медленнее
  • DRF чаще всего синхронен — асинхронные преимущества Django 6 на него почти не распространяются

💡 Подходит для проектов, где важны экосистема Django, встроенные инструменты (auth, admin).

Детали по ⚡ Django + Ninja

  • Django Ninja быстрее DRF и близок к производительности FastAPI
  • Pydantic-сериализация и лёгкая структура DSL-маршрутов — меньший overhead

📌 Хороший компромисс: сохраняешь ORM и многое из Django, но с высокой скоростью API.

Детали по 🏁 Django (чистый, без DRF/Ninja)

  • При обычных веб-эндпоинтах и шаблонах — вполне достаточен для страниц, admin-панелей
  • В API-контексте синхронный характер и нагрузка middleware дают более низкие RPS по сравнению с FastAPI

📌 Почему Django 6 упоминается отдельно

Django 6 расширяет поддержку async, но большинство API-слоёв (особенно DRF) по-прежнему работают синхронно, поэтому чистый прирост скорости ограничен по сравнению с фреймворками, изначально построенными на ASGI.

🎯 Цель по железу (2 ядра / 4 GB)

Реалистичные ориентиры для Django API:

  • Django + Ninja: 15–30k RPS (без тяжёлой БД)
  • Django + DRF: 7–15k RPS
  • Средний p95 latency: 30–120 ms
  • CPU ≈ 70–85% под нагрузкой, RAM ≈ 1.5–2.5 GB

ASGI + Gunicorn (критично)

Почему не WSGI?

  • WSGI — блокирующая модель → хуже масштабируется
  • ASGI — событийная модель → лучше загрузка CPU
pip install gunicorn uvicorn
bind = "127.0.0.1:8000"
workers = 4
worker_class = "uvicorn.workers.UvicornWorker"
threads = 1
timeout = 30
max_requests = 10000
max_requests_jitter = 1000

Для IO-bound: workers = CPU × 2

Запуск

gunicorn config.asgi:application \
  --config gunicorn.conf.py

Nginx (огромный буст)

worker_processes auto;
worker_connections 8192;
multi_accept on;
upstream django {
    server 127.0.0.1:8000;
    keepalive 64;
}

location / {
    proxy_pass http://django;

}

Схемы (Pydantic)

class UserCreate(Schema):
    email: str
    name: str
    password: str

class UserOut(Schema):
    id: int
    email: str
    name: str

In — валидация входящих данных

Out — контроль, что уходит клиенту (без пароля)

CRUD и ORM

@router.get("/users", response=List[UserOut])
def list_users(request, limit: int = 20, offset: int = 0):
    return list(User.objects.all()[offset:offset+limit])

@router.post("/users", response=UserOut)
def create_user(request, payload: UserCreate):
    return User.objects.create(**payload.dict())

Оптимизация: select_related("profile").prefetch_related("orders")

Валидация и обработка ошибок

  • Pydantic проверяет типы → при ошибке 422 Unprocessable Entity
@router.get("/users/{user_id}")
def get_user(request, user_id: int):
    try:
        return User.objects.get(id=user_id)
    except User.DoesNotExist:
        return Response({"detail": "Not found"}, status=404)
api.add_exception_handler(ValueError, custom_exception_handler)

Async vs Sync

@router.get("/users/async")
async def list_users_async(request):
    return [u async for u in User.objects.all()]

Async даёт выигрыш при IO-bound (БД, внешние API)

Django 4.1+ поддерживает sync_to_async для ORM

⚠️ ASGI-сервер (Uvicorn) нужен для async

    response = client.get("/hello")
    assert response.status_code == 200
