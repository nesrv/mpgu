## Слайд 1 — Архитектурные метрики (3 мин)

Добрый день, коллеги. Сегодня мы поговорим о том, как выжать максимум из Python-бэкенда и не облажаться при деплое. Но прежде чем лезть в код — давайте разберёмся с метриками. Потому что если вы не можете измерить производительность, вы не можете её улучшить.

**RPS** — Requests Per Second, или QPS (Queries Per Second) — это ваш главный KPI. Сколько запросов в секунду обрабатывает система. Когда продакт спрашивает «а сколько юзеров выдержит?» — он на самом деле спрашивает про RPS. Типичное веб-приложение на старте — это 100-500 RPS. Когда говорят «хайлоад» — речь обычно о тысячах и десятках тысяч RPS.

**Peak RPS** — пиковая нагрузка. Это когда в Black Friday все ломятся на сайт, или когда ваш пост завирусился. Peak может быть в 10-100 раз выше обычной нагрузки. И если вы не готовы — лежите. Классика: «сайт упал от наплыва посетителей». Это значит, что кто-то не подумал о peak RPS.

**Read/Write ratio** — соотношение операций чтения и записи. В типичном веб-приложении это 80/20 или даже 95/5. То есть люди гораздо чаще читают, чем пишут. Это критически важно для архитектуры: read-heavy системы можно агрессивно кэшировать, а write-heavy требуют другого подхода — очереди, батчинг, асинхронная запись.

**Payload size** — размер полезной нагрузки. Мы смотрим на средний размер и на p95 (95-й перцентиль). Почему p95? Потому что среднее — это как «средняя температура по больнице». Вам важно знать, что 95% запросов укладываются в определённый размер, а 5% — это аномалии, которые можно обрабатывать отдельно.

Запомните эти четыре метрики — это ваш baseline для любого разговора о производительности.

## Слайд 2 — Масштабируемость (3 мин)

Теперь поговорим о масштабируемости. Это не про «купить сервер побольше» — это про архитектурные решения, которые определяют, сможете ли вы вообще расти.

**Horizontal scalability** — горизонтальное масштабирование. Это когда вы добавляете ещё серверов, а не наращиваете мощность одного. В идеале — линейное масштабирование: 2 сервера = 2x производительность. На практике так не бывает из-за накладных расходов на координацию, но к этому надо стремиться.

Почему горизонтальное лучше вертикального? Потому что вертикальное имеет потолок — есть самый мощный сервер в мире, и дальше некуда. А горизонтальное — бесконечно, теоретически. Плюс это дешевле: 10 маленьких серверов часто стоят меньше, чем один большой с 10x мощностью.

**Statefulness** — состояние. Это ключевой вопрос. Stateless-сервис не хранит ничего между запросами. Каждый запрос — как первый. Stateful — хранит сессию, кэш в памяти, что-то ещё.

Почему stateless — это хорошо? Потому что любой инстанс может обработать любой запрос. Упал один — не страшно, другой подхватит. Нужно больше мощности — добавляем инстансы. Stateful-сервисы масштабировать больно: нужна sticky session, репликация состояния, всё сложно.

Практический совет: сессии — в Redis, кэш — в Redis, всё состояние — наружу. Сам сервис — stateless.

**Fan-out** — сколько внутренних вызовов генерирует один внешний запрос. Если пользователь дёргает один эндпоинт, а тот внутри делает 15 запросов к разным сервисам и базам — это fan-out 15. Чем выше fan-out, тем выше latency и тем хрупче система. Один из 15 сервисов тормозит — и весь запрос тормозит.

## Слайд 3 — Сравнение производительности Python-фреймворков (3 мин)

Окей, теперь давайте посмотрим на конкретные цифры. У нас есть VPS с 2 ядрами и 4 гигами RAM — типичная конфигурация для небольшого проекта или стейджинга.

Смотрим таблицу. **FastAPI** — король производительности в Python-мире. 20-40 тысяч RPS на простых API. Почему? Потому что он построен на ASGI и Starlette, асинхронный по умолчанию, минимальный overhead. Если вам нужна скорость и вы не привязаны к Django-экосистеме — это ваш выбор.

**Django без API-слоя** — 1-5 тысяч RPS. Это чистый Django с views. Не плохо для рендеринга HTML, админки, но для API — не оптимально. Много middleware, ORM добавляет overhead.

**Django + DRF** — Django REST Framework. Тут интересно: 2-12 тысяч RPS. Разброс большой, потому что сильно зависит от того, как вы написали сериализаторы. Если там SerializerMethodField на каждое поле и N+1 запросы — будет 2k. Если оптимизировали — может быть и 10k+.

**Django + Ninja** — а вот это интересно! Почти FastAPI-уровни производительности, но с Django ORM, админкой и всей экосистемой. Как так? Ninja — это легковесная обёртка с Pydantic-сериализацией, без тяжёлого middleware DRF. По сути, вы получаете FastAPI-подобный API, но внутри Django.

**Django 6** — последняя версия с улучшенной async-поддержкой. Но тут важно понимать: сам Django стал чуть быстрее, но DRF всё ещё синхронный. Так что если вы на DRF — особого прироста не ждите.

Главный вывод: выбор фреймворка — это не про «что модно», а про ваши требования. FastAPI — для скорости. Django + DRF — для экосистемы. Django + Ninja — компромисс.

## Слайд 4 — Детали по FastAPI (3 мин)

Давайте глубже копнём в FastAPI, потому что понимание того, почему он быстрый, поможет вам писать быстрый код на любом фреймворке.

FastAPI — в 3-5 раз быстрее Django/DRF на простых эндпоинтах. Это не маркетинг, это бенчмарки. Почему?

Первое — **асинхронность по умолчанию**. Когда вы пишете `async def endpoint()`, вы не блокируете воркер на время I/O-операций. Пока ждёте ответа от базы — можете обрабатывать другие запросы. Это event loop, это эффективное использование ресурсов.

Второе — **минимальный overhead**. FastAPI — это тонкая обёртка над Starlette. Нет тяжёлого middleware-стека, нет ORM по умолчанию, нет сессий. Только то, что вам нужно.

Третье — **Pydantic**. Валидация и сериализация через Pydantic написаны на Rust (pydantic-core в v2), это очень быстро. DRF-сериализаторы на чистом Python — медленнее в разы.

Когда выбирать FastAPI? **Микросервисы** — однозначно. Маленький сервис, делает одну вещь, делает быстро. **Real-time API** — WebSockets, Server-Sent Events. **Высокие нагрузки** — когда RPS измеряется десятками тысяч.

На VPS 2 ядра, 4 GB FastAPI устойчиво держит десятки тысяч RPS. Но это «на простых API» — без тяжёлых запросов к базе. Как только добавляете реальную бизнес-логику и базу — цифры падают. Это нормально, просто помните: бенчмарки и реальность — разные вещи.

Важный момент: deployment. FastAPI запускается через Uvicorn или Gunicorn с Uvicorn-воркерами. Не через WSGI — это важно.

## Слайд 5 — Детали по Django + DRF (3 мин)

Теперь Django REST Framework. Да, он медленнее. Но его используют везде, и на это есть причины.

DRF даёт **огромный overhead**. Сериализация, ViewSets, роутеры, permissions, throttling, pagination — всё это стоит процессорного времени. Простой GET-запрос проходит через кучу слоёв, прежде чем вернуть JSON.

По бенчмаркам DRF в 2-5 раз медленнее FastAPI. На практике разница может быть ещё больше, если вы не оптимизируете.

Типичные грабли в DRF:

**SerializerMethodField** — каждый такой филд выполняет Python-код для каждого объекта. Если у вас список из 100 объектов и 5 method fields — это 500 вызовов функций. Используйте `.annotate()` на уровне QuerySet.

**N+1 queries** — классика. `select_related()` для ForeignKey, `prefetch_related()` для ManyToMany. Без этого каждый объект в списке генерирует отдельный запрос к базе.

**Пагинация** — если она вам не нужна, отключите. Каждая пагинация — это COUNT-запрос к базе.

Когда выбирать DRF? Когда вам важна **экосистема Django**. Admin-панель из коробки, auth, permissions, тысячи готовых пакетов. Когда нужен **Browsable API** — для документации и отладки. Когда команда уже знает Django и переучиваться накладно.

DRF — это зрелый, проверенный инструмент. Он не быстрый, но он надёжный и предсказуемый. Для многих проектов этого достаточно.

## Слайд 6 — Детали по Django + Ninja (3 мин)

А теперь моя личная рекомендация для тех, кто хочет и скорость, и Django — **Django Ninja**.

По бенчмаркам Ninja **быстрее DRF** и близок к FastAPI. Как это возможно? Всё просто: Ninja — это минималистичная обёртка. Нет тяжёлых ViewSets, нет сложной сериализации. Pydantic для валидации, простые функции-хендлеры.

Сравните:

```python
# DRF
class UserViewSet(viewsets.ModelViewSet):
    queryset = User.objects.all()
    serializer_class = UserSerializer
    permission_classes = [IsAuthenticated]
```

```python
# Ninja
@api.get("/users", response=List[UserSchema], auth=ApiKey())
def list_users(request):
    return User.objects.all()
```

Ninja-код проще, читается легче, работает быстрее. Вы сохраняете Django ORM, можете использовать Django-модели, admin-панель работает, миграции работают — всё как обычно. Но API-слой — быстрый.

Когда Ninja — хороший выбор? Когда у вас уже Django-проект и нужно добавить быстрый API. Когда хотите FastAPI-подобный синтаксис, но не хотите переезжать с Django. Когда DRF тормозит, а рефакторить на FastAPI — дорого.

Практический момент: Ninja отлично работает с ASGI. То есть можно запустить `gunicorn config.asgi:application -k uvicorn.workers.UvicornWorker` и получить async-преимущества.

На практике Django + Ninja — это хороший компромисс: сохраняешь экосистему, получаешь скорость.

## Слайд 7 — Django без API-слоя (3 мин)

Окей, а что если вообще не использовать DRF или Ninja? Просто Django views с `JsonResponse`?

Для **обычных веб-страниц** — вполне достаточно. Django отлично рендерит HTML, у него мощный template engine, формы работают из коробки. Для админок, dashboard'ов, внутренних инструментов — идеально.

Но для **API** — не оптимально. Почему?

Во-первых, **middleware**. Стандартный Django идёт с кучей middleware: CSRF, sessions, messages, locale. Каждый middleware — это код, который выполняется на каждый запрос. Для HTML-страниц это нужно, для API — лишний overhead.

Во-вторых, **синхронность**. Классические Django views синхронные. Пока view ждёт ответа от базы данных — воркер заблокирован. Это неэффективно.

В-третьих, **нет удобств API**. Сериализация руками, валидация руками, документация руками. DRF и Ninja дают это из коробки.

Цифры: чистый Django — 2-10 тысяч RPS в зависимости от сложности. Это не плохо, но и не хайлоад.

Когда использовать чистый Django без API-слоя? **Монолитные веб-приложения** с серверным рендерингом. **Admin-панели** и внутренние инструменты. **Прототипы**, где скорость разработки важнее производительности.

Но если вы строите API, который будут дёргать мобильные приложения, фронтенд на React, или другие сервисы — используйте DRF или Ninja. Это правильный инструмент для задачи.

## Слайд 8 — Примерные ориентиры RPS (3 мин)

Давайте зафиксируем цифры для ваших ожиданий. Это обобщение бенчмарков 2025-2026 годов, простые GET/POST JSON API без тяжёлой логики.

**FastAPI**: 20-40 тысяч RPS, latency 10-30 ms. Это потолок для Python. Быстрее — только Go, Rust или другие компилируемые языки.

**Django + Ninja**: 15-30 тысяч RPS, latency 15-40 ms. Очень близко к FastAPI. Разница в 20-30% — это плата за Django ORM и middleware.

**Django + DRF**: 7-15 тысяч RPS, latency 50-120 ms. Вдвое медленнее Ninja. Если вы на DRF и вам хватает — отлично. Если не хватает — миграция на Ninja несложная.

**Чистый Django**: 2-10 тысяч RPS, latency 80-150 ms. Самый медленный вариант для API.

Важные оговорки:

Эти цифры — для **простых эндпоинтов**. Как только добавляете сложные SQL-запросы, внешние API-вызовы, тяжёлую бизнес-логику — RPS падает в разы.

**Реальные цифры зависят** от кода, базы данных, кэширования, deployment. Я видел Django + DRF, который выдавал 500 RPS из-за кривых запросов. И видел его же на 15k после оптимизации.

**База данных** — обычно bottleneck. Не фреймворк тормозит, а ваши SQL-запросы. Профилируйте.

Используйте эти цифры как ориентир, но всегда замеряйте свою систему. `wrk`, `ab`, `locust` — ваши друзья.

## Слайд 9 — Практические соображения при выборе (3 мин)

Теперь давайте к практике. Как выбрать фреймворк для вашего проекта?

**Сценарий 1: Максимальная пропускная способность**

Вам нужен API, который выдержит десятки тысяч RPS. Пользователей много, latency критична. Выбор: **FastAPI** или **Django + Ninja**.

FastAPI — если начинаете с нуля и не нужна Django-экосистема. Ninja — если уже на Django или нужны ORM, admin, auth из коробки.

Обязательно: ASGI, Gunicorn + Uvicorn, Redis для кэша, PostgreSQL с индексами.

**Сценарий 2: Функциональность и экосистема**

Вам нужен Browsable API, сложные permissions, готовые решения для auth, интеграции с кучей пакетов. Скорость — вторична, главное — скорость разработки.

Выбор: **Django + DRF**. Он зрелый, документированный, под него всё есть. django-filter, drf-yasg, djoser — тысячи готовых решений.

**Сценарий 3: Баланс скорости и удобства**

Хотите и быстро работать, и быстро писать. Не хотите жертвовать ни производительностью, ни удобством.

Выбор: **Django + Ninja**. Это компромисс. FastAPI-подобный синтаксис, Django-экосистема, приличная скорость.

**Главный принцип**: выбирайте инструмент под задачу, а не под хайп. Если у вас 1000 пользователей в день — производительность вообще не проблема. Если миллион — тогда да, думайте.

## Слайд 10 — Почему Django 6 отдельно (3 мин)

Вы могли слышать про Django 6 и его «революционную async-поддержку». Давайте разберёмся, что это значит на практике.

Django 6 действительно расширяет async. Теперь больше частей фреймворка работают асинхронно: ORM-операции, middleware, некоторые встроенные функции.

Но вот проблема: **большинство API-слоёв по-прежнему синхронные**.

DRF — синхронный. Его ViewSets, сериализаторы, permissions — всё sync. Когда DRF станет async? Неизвестно. Разработчики работают над этим, но это сложная задача.

Поэтому если вы на Django + DRF и обновились до Django 6 — **особого прироста скорости не ждите**. Ваш код всё ещё синхронный, воркеры всё ещё блокируются.

Где Django 6 async реально помогает? В чистых Django views без DRF. В Django Ninja, который поддерживает async. В кастомном коде, где вы явно используете `async def`.

Практический совет: не обновляйтесь на Django 6 ради производительности, если вы на DRF. Обновляйтесь ради безопасности, новых фич, но не ждите магического ускорения.

Если хотите async-преимущества — переходите на Ninja или FastAPI. Там это работает из коробки.

## Слайд 11 — Целевые показатели для 2 CPU / 4 GB (3 мин)

Давайте зафиксируем, чего реально можно добиться на типичном VPS: 2 ядра, 4 гигабайта RAM. Это ваш бюджетный сервер, стейджинг, или небольшой продакшен.

**Django + Ninja**: 15-30 тысяч RPS без тяжёлых запросов к базе. Это если вы отдаёте закэшированные данные или простые CRUD-операции.

**Django + DRF**: 7-15 тысяч RPS. С оптимизациями ближе к 15k, без них — ближе к 7k или ниже.

**Средний p95 latency**: 30-120 ms. p95 — это когда 95% запросов быстрее этого значения. Остальные 5% — могут быть медленнее, это нормально.

**Ресурсы под нагрузкой**: CPU на 70-85%, RAM на 1.5-2.5 GB. Если CPU выше 90% постоянно — вы на пределе, нужно масштабироваться. Если RAM под потолок — смотрите утечки памяти.

Важное уточнение: эти цифры — для **оптимизированного** приложения. DEBUG=False, правильное количество воркеров, индексы в базе, кэш настроен.

Если у вас на этом железе 500 RPS — значит где-то узкое место. Профилируйте, ищите. Обычно это N+1 queries, тяжёлые сериализаторы, или просто DEBUG=True.

Для реального продакшена с высокой нагрузкой 2 ядра — мало. Планируйте 4-8 ядер минимум, плюс отдельный сервер для базы данных.

## Слайд 12 — Архитектура: обязательный минимум (3 мин)

Теперь к архитектуре. Вот минимальный стек для продакшена:

```
Client → Nginx → Gunicorn (workers) → Django (ASGI) → PostgreSQL / Redis
```

Почему именно так?

**Nginx** впереди — это не опционально. Nginx отдаёт статику (CSS, JS, картинки) без нагрузки на Python. Nginx буферизует медленных клиентов — если кто-то на 2G качает ответ, Gunicorn не ждёт. Nginx делает SSL-терминацию, rate limiting, gzip.

Без Nginx ваш Gunicorn напрямую смотрит в интернет — это плохо. Медленные клиенты блокируют воркеры, DoS-атаки кладут сервер.

**Gunicorn** — application server. Он управляет воркерами, перезапускает упавшие, распределяет нагрузку. Для ASGI используем UvicornWorker.

**Django ASGI** — асинхронная версия Django. Важно: не WSGI, а ASGI. Разница — как между однопоточным и многопоточным приложением. ASGI эффективнее использует ресурсы.

**PostgreSQL** — база данных. Не SQLite, не MySQL. PostgreSQL — стандарт для серьёзных проектов. Индексы, JSONB, полнотекстовый поиск — всё есть.

**Redis** — кэш, сессии, очереди. Без Redis вы каждый раз ходите в PostgreSQL. С Redis — достаёте из памяти за микросекунды.

Эта архитектура масштабируется горизонтально: добавляете сервера с Nginx/Gunicorn за load balancer, база и Redis — общие.

## Слайд 13 — ОС и базовая оптимизация VPS (3 мин)

Переходим к настройке сервера. Начинаем с операционной системы — Ubuntu 22.04 или 24.04, это стандарт.

Устанавливаем базовые пакеты:

```bash
sudo apt update && sudo apt upgrade -y
sudo apt install -y python3-pip python3-venv nginx redis-server postgresql postgresql-contrib build-essential
```

Здесь всё, что нужно: Python, веб-сервер, кэш, база данных, компиляторы для pip-пакетов.

Теперь **критически важная** вещь — настройка sysctl. Это системные параметры ядра, которые влияют на производительность сети.

```ini
net.core.somaxconn = 65535
net.ipv4.tcp_tw_reuse = 1
net.ipv4.ip_local_port_range = 1024 65000
fs.file-max = 100000
```

Что это?

**somaxconn** — максимальная длина очереди соединений. По умолчанию 128 — это мало. При высокой нагрузке соединения будут дропаться.

**tcp_tw_reuse** — повторное использование TIME_WAIT сокетов. Ускоряет установку соединений.

**ip_local_port_range** — диапазон портов для исходящих соединений. Нужно для высокой конкурентности.

**file-max** — максимум открытых файлов. Каждое соединение — это файл в Linux.

Применяем:

```bash
sudo sysctl -p
```

Без этих настроек на высокой нагрузке будут странные ошибки: «connection refused», «too many open files». Лучше сделать сразу.

## Слайд 14 — Django: ключевые настройки (3 мин)

```python

```

## Слайд 15 — ASGI + Gunicorn: почему не WSGI (3 мин)

Теперь поговорим о том, как запускать Django. Есть два варианта: WSGI и ASGI. Разница — принципиальная.

**WSGI** — Web Server Gateway Interface. Синхронная, блокирующая модель. Один воркер обрабатывает один запрос. Пока ждёт ответа от базы — заблокирован, ничего не делает.

**ASGI** — Asynchronous Server Gateway Interface. Асинхронная, событийная модель. Один воркер может обрабатывать тысячи запросов одновременно. Пока один запрос ждёт базу — обрабатывает другие.

Аналогия: WSGI — это ресторан, где один официант обслуживает один столик до конца. ASGI — это официант, который принимает заказы у нескольких столиков, пока кухня готовит.

Для I/O-bound нагрузки (а это почти любой веб — ходим в базу, в Redis, во внешние API) ASGI значительно эффективнее.

Устанавливаем:

```bash
pip install gunicorn uvicorn
```

**Gunicorn** — менеджер процессов. **Uvicorn** — ASGI-сервер. Gunicorn запускает несколько Uvicorn-воркеров.

Почему не просто Uvicorn? Gunicorn умеет перезапускать воркеры, управлять памятью, логировать. Uvicorn — только обработка запросов.

Связка Gunicorn + UvicornWorker — это стандарт для продакшена Django с ASGI.

## Слайд 16 — Конфигурация Gunicorn и запуск (3 мин)

Создаём файл `gunicorn.conf.py`:

```python
bind = "127.0.0.1:8000"
workers = 4
worker_class = "uvicorn.workers.UvicornWorker"
threads = 1
timeout = 30
keepalive = 5
max_requests = 10000
max_requests_jitter = 1000
accesslog = "-"
errorlog = "-"
loglevel = "warning"
```

Разберём каждый параметр.

**bind** — слушаем localhost:8000. Nginx будет проксировать сюда.

**workers = 4** — количество процессов. Для IO-bound нагрузки правило: CPU × 2. У нас 2 ядра, значит 4 воркера.

**worker_class** — используем UvicornWorker для ASGI.

**threads = 1** — для ASGI потоки не нужны, асинхронность внутри воркера.

**timeout = 30** — если запрос обрабатывается дольше 30 секунд, воркер убивается. Защита от зависших запросов.

**keepalive = 5** — держим соединение открытым 5 секунд для повторных запросов.

**max_requests = 10000** — после 10000 запросов воркер перезапускается. Защита от утечек памяти.

**max_requests_jitter = 1000** — случайный разброс, чтобы все воркеры не рестартовали одновременно.

Запуск:

```bash
gunicorn config.asgi:application --config gunicorn.conf.py
```

Обратите внимание: `config.asgi:application`, не `config.wsgi`. Это важно!

## Слайд 17 — Nginx: огромный буст производительности (3 мин)

Nginx — это не просто «поставить перед Django». Это реально большой прирост производительности.

Базовая конфигурация `/etc/nginx/nginx.conf`:

```nginx
worker_processes auto;
worker_connections 8192;
multi_accept on;
```

**worker_processes auto** — по количеству ядер автоматически.

**worker_connections 8192** — сколько соединений может держать один воркер. По умолчанию 512 — мало.

**multi_accept on** — принимать несколько соединений за раз.

Site config для Django:

```nginx
upstream django {
    server 127.0.0.1:8000;
    keepalive 64;
}

server {
    listen 80;
    server_name your-domain.com;

    location / {
        proxy_pass http://django;
        proxy_http_version 1.1;
        proxy_set_header Connection "";
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $remote_addr;
        proxy_read_timeout 30;
    }
}
```

Ключевое здесь — **keepalive 64**. Это persistent connections между Nginx и Gunicorn. Без этого на каждый запрос — новое TCP-соединение, это дорого.

`proxy_http_version 1.1` и `Connection ""` — нужны для keepalive.

Результат: **+20-30% RPS** просто за счёт правильной конфигурации Nginx. Это не шутка, это реальные цифры из бенчмарков.

## Слайд 18 — PostgreSQL: узкое место №1 (3 мин)

Запомните: в 80% случаев тормозит не Django, не Gunicorn — тормозит база данных. PostgreSQL — это обычно bottleneck.

Базовая настройка `/etc/postgresql/*/main/postgresql.conf`:

```ini
shared_buffers = 1GB
effective_cache_size = 3GB
work_mem = 16MB
maintenance_work_mem = 256MB
max_connections = 100
```

**shared_buffers** — память для кэша данных. Правило: 25% от RAM. У нас 4GB → 1GB.

**effective_cache_size** — сколько памяти PostgreSQL может рассчитывать для кэширования (включая OS cache). 75% от RAM.

**work_mem** — память на одну операцию сортировки/хэширования. Осторожно: умножается на количество соединений.

**max_connections** — с CONN_MAX_AGE=60 в Django вам не нужно много. 100 хватит для большинства случаев.

Теперь **индексы** — это критически важно:

```sql
CREATE INDEX CONCURRENTLY idx_user_email ON users(email);
```

Без индексов SELECT по email на миллионе записей — секунды. С индексом — миллисекунды.

**CONCURRENTLY** — создаём индекс без блокировки таблицы. В продакшене — только так.

Практический совет: включите `log_min_duration_statement = 100` — будет логировать все запросы дольше 100ms. Это ваш источник информации о медленных запросах.

80% «медленного Django» — это плохие SQL-запросы. Исправьте их, и всё полетит.

## Слайд 19 — Redis: обязательно для скорости (3 мин)

Redis — in-memory key-value хранилище. Скорость — микросекунды. Используйте его для:

**Кэш** — самое очевидное. Результаты тяжёлых запросов, данные, которые редко меняются.

```python
from django.core.cache import cache
data = cache.get("expensive_query")
if not data:
    data = expensive_query()
    cache.set("expensive_query", data, timeout=300)
```

**Rate limiting** — ограничение количества запросов. Храним счётчики в Redis.

**Session storage** — если у вас stateless-архитектура, сессии должны быть вне воркера. Redis — идеально.

Настройка в `settings.py`:

```python
CACHES = {
    "default": {
        "BACKEND": "django_redis.cache.RedisCache",
        "LOCATION": "redis://127.0.0.1:6379/1",
        "OPTIONS": {
            "CLIENT_CLASS": "django_redis.client.DefaultClient"
        },
    }
}
```

Пакет `django-redis` — ставить обязательно.

Практические советы:

**TTL** (Time To Live) — всегда ставьте. Без TTL кэш растёт бесконечно.

**Key naming** — используйте понятные префиксы: `user:123:profile`, `product:456:price`.

**Инвалидация** — продумайте, когда кэш сбрасывать. Это сложнее, чем кажется. «There are only two hard things in CS: cache invalidation and naming things.»

## Слайд 20 — Django Ninja vs DRF: разница в коде (3 мин)

Давайте сравним, как выглядит код на Ninja и DRF.

**Django + Ninja:**

```python
@api.get("/users")
def users(request):
    return list(User.objects.values("id", "email"))
```

Три строчки. Функция, декоратор, return. Pydantic валидирует, Ninja сериализует.

**Django + DRF:**

```python
class UserSerializer(serializers.ModelSerializer):
    class Meta:
        model = User
        fields = ['id', 'email']

class UserViewSet(viewsets.ModelViewSet):
    queryset = User.objects.all()
    serializer_class = UserSerializer
```

Больше кода, больше классов, больше абстракций.

В чём преимущества Ninja?

✔ **Pydantic** — быстрая валидация, автодокументация.
✔ **Минимальный overhead** — нет тяжёлых абстракций DRF.
✔ **Async-friendly** — легко писать async-эндпоинты.

Когда DRF всё ещё нужен?

- **SerializerMethodField** — хотя лучше избегать.
- **Nested сериализаторы** — DRF делает это удобнее.
- **Browsable API** — для документации и отладки.
- **Готовые ViewSets** — если нужен стандартный CRUD без кастомизации.

Если выбираете Ninja:

```python
User.objects.select_related().prefetch_related()
```

Всегда используйте `select_related()` и `prefetch_related()`. Ninja не защитит вас от N+1 — это всё ещё Django ORM.

## Слайд 22 — Частые ошибки, убивающие производительность (3 мин)

Теперь к антипаттернам. Вот что гарантированно убивает вашу производительность:

❌ **DEBUG=True** — утечка памяти, медленное логирование, полный traceback клиентам. В продакшене — никогда.

❌ **WSGI вместо ASGI** — синхронная модель, воркеры блокируются на I/O. Теряете 50-70% производительности.

❌ **1 worker** — один воркер = один одновременный запрос. Все остальные ждут в очереди.

❌ **ORM в цикле** — классический N+1:

```python
for user in users:
    print(user.profile.name)  # +1 запрос на каждого user
```

Используйте `select_related()` и `prefetch_related()`.

❌ **Отсутствие индексов** — SELECT по неиндексированному полю на миллионе записей = секунды. С индексом = миллисекунды.

❌ **Нет Redis** — каждый раз в базу за данными, которые можно закэшировать.

❌ **Gunicorn без max_requests** — память течёт, воркеры разбухают, OOM killer приходит.

Каждая из этих ошибок может стоить 2-10x производительности. Комбинация нескольких — и у вас 100 RPS вместо 10000.

Чеклист перед деплоем: пройдитесь по этому списку. Серьёзно.

## Слайд 23 — Краткий рецепт «максимум из Django» (3 мин)

Окей, давайте резюмируем. Если хотите выжать максимум — вот чеклист:

**ASGI + Gunicorn + UvicornWorker** — не WSGI. Это база.

**4 workers** — для 2 ядер. Формула: CPU × 2.

**Nginx keepalive** — persistent connections между Nginx и Gunicorn. +20-30% RPS.

**PostgreSQL с индексами** — профилируйте запросы, индексируйте популярные поля.

**Redis** — кэш, сессии, rate limiting. Без Redis далеко не уедете.

**Django Ninja вместо DRF** — если производительность критична. Ninja быстрее.

**Минимум middleware** — убирайте всё, что не используете.

Это не rocket science. Это стандартные практики, которые дают предсказуемый результат.

На VPS 2 ядра / 4 GB с этими настройками Django + Ninja выдаст 15-30k RPS. Django + DRF — 8-15k. Этого достаточно для большинства проектов.

Если не достаточно — горизонтальное масштабирование: больше серверов, load balancer. Но сначала убедитесь, что вы выжали максимум из одного сервера.

## Слайд 24 — Разбор конфигурации WSGI (3 мин)

Теперь давайте разберём конкретный пример. Вот команда запуска:

```bash
gunicorn config.wsgi:application \
  --bind 127.0.0.1:8000 \
  --workers 3 \
  --threads ?
```

Что это означает?

**config.wsgi:application** — WSGI, не ASGI. Синхронная модель.

**3 workers** — три процесса.

**threads ?** — если не указано, Gunicorn ставит `threads = 1`.

**Итого: 3 workers × 1 thread = 3 одновременных запроса.**

Три. Всего три запроса параллельно. Остальные — в очереди.

Это не масштабируется. Если приходит 100 запросов в секунду, и каждый обрабатывается 100ms, вам нужно минимум 10 одновременных слотов. А у вас 3.

На 2 ядрах это недоиспользование ресурсов. CPU будет простаивать, пока воркеры ждут I/O.

Почему так делают? Иногда по незнанию, иногда «работает и ладно». Для админки с 10 пользователями — сойдёт. Для API под нагрузкой — нет.

Давайте посмотрим, что реально получается с такой конфигурацией.

## Слайд 25 — Производительность WSGI с threads=1 (3 мин)

Итак, `--workers 3 --threads 1` — три одновременных запроса.

Реалистичные цифры на 2 CPU / 4 GB:

| Тип эндпоинта         | RPS          | p95 latency |
|-----------------------|--------------|-------------|
| Простой JSON (без БД) | 2-4k RPS     | 80-150 ms   |
| ORM SELECT            | 1-2k RPS     | 150-300 ms  |
| DRF сериализация      | 500-1.2k RPS | 250-500 ms  |

Потолок — 2-4 тысячи RPS. И это при идеальных условиях.

Почему так мало?

**CPU недогружен** — воркеры блокируются на I/O (база, Redis, внешние API). Пока ждут — CPU простаивает.

**WSGI = блокировка** — один запрос полностью занимает воркера до завершения.

**3 воркера мало** — для 2 ядер нужно минимум 5 (формула CPU × 2 + 1).

Что будет при росте нагрузки? Latency растёт, запросы копятся в очереди, Nginx начинает выдавать 502 (upstream timeout).

Это типичная картина «сайт тормозит под нагрузкой». Не потому что Django медленный — потому что конфигурация не рассчитана на нагрузку.

## Слайд 26 — Производительность WSGI с threads=2 (3 мин)

Добавим потоки:

```bash
--workers 3 --threads 2
```

**Итого: 6 одновременных запросов.**

Уже лучше. Что получаем:

| Тип          | RPS          |
|--------------|--------------|
| Простой API  | 4-7k RPS     |
| Django + ORM | 2-3k RPS     |
| Django + DRF | 1.5-2.5k RPS |

Прирост — примерно в 1.5-2 раза. Неплохо для одной настройки.

Но есть нюансы.

**GIL** — Global Interpreter Lock. Python может выполнять только один поток в один момент. Потоки помогают при I/O (пока один ждёт базу, другой работает), но не при CPU-bound операциях.

**Рост latency** — при конкурентности потоков p95/p99 становятся менее предсказуемыми.

**Память** — каждый воркер потребляет больше RAM из-за нескольких потоков.

Практический предел для threads — 2-4. Больше — начинаются проблемы с GIL contention, context switching, предсказуемость падает.

Это всё ещё WSGI со своими ограничениями. Асинхронность решает эти проблемы элегантнее.

## Слайд 27 — Почему threads=4 часто хуже (3 мин)

Логика подсказывает: больше потоков = больше производительность. Но Python — не логичный язык.

```bash
--workers 3 --threads 4
```

**Итого: 12 потоков.**

Что происходит на практике? Часто **хуже**, чем threads=2.

Почему?

**GIL contention** — потоки борются за GIL, переключение между ними стоит времени.

**Context switching** — операционная система тратит время на переключение между 12 потоками.

**p95/p99 растёт** — отдельные запросы начинают тормозить непредсказуемо.

**CPU «пилит»** — загрузка высокая, но RPS почти не растёт.

Я видел это много раз: команда увеличивает threads с 2 до 8, ждёт 4x рост, получает тот же RPS или хуже, плюс нестабильный latency.

Золотое правило для WSGI:
- **threads = 1-2** — безопасный выбор
- **threads = 4** — тестируйте, может работать
- **threads > 4** — почти всегда плохая идея

Если нужно больше конкурентности — переходите на ASGI. Там один воркер может держать тысячи соединений без проблем с GIL.

## Слайд 28 — Реальные ограничения WSGI (3 мин)

Давайте честно: **WSGI — это устаревшая модель** для высоконагруженных API.

Почему?

**Блокирующая модель** — каждый запрос держит воркер. Обращение к базе — воркер ждёт. HTTP-запрос к внешнему API — воркер ждёт. Это неэффективно.

**Масштабирование процессами** — чтобы обрабатывать больше запросов, нужно больше воркеров. Больше воркеров = больше RAM. На 4GB вы упрётесь быстро.

**3 workers для 2 CPU — мало** — золотое правило для WSGI:

```
workers = CPU × 2 + 1
```

Для 2 ядер это **5 workers**. Не 3.

Что вы реально получите с `--workers 3 --threads 1`:

- **2-4k RPS потолок**
- Рост latency под нагрузкой
- 502 ошибки от Nginx
- Очередь запросов

Это нормально для:
- Админки
- Low-traffic сайтов
- Внутренних инструментов

Это **не нормально** для:
- Публичного API
- High-load приложений
- Мобильных бэкендов

Если вам нужно больше — ASGI.

---

## Слайд 29 — Оптимизация WSGI (если нельзя ASGI) (3 мин)

Иногда ASGI не вариант. Легаси-код, зависимости, не поддерживающие async. Что делать?

Максимум из WSGI:

```bash
gunicorn config.wsgi:application \
  --workers 5 \
  --threads 2 \
  --timeout 30 \
  --max-requests 10000 \
  --max-requests-jitter 1000
```

**workers 5** — для 2 CPU это оптимально по формуле CPU × 2 + 1.

**threads 2** — небольшое увеличение конкурентности без проблем GIL.

**timeout 30** — убиваем зависшие запросы.

**max-requests 10000** — перезапуск воркеров для борьбы с утечками памяти.

**max-requests-jitter 1000** — чтобы не перезапускались одновременно.

Что даст:

| Стек           | RPS      |
|----------------|----------|
| Django (views) | 6-9k     |
| Django + DRF   | 3-5k     |

Это **предел WSGI** на данном железе. Дальше — только ASGI или горизонтальное масштабирование.

Если 5-9k RPS достаточно — отлично, не усложняйте. Если нужно больше — миграция на ASGI неизбежна.

## Слайд 30 — Сравнение WSGI и ASGI (3 мин)

Давайте визуализируем разницу:

| Конфигурация         | RPS        |
|----------------------|------------|
| WSGI (3w, 1t)        | 2-4k       |
| WSGI (5w, 2t)        | 6-9k       |
| **ASGI (4 workers)** | **15-30k** |

ASGI в 3-5 раз быстрее оптимизированного WSGI. Это не магия — это правильная модель для I/O-bound нагрузки.

**Миграция с WSGI на ASGI:**

1. Замените `config.wsgi:application` на `config.asgi:application`
2. Добавьте `worker_class = "uvicorn.workers.UvicornWorker"` в gunicorn.conf.py
3. Установите uvicorn: `pip install uvicorn`
4. Готово

Серьёзно, это всё. Django 4.0+ поддерживает ASGI из коробки. Если ваш код не использует специфичные WSGI-фичи — миграция тривиальна.

**Короткий вывод:**

🔴 **gunicorn + wsgi + 3 workers** = нормально для админки и low-traffic, плохо для high-load API.

🟢 **gunicorn + asgi + 4 workers** = правильный выбор для современного Python-бэкенда.

## Слайд 31 — Системный дизайн бэкенда (3 мин)

Теперь поднимемся на уровень выше — системный дизайн. Прежде чем писать код, нужно ответить на ключевые вопросы.

**Load estimation** — сколько RPS ожидаем? Не «много» или «мало», а конкретные цифры. Read/Write ratio? Если 95/5 — можно агрессивно кэшировать. Если 50/50 — другая стратегия.

**Latency SLA** — какие допустимы задержки? p95 < 100ms? p99 < 500ms? Это влияет на выбор архитектуры.

**Data volume** — объём данных, скорость роста, сколько хранить. Миллион записей — одно. Миллиард — совсем другое.

**Consistency** — нужна строгая консистентность или eventual consistency окей? Для финансов — строгая. Для лайков — eventual.

Типичные компоненты API-системы:

| Компонент   | Роль                          |
|-------------|-------------------------------|
| API Gateway | Маршрутизация, rate limit     |
| App server  | Бизнес-логика, сериализация   |
| Database    | Источник истины               |
| Cache       | Снижение нагрузки на БД       |
| Queue       | Асинхронные задачи            |

**MVP стек:** Django + Ninja + PostgreSQL + Redis. Просто, достаточно для старта.

**High-load стек:** Nginx → Gunicorn/ASGI → Django Ninja → PostgreSQL (read replicas) + Redis + Celery/Kafka.

Не overengineer'те на старте. Начните с MVP, масштабируйтесь по мере роста.

## Слайд 32 — Системный дизайн Django-Ninja (3 мин)

Где Django-Ninja вписывается в архитектуру?

**Monolith API** — один Django-проект, много эндпоинтов. ORM для работы с данными, admin для управления. Классика.

**API-first** — Django как бэкенд для мобильных приложений, SPA, других сервисов. Ninja даёт быстрый JSON API с Pydantic-схемами.

**Hybrid** — Django views для рендеринга HTML + Ninja API для AJAX/мобилки. Лучшее из двух миров.

Когда выбирать что:

| Сценарий                                   | Выбор         |
|--------------------------------------------|---------------|
| Нужен ORM + admin + быстрый API            | Django + Ninja |
| Максимум RPS, минимум Django-зависимостей  | FastAPI        |
| Сложные сериализаторы, Browsable API       | Django + DRF   |

Паттерны проектирования для Ninja:

**Stateless API** — сессии в Redis, не в памяти воркера. Любой инстанс может обработать любой запрос.

**Read-heavy** — `select_related()`, `prefetch_related()`, кэш в Redis. 90% запросов — из кэша.

**Write-heavy** — Celery для фоновых задач, батчинг операций. Не пишите в базу синхронно на каждый запрос.

**Горизонтальное масштабирование** — несколько инстансов за Nginx или ALB. Stateless-архитектура это позволяет.

from django.urls import path

## Слайд 35 — Структура проекта (3 мин)

Когда проект растёт, структура становится важной. Вот рекомендуемая организация:

```
myproject/
├── config/           # settings, urls
├── api/
│   ├── __init__.py   # главный NinjaAPI
│   ├── v1/
│   │   ├── users.py
│   │   ├── products.py
│   │   └── __init__.py
│   └── deps.py       # зависимости (auth, db)
├── apps/
│   ├── users/
│   └── catalog/
└── manage.py
```

**config/** — конфигурация Django: settings.py, urls.py, wsgi.py, asgi.py.

**api/** — всё, что связано с API. Отдельная папка, не размазано по apps.

**api/v1/** — версионирование. Когда будет v2 — добавите папку, старые клиенты не сломаются.

**apps/** — Django-приложения с моделями, admin, бизнес-логикой.

**deps.py** — общие зависимости: аутентификация, database sessions, что переиспользуется.

Модульные роутеры:

```python
# api/__init__.py
from ninja import NinjaAPI
from .v1 import router as v1_router

api = NinjaAPI(version="1.0")
api.add_router("/v1", v1_router)
```

Каждый модуль (users, products) — свой Router. Главный API собирает их вместе. Это масштабируется: 10 модулей, 100 модулей — структура остаётся понятной.

## Слайд 36 — Роутеры и эндпоинты (3 мин)

Роутеры — это способ группировать эндпоинты.

```python
from ninja import Router

router = Router(tags=["users"])

@router.get("/users")
def list_users(request):
    return User.objects.all()

@router.get("/users/{user_id}")
def get_user(request, user_id: int):
    return User.objects.get(id=user_id)

@router.post("/users")
def create_user(request, payload: UserCreateSchema):
    return User.objects.create(**payload.dict())
```

**tags=["users"]** — группировка в Swagger-документации.

**Path-параметры**: `{user_id}` в URL → `user_id: int` в функции. Ninja автоматически конвертирует и валидирует.

**Query-параметры** — через аргументы функции:

```python
@router.get("/users")
def list_users(request, limit: int = 20, offset: int = 0):
    return User.objects.all()[offset:offset+limit]
```

HTTP-методы:
- `@router.get` — получение данных
- `@router.post` — создание
- `@router.put` — полное обновление
- `@router.patch` — частичное обновление
- `@router.delete` — удаление

Ninja следует REST-конвенциям, но не навязывает их. Хотите RPC-стиль с POST везде? Пожалуйста.

## Слайд 37 — Схемы Pydantic (3 мин)

Схемы — сердце Ninja. Они определяют, что приходит и что уходит.

```python
from ninja import Schema
from pydantic import validator

class UserCreate(Schema):
    email: str
    name: str
    password: str

    @validator("email")
    def email_must_be_valid(cls, v):
        if "@" not in v:
            raise ValueError("Invalid email")
        return v

class UserOut(Schema):
    id: int
    email: str
    name: str
```

**Schema** — базовый класс Ninja, наследник Pydantic BaseModel.

**@validator** — кастомная валидация. Если email невалидный — 422 ошибка с понятным сообщением.

Зачем **отдельные In/Out схемы**?

**UserCreate (In)** — что приходит от клиента. Есть password, нет id (его создаст база).

**UserOut (Out)** — что уходит клиенту. Есть id, нет password (безопасность).

Использование:

```python
@router.post("/users", response=UserOut)
def create_user(request, payload: UserCreate):
    user = User.objects.create(**payload.dict())
    return user
```

**payload: UserCreate** — Ninja автоматически парсит JSON тела и валидирует.

**response=UserOut** — Ninja сериализует ответ по схеме, не пропустит лишние поля.

Это type safety на уровне API. Ошибки ловятся до рантайма.

## Слайд 38 — CRUD и ORM (3 мин)

Типичный CRUD на Ninja:

```python
from typing import List
from django.shortcuts import get_object_or_404

@router.get("/users", response=List[UserOut])
def list_users(request, limit: int = 20, offset: int = 0):
    return list(User.objects.all()[offset:offset+limit])

@router.get("/users/{user_id}", response=UserOut)
def get_user(request, user_id: int):
    return get_object_or_404(User, id=user_id)

@router.post("/users", response=UserOut)
def create_user(request, payload: UserCreate):
    return User.objects.create(**payload.dict())

@router.put("/users/{user_id}", response=UserOut)
def update_user(request, user_id: int, payload: UserCreate):
    user = get_object_or_404(User, id=user_id)
    for attr, value in payload.dict().items():
        setattr(user, attr, value)
    user.save()
    return user

@router.delete("/users/{user_id}")
def delete_user(request, user_id: int):
    user = get_object_or_404(User, id=user_id)
    user.delete()
    return {"success": True}
```

**Оптимизация запросов** — критически важно:

```python
User.objects.select_related("profile").prefetch_related("orders")[:20]
```

**select_related** — JOIN для ForeignKey, OneToOne. Один запрос вместо N+1.

**prefetch_related** — отдельный запрос для ManyToMany, reverse FK. Два запроса вместо N+1.

Без этого список из 100 пользователей = 101 запрос к базе. С этим = 1-2 запроса.

## Слайд 39 — Аутентификация (3 мин)

Ninja поддерживает несколько способов аутентификации.

**API Key в заголовке:**

```python
from ninja.security import APIKeyHeader

class ApiKey(APIKeyHeader):
    param_name = "X-API-Key"

    def authenticate(self, request, key):
        if key == settings.API_KEY:
            return key
        return None

api = NinjaAPI(auth=ApiKey())
```

Все эндпоинты этого API теперь требуют заголовок `X-API-Key`.

**JWT (Bearer token):**

```python
from ninja.security import HttpBearer

class AuthBearer(HttpBearer):
    def authenticate(self, request, token):
        payload = validate_jwt(token)
        if payload:
            return payload
        return None
```

**Per-route auth** — разные эндпоинты, разная авторизация:

```python
@router.get("/public")
def public_endpoint(request):
    return {"data": "anyone can see this"}

@router.get("/private", auth=AuthBearer())
def private_endpoint(request):
    return {"user": request.auth}
```

`request.auth` — результат, который вернул `authenticate()`. Обычно это user или payload токена.

Ninja гибкий: можно комбинировать методы, делать цепочки авторизации, кастомные проверки.

## Слайд 40 — Валидация и обработка ошибок (3 мин)

**Автоматическая валидация** — Pydantic проверяет типы и constraints:

```python
class UserCreate(Schema):
    email: str
    age: int = Field(ge=0, le=150)  # 0 <= age <= 150
```

Если клиент пришлёт `{"email": 123, "age": -5}` — Ninja вернёт 422 с детальным описанием ошибок. Без единой строчки кода с вашей стороны.

**Кастомные исключения:**

```python
from ninja.responses import Response

@router.get("/users/{user_id}")
def get_user(request, user_id: int):
    try:
        return User.objects.get(id=user_id)
    except User.DoesNotExist:
        return Response({"detail": "User not found"}, status=404)
```

Или используйте `get_object_or_404` — он сам выбросит 404.

**Глобальный обработчик ошибок:**

```python
def custom_exception_handler(request, exc):
    return api.create_response(
        request,
        {"error": str(exc), "type": type(exc).__name__},
        status=400
    )

api = NinjaAPI()
api.add_exception_handler(ValueError, custom_exception_handler)
api.add_exception_handler(PermissionError, custom_exception_handler)
```

Все ValueError в любом эндпоинте будут обрабатываться единообразно.

Это важно для консистентного API: клиент всегда знает, какой формат ошибки ожидать.

## Слайд 41 — Async vs Sync (3 мин)

Django и Ninja поддерживают оба варианта.

**Sync-эндпоинт** (классический):

```python
@router.get("/users")
def list_users(request):
    return list(User.objects.all())
```

**Async-эндпоинт:**

```python
@router.get("/users/async")
async def list_users_async(request):
    return [user async for user in User.objects.all()]
```

Когда async даёт выигрыш?

**I/O-bound операции**: запросы к базе, внешние HTTP-вызовы, файловые операции. Пока ждём ответа — обрабатываем другие запросы.

**Django 4.1+** поддерживает async ORM. Но осторожно: не все операции async-нативные, Django использует `sync_to_async` под капотом.

Можно смешивать:

```python
@router.get("/sync")
def sync_endpoint(request):
    return {"type": "sync"}

@router.get("/async")
async def async_endpoint(request):
    return {"type": "async"}
```

⚠️ **Важно**: для async нужен ASGI-сервер. Uvicorn, Gunicorn + UvicornWorker. На WSGI async-эндпоинты не дадут преимуществ.

Практический совет: начните с sync. Когда упрётесь в производительность и профилирование покажет I/O-bottleneck — мигрируйте на async точечно.

## Слайд 42 — Тестирование Django Ninja (3 мин)

Ninja приходит с TestClient:

```python
from ninja.testing import TestClient
from api import api

client = TestClient(api)

def test_hello():
    response = client.get("/hello")
    assert response.status_code == 200
    assert response.json()["message"] == "Hello, Django Ninja!"

def test_create_user():
    response = client.post("/v1/users", json={
        "email": "test@example.com",
        "name": "Test User",
        "password": "secret123"
    })
    assert response.status_code == 201
    assert response.json()["email"] == "test@example.com"
```

**TestClient** — без реального HTTP, быстро. Для unit-тестов.

**django.test.Client** — полный Django request cycle с middleware. Для integration-тестов.

Паттерны тестирования:

**Fixtures** — создавайте тестовые данные:

```python
@pytest.fixture
def user():
    return User.objects.create(email="test@test.com", name="Test")

def test_get_user(user):
    response = client.get(f"/users/{user.id}")
    assert response.status_code == 200
```

**Параметризация** — один тест, много кейсов:

```python
@pytest.mark.parametrize("email,valid", [
    ("test@test.com", True),
    ("invalid", False),
])
def test_email_validation(email, valid):
    response = client.post("/users", json={"email": email, ...})
    expected = 201 if valid else 422
    assert response.status_code == expected
```

Тесты — не опционально. Они сэкономят вам часы отладки в продакшене.

## Слайд 43 — Чеклист деплоя Django Ninja (3 мин)

Финальный слайд — чеклист перед продакшеном. Пройдитесь по каждому пункту.

- [ ] **DEBUG = False**, **ALLOWED_HOSTS** настроены — без этого не деплоить

- [ ] **ASGI**: Gunicorn + UvicornWorker — не WSGI

- [ ] **Nginx** перед приложением — статика, SSL, буферизация

- [ ] **PostgreSQL + индексы** — профилируйте запросы, индексируйте

- [ ] **Redis** для кэша/сессий — если используете кэш или сессии

- [ ] **Минимум middleware** — уберите ненужное (Sessions, CSRF для API)

- [ ] **select_related / prefetch_related** — для всех эндпоинтов со списками

- [ ] **Rate limiting** — Nginx или django-ratelimit, защита от DoS

- [ ] **Логирование и мониторинг** — Sentry для ошибок, Prometheus для метрик

Бонусные пункты:

- [ ] **HTTPS** — Let's Encrypt бесплатный, нет оправданий для HTTP

- [ ] **Health check endpoint** — `/api/health` для load balancer

- [ ] **Graceful shutdown** — Gunicorn `--graceful-timeout 30`

- [ ] **Backup базы** — pg_dump по крону, тестируйте восстановление

Это не паранойя — это профессионализм. Каждый пункт — урок, выученный на чьих-то ошибках.
