## Слайд 2. Что такое деплой

Деплой — это процесс доставки приложения в production. Простыми словами: вы делаете код доступным пользователям. С точки зрения системы, как на слайде, деплой всегда включает четыре компонента. Сервер — физический или виртуальный, где крутится ваш код. Окружение — Python, зависимости, переменные. Конфигурация — настройки приложения, базы, прокси. И обязательно безопасность — иначе всё упадёт в первые же дни. Запомните главное: написанный код и работающий сервис — разные вещи. Код — это сырьё, деплой — доставка до пользователя.

---

## Слайд 3. Введение в деплой

Деплой — это не `python manage.py runserver`. На слайде это явно подчёркнуто. Runserver — только для разработки, на проде его не используют. Production — это боевое окружение, где живут реальные пользователи. Требования: стабильность, безопасность, масштабируемость. 
зачем изучать деплой? 
Во-первых, код не равен работающему сервису. 
Во-вторых, ошибки чаще всего не в алгоритмах, а именно в деплое: забытая миграция, неверный env, сломанный SSL. 
В-третьих, понимание деплоя — это уверенность разработчика. На собеседованиях этот навык сразу выделяет кандидата.

## Слайд 4. Production vs dev

На слайде — таблица сравнения. 
Сервер: в dev — localhost, в production — удалённый сервер. 
Отладка: в dev включена, в prod отключена — иначе утечки и риски.
HTTPS: в dev часто нет, в prod обязателен. 
Перезапуск: в dev вручную, в prod — автоматически, через systemd или оркестратор.
Логи: в dev консоль, в prod — файлы или централизованная система. 
Эти вещи кажутся мелочами, но именно они создают разницу между «работает у меня» и «работает у всех».
Каждый пункт — потенциальная боль, если его проигнорировать.

## Слайд 9. Uvicorn и Gunicorn — зачем нужен отдельный сервер

Зачем нужен отдельный сервер приложений? 

На слайде написано: runserver — только для разработки. 

И перечислены причины. Он не масштабируется — один процесс. 

Не поддерживает HTTPS — только HTTP. 

Однопоточный в Django — один запрос в момент. 

Не production-ready — не рассчитан на нагрузку и отказоустойчивость. 

В бою используют Gunicorn или Uvicorn: несколько воркеров, работа за Nginx, возможность перезапуска при падении.

* **Gunicorn** — Green Unicorn, «зелёный единорог». Green — от greenlets (лёгкая конкуррентность в Python), Unicorn — игра слов: UNIX-ориентированный HTTP-сервер. WSGI-сервер для синхронных приложений.
* **Uvicorn** — UV +icorn: UV от ultraviolet (ультрафиолет), по аналогии с Gunicorn. ASGI-сервер для асинхронных приложений, поддерживает WebSocket и HTTP/2.

Runserver — для быстрой проверки, не для прода.

## Слайд 12. Gunicorn + Uvicorn — рекомендация

На слайде — что делает сервер приложений:

запускает Python-код, обрабатывает HTTP, работает за Nginx. 

Рекомендация для FastAPI: Gunicorn с UvicornWorker. 

Команда на слайде: gunicorn main:app, флаг -k uvicorn.workers.UvicornWorker, -w 4 воркера, bind на порт.

Плюсы: устойчивость к падениям, несколько воркеров, полные ASGI-возможности. 

Флаг -k задаёт класс воркера, -w — число процессов. 

Запомните эту связку — для FastAPI в проде это типичный вариант.

## Слайд 13. Nginx

На слайде — четыре роли Nginx.
Первая: reverse proxy — направляет запросы к приложению.

Вторая: статика — отдаёт CSS, JS, изображения быстрее, чем Python. 

Третья: SSL/TLS — терминация HTTPS, сертификаты. 

Четвёртая: балансировка — распределение нагрузки на несколько инстансов приложения.

Nginx не запускает Python, он только принимает запросы и передаёт их дальше.

Он — первая линия между интернетом и вашим приложением.

## Слайд 14. Пример конфига Nginx

На слайде — типичная схема: 

Client → Nginx → Gunicorn/Uvicorn → App. 

И конфиг. server слушает 80, server_name — ваш домен. 

Для location /static/ — alias на папку со статикой, Nginx отдаёт файлы сам.

Для `location /` разберём подробнее:

* **`location /`** — матчит все запросы, кроме уже обработанных (например, /static/). Это «поймай всё остальное» и передай в приложение.

* **`proxy_pass http://127.0.0.1:8000`** — перенаправление запроса на локальный порт 8000, где слушает Gunicorn/Uvicorn. 127.0.0.1 — приложение недоступно снаружи, только Nginx может к нему обращаться (безопасность).

* **`proxy_set_header Host $host`** — прокидывает исходный заголовок Host (домен, например example.com) в приложение. Без него Django/FastAPI видят Host: 127.0.0.1:8000 и могут неправильно строить ссылки, проверять ALLOWED_HOSTS и т.д.

* **`proxy_set_header X-Real-IP $remote_addr`** — передаёт реальный IP клиента. `$remote_addr` — IP того, кто подключился к Nginx. Без этого в логах и коде будет 127.0.0.1 вместо настоящего IP пользователя — потеряете возможность логирования, геолокации, rate limiting по клиенту.

Без этих заголовков приложение «не видит» настоящего клиента и домен — отладка и безопасность страдают. Это минимальный рабочий конфиг. 

Дальше добавляют SSL, кеш, rate limiting.

## Слайд 15. Зачем Nginx и что он не делает

**Как устроен Nginx.** Написан на C, event-driven архитектура — асинхронная обработка соединений без блокировок.

Один master-процесс и несколько worker-процессов, каждый обслуживает тысячи соединений. Не тянет отдельный поток на каждый запрос, поэтому эффективен при высокой нагрузке. 

Nginx — язык-агностик: он просто проксирует HTTP к любому бэкенду. 

Кроме Python (Django, FastAPI, Flask) его ставят перед Node.js (Express, Nest), Java (Spring Boot), Ruby (Rails), PHP (через PHP-FPM), Go (Gin, Echo), .NET и другими. 

Один и тот же Nginx — для любых фреймворков.

На слайде — четыре причины использовать Nginx.

Приложение слушает localhost — так безопаснее, снаружи его не видно. 

Nginx обрабатывает статику быстрее, чем Python. 

Один порт 80/443 для нескольких сервисов — один домен, разные приложения. 

Плюс кеширование, сжатие, rate limiting. 

Ниже — что Nginx не делает: не запускает Python, не знает Django и FastAPI, только проксирует. Вся логика — в приложении. 

Nginx — это «умный распределитель», не «бэкенд».

## Слайд 22. PaaS — Platform as a Service

На слайде — определение.
PaaS — платформа, где инфраструктура скрыта от разработчика.

Примеры: Railway, Render, Fly.io, Heroku. 

Модель: git push → build → run. 

Минимум настройки, максимум скорости. 

Вы пушите код, платформа сама поднимает окружение и деплоит. 

Подходит для MVP, обучения, демо. 

Когда нужно быстро показать работающий прототип — PaaS в помощь.

- Провайдер сам поднимает Python, зависимости, reverse proxy
- Миграции — через переменные окружения или скрипты pre-deploy
- Логи, метрики, SSL — встроены в панель
- Один репозиторий → несколько окружений (staging, production)

### Что вы *не* делаете

- Не настраиваете Nginx, Gunicorn, systemd
- Не выбираете ОС и образы
- Не настраиваете сеть и firewall

## Слайд 23. PaaS — плюсы и минусы

На слайде — плюсы: быстрый старт, встроенный CI/CD, не нужна глубокая DevOps-экспертиза.

Минусы: меньше контроля, ограничения и лимиты, цена растёт с нагрузкой. 

Когда использовать: MVP, обучение, демо. 

Для серьёзного продакшна с ростом трафика чаще переходят на VPS или контейнеры. 

PaaS — отличный старт, но при масштабировании упрётесь в ограничения и стоимость.

## Слайд 24. cPanel-хостинг

На слайде — что такое cPanel (показать реальную cPanel )

Классический shared-хостинг с графической панелью. 

Ограниченный доступ к системе. Часто используется Passenger для Python.

Python там не основной сценарий — хостинг заточен под PHP.

Для новых проектов уже не первый выбор, но legacy-сайты на нём ещё живут.

## Слайд 25. cPanel — когда использовать

На слайде — плюсы: дёшево, есть GUI, не нужен DevOps.

Минусы: плохо масштабируется, подход устаревший. 

Подходит для простых Django-сайтов, legacy-проектов, низкого трафика. 

Если планируете рост — лучше сразу смотреть на VPS или PaaS. 

cPanel — компромисс между ценой и гибкостью, для pet-проектов и мелких заказов иногда достаточно.

## Слайд 26. VPS — Virtual Private Server

На слайде — полный контроль, SSH-доступ.

## Слайд 28. VPS — плюсы и минусы

На слайде — плюсы: гибкость, дешевле PaaS при росте, нормальный продакшен-подход.

Минусы: ручная настройка, ответственность за безопасность и обновления. 

Всё настраиваете вы: firewall, SSL, бэкапы, мониторинг. 

Зато полный контроль и при росте трафика VPS часто выгоднее PaaS. 

Для джуна это хорошая практика — после настройки VPS понимание деплоя сильно вырастает.

## Слайд 29. VPS + Docker

На слайде — зачем Docker. 

Одинаковое окружение — dev = prod, нет «работает у меня».

Быстрый деплой — образ собран, развернули за минуты. 

Изоляция — сервисы в контейнерах, не мешают друг другу. 

«Работает у меня» перестаёт быть отмазкой — если работает в контейнере, заработает и на проде.

Docker для Python-проектов стал фактическим стандартом.

## Слайд 31. Docker — ключевые файлы

На слайде — четыре компонента.

Dockerfile — сборка образа приложения, шаги установки и копирования. 

docker-compose.yml — оркестрация сервисов, связи, volumes, ports. .env — переменные окружения, не в коде. volumes — персистентность БД, чтобы данные не терялись при пересоздании контейнеров.

Эти четыре вещи нужно понимать для уверенной работы с Docker в продакшене.

---

## Слайд 32. Docker — плюсы и минусы

На слайде — плюсы: стандарт индустрии, удобно масштабировать, идеально для CI/CD. 

Минусы: дополнительный порог входа. И цитата: 

Docker — не магия, а упаковка процессов. 

Контейнер — это изолированный процесс с своим файловым деревом.

Понимание этого снимает страхи и мифы. 

Docker не заменяет ОС, он использует ядро хоста.

---

## Слайд 33. CI/CD

На слайде — определение. 

CI — Continuous Integration: автоматические тесты при каждом коммите. 

CD — Continuous Deployment: автоматический деплой после успешных проверок. 

Вместе это убирает ручной деплой и снижает ошибки. 

Написали код, запушили — тесты провалились, деплой не пошёл. 

Всё зелёное — код сам уехал на сервер. 

В современной разработке CI/CD — ожидаемый минимум.

---

## Слайд 34. CI/CD — типовой pipeline

На слайде — цепочка. 

git push → тесты (pytest, lint) → сборка Docker-образа → push в registry → деплой на VPS или Kubernetes. 

Всё автоматически. 

Ошибка на любом шаге останавливает процесс. 

Не задеплоите сломанный код. 

Пайплайн описывается в конфиге — .github/workflows, .gitlab-ci.yml, Jenkinsfile.

Один раз настроили — пользуетесь постоянно.

## Слайд 35. CI/CD — инструменты

На слайде — четыре инструмента.

GitHub Actions — если репо на GitHub. 

GitLab CI — если на GitLab. Jenkins — self-hosted, гибкий. 

Bitbucket Pipelines — для Bitbucket. 

Выбор зависит от хостинга репозитория. 

Для старта хватает GitHub Actions или GitLab CI — оба дают бесплатные минуты и понятный UI. Jenkins — когда нужен полный контроль и сложные сценарии.

## Слайд 36. CI/CD — пример сценария

На слайде — шесть шагов. 
Push в main. 
Запуск тестов. 
Сборка Docker-образа. 
Публикация в registry.
Деплой на VPS или Kubernetes.
Health-check — проверка, что приложение отвечает.
Ручной деплой в полночь с надеждой «пронесёт» уходит в прошлое. 
CI/CD — про предсказуемость и контроль. 
Каждый шаг логируется, при падении видно где и почему.

## Слайд 37. Django и Docker Compose

На слайде — структура проекта.
Dockerfile, docker-compose.yml, папка django_project с manage.py, nginx с default.conf, .env. 

Всё в репозитории. 

Клонировали — docker-compose up — полный стек поднят локально или на сервере. 

Это золотой стандарт для небольших и средних Django-проектов. 

Один репо, один набор команд — и dev, и prod.

## Слайд 38. Django + Docker Compose — фрагмент

На слайде — фрагмент docker-compose. 

Сервис web: build из текущей директории, command gunicorn, volumes для статики, depends_on от db. 
Сервис db: образ postgres:17, volume для данных, переменные окружения.

Минимум для работы. 

Дальше можно добавить redis, nginx, celery — по необходимости. depends_on — чтобы web стартовал после db.

## Слайд 39. Django + Docker — типовые команды

На слайде — три команды. 

docker-compose up -d --build — поднять всё в фоне с пересборкой. 

docker-compose exec web python manage.py migrate — применить миграции внутри контейнера.

docker-compose exec web python manage.py collectstatic --noinput — собрать статику. 

После этого приложение доступно. 

Эти команды часто переносят в CI/CD или в скрипт деплоя.

## Слайд 40. FastAPI и Kubernetes

На слайде — зачем Kubernetes. 

Оркестрация множества контейнеров. 

Автомасштабирование — под нагрузкой поды множатся.

Self-healing — упавший под перезапускается. 

Rolling updates без даунтайма. 

FastAPI как stateless API отлично ложится на K8s: horizontal scaling, cloud-native. 

Но K8s — про масштаб и команду.

Для первого pet-проекта он избыточен.

---

## Слайд 41. Kubernetes — основные объекты

На слайде — четыре объекта. 

Pod — один или несколько контейнеров, минимальная единица. 

Deployment — управление репликами, rolling updates. 

Service — доступ к подам, сетевой абстракции. 

Ingress — маршрутизация HTTP, один домен — несколько сервисов. 

Это базовый набор. 

Для старта хватит Deployment и Service.

Ingress — когда несколько приложений за одним доменом.

---

## Слайд 42. FastAPI в K8s — Deployment

На слайде — пример манифеста. 

apiVersion apps/v1, kind Deployment, replicas 3, образ из registry, containerPort 8000. 

K8s сам раскидывает поды по нодам, перезапускает при падении, делает rolling update.

Ниже — когда нужен K8s: high-load, микросервисы, большие команды. 

Когда не нужен: MVP, маленькие проекты, один сервер.

Не тяните K8s в проект «на будущее» — это оверхед.

## Слайд 43. Сравнительная таблица

На слайде — таблица. 

PaaS и cPanel — низкий контроль, проще вход. 

VPS — высокий контроль, больше ручной работы.

VPS + Docker — стандарт для серьёзных проектов. 

VPS + Docker + CI/CD — зрелый процесс. 

Kubernetes — максимальный контроль и сложность.

Выбирайте по задаче, не «потому что модно». 

Для MVP — PaaS, для роста — Docker, для масштаба — K8s.
