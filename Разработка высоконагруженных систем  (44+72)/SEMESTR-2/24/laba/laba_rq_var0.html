Вариант заданий: фоновые задачи (эволюционный подход)

Логика: сначала осваиваем новый API Django 6 (@task, .enqueue()), потом подключаем реальную очередь (django-rq). Код задач не меняется — меняется только бэкенд в настройках. Формы и кнопки используют HTMX (как в стенде) — ответы подставляются без перезагрузки.

Часть 1. Django 6 Tasks — контракт и Immediate

Задание 1.1. Кнопка «Скачать каталог в PDF» и первая Django Task

Цель: познакомиться с единым контрактом Django для фоновых задач. Добавляем на главную кнопку, по нажатию ставится задача генерации PDF. С бэкендом Immediate задача выполняется сразу — отдельный воркер не нужен.

Шаги:

  1. Проверьте версию Django: python manage.py version — должна быть 6.0+.
  2. Установите reportlab: pip install reportlab
  3. Проверьте настройки: в settings.py — MEDIA_ROOT, MEDIA_URL. Если нет: MEDIA_ROOT = BASE_DIR / "media", MEDIA_URL = "media/".
  4. Создайте shop/tasks.py:
from django.tasks import task import os from django.conf import settings @task() def export_catalog_pdf() -> str: """Генерирует PDF-каталог товаров и сохраняет в media/exports/.""" from .models import VideoCard from reportlab.lib.pagesizes import A4 from reportlab.pdfgen import canvas path = os.path.join(settings.MEDIA_ROOT, "exports", "catalog.pdf") os.makedirs(os.path.dirname(path), exist_ok=True) c = canvas.Canvas(path, pagesize=A4) c.setFont("Helvetica", 14) c.drawString(50, 800, "Каталог видеокарт") for i, p in enumerate(VideoCard.objects.all()): c.drawString(50, 770 - i * 20, f"{p.name} — {p.price:.0f} ₽") c.save() return path

Почему задача без аргументов: для простоты первая задача не принимает параметров. В следующих заданиях узнаем, что аргументы должны быть сериализуемы (примитивы, id моделей).

  1. View для кнопки в shop/views.py — при HTMX возвращает фрагмент (сохраняйте при переходе на django-rq):
from django.shortcuts import render, redirect from shop.tasks import export_catalog_pdf def export_catalog(request): """По нажатию кнопки ставит задачу и возвращает ответ.""" export_catalog_pdf.enqueue() if request.headers.get("HX-Request"): return render(request, "shop/components/export_status.html", {"message": "Экспорт запущен"}) return redirect("home")

Фрагмент shop/templates/shop/components/export_status.html:

<p class="export-ok">{{ message }}</p>

Кнопка на главной (HTMX, без перезагрузки):

<div id="export-status"></div> <form method="post" action="{% url 'export_catalog' %}" hx-post="{% url 'export_catalog' %}" hx-target="#export-status" hx-swap="innerHTML"> {% csrf_token %} <button type="submit">Скачать каталог в PDF</button> </form>
  1. Добавьте маршруты в config/urls.py:
path("export-catalog/", export_catalog, name="export_catalog"), path("", home, name="home"),
  1. Проверка: с Immediate при нажатии файл media/exports/catalog.pdf создаётся, сообщение «Экспорт запущен» появляется без перезагрузки (HTMX). С django-rq ответ быстрее, PDF генерируется в фоне.
Итог: вы используете @task и .enqueue(). Код одинаков для Immediate, Dummy и RQ.

Задание 1.3. DummyBackend в тестах

Цель: научиться тестировать код с задачами так, чтобы сами задачи не выполнялись — без Redis, без реальных писем и файлов.

Зачем это нужно. При вызове export_catalog_pdf.enqueue() поведение зависит от бэкенда:

  • Immediate — задача выполняется сразу (генерирует PDF, отправляет письмо и т.д.)
  • RQ/Celery — задача попадает в очередь и выполняется воркером
  • Dummy — задача не выполняется, только фиксируется факт вызова enqueue

С DummyBackend в тестах enqueue вызывается, но код задачи не запускается. Можно проверить, что «задачу поставили в очередь с правильными аргументами», без побочных эффектов.

Шаги:

  1. Создайте или дополните тестовый модуль (shop/tests.py).
  2. Переопределите настройки для тестов:
from django.test import TestCase, override_settings @override_settings( TASKS={ "default": {"BACKEND": "django.tasks.backends.dummy.DummyBackend"}, }, ) class ExportTasksTest(TestCase): def test_export_catalog_enqueue_does_not_run_task(self): """При DummyBackend enqueue не выполняет задачу, но не падает.""" from shop.tasks import export_catalog_pdf result = export_catalog_pdf.enqueue() self.assertIsNotNone(result) # Код задачи не выполнялся — PDF не генерировался
  1. Запустите тесты: python manage.py test shop.tests.ExportTasksTest
  2. Проверка: тест проходит, PDF не создаётся — значит, код задачи не выполнялся.
Итог: в проде — Immediate или RQ; в тестах — Dummy. В CI не нужен Redis; тесты быстрые и изолированные.

Задание 2.2. Запуск воркера и проверка фона

Цель: убедиться, что задача выполняется в отдельном процессе, а не в веб-воркере.

  1. Запустите Redis.
  2. Терминал 1: python manage.py runserver
  3. Терминал 2: python manage.py rqworker default
  4. Нажмите кнопку «Скачать каталог в PDF».
  5. Наблюдение: файл media/exports/catalog.pdf создаётся воркером; HTTP-ответ приходит быстро.
Объяснение: веб-сервер только кладёт задачу в Redis и возвращает ответ. Воркер забирает задачу и выполняет export_catalog_pdf в своём процессе.

Задание 3.5. Celery (опционально)

Цель: познакомиться с Celery. Переписать одну задачу с django-rq на Celery.

Когда использовать Celery: сложные пайплайны (цепочки, группы), расписание (beat), RabbitMQ, большая нагрузка. Для простых задач django-rq достаточно.

  1. pip install celery django-celery-results
  2. config/celery.py:
from celery import Celery import os os.environ.setdefault("DJANGO_SETTINGS_MODULE", "config.settings") app = Celery("config") app.config_from_object("django.conf:settings", namespace="CELERY") app.autodiscover_tasks()

В config/__init__.py: from .celery import app as celery_app, __all__ = ("celery_app",)

  1. settings.py: INSTALLED_APPS += ["django_celery_results"], CELERY_BROKER_URL, CELERY_RESULT_BACKEND = "django-db", CELERY_ACCEPT_CONTENT = ["json"], CELERY_TASK_SERIALIZER = "json"
  2. python manage.py migrate django_celery_results
  3. Celery-задача в shop/tasks.py:
from config.celery import app @app.task def export_catalog_pdf_celery() -> str: """Тот же экспорт PDF, но через Celery.""" from .models import VideoCard import os from django.conf import settings from reportlab.lib.pagesizes import A4 from reportlab.pdfgen import canvas path = os.path.join(settings.MEDIA_ROOT, "exports", "catalog_celery.pdf") os.makedirs(os.path.dirname(path), exist_ok=True) c = canvas.Canvas(path, pagesize=A4) c.setFont("Helvetica", 14) c.drawString(50, 800, "Каталог видеокарт (Celery)") for i, p in enumerate(VideoCard.objects.all()): c.drawString(50, 770 - i * 20, f"{p.name} — {p.price:.0f} ₽") c.save() return path
  1. Вызов: export_catalog_pdf_celery.delay()
  2. Воркер: celery -A config worker -l info

Часть 4. Деплой

Задание 4.1. Воркер в Docker Compose

Цель: воркер должен работать как отдельный контейнер и перезапускаться при падении.

  1. Откройте docker-compose.prod.yml (или создайте).
  2. Добавьте сервисы redis и rq-worker:
services: redis: image: redis:7-alpine volumes: - redis_data:/data web: build: . command: gunicorn config.wsgi:application --bind 0.0.0.0:8080 environment: - REDIS_URL=redis://redis:6379/0 depends_on: - redis rq-worker: build: . command: python manage.py rqworker default environment: - REDIS_URL=redis://redis:6379/0 depends_on: - redis - web volumes: redis_data: {}
  1. В settings.py убедитесь, что RQ_QUEUES использует REDIS_URL из окружения.
  2. Проверка: docker-compose -f docker-compose.prod.yml up — поднимаются web, redis, rq-worker. Кнопка PDF или POST на /api/export-catalog выполняется в контейнере rq-worker.

Задание 4.2. Unit-файл systemd (опционально)

Цель: запускать воркер как службу на VM без Docker.

  1. Создайте /etc/systemd/system/rq-worker.service (или deploy/rq-worker.service):
[Unit] Description=RQ worker for Django shop After=network.target redis.service [Service] User=www-data Group=www-data WorkingDirectory=/opt/shop ExecStart=/opt/shop/venv/bin/python manage.py rqworker default Environment="REDIS_URL=redis://localhost:6379/0" Restart=always RestartSec=10 [Install] WantedBy=multi-user.target
  1. Команды на сервере:
sudo systemctl daemon-reload sudo systemctl enable rq-worker sudo systemctl start rq-worker sudo systemctl status rq-worker
  1. Логи: journalctl -u rq-worker -f
