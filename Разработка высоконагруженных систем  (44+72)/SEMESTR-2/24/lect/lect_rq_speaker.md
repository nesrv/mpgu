## Слайд 1: Фоновые задачи и очереди

Добрый день. Сегодня разберём тему, которая в индустрии считается must-have: как не блокировать HTTP-запрос тяжёлой работой и как устроены очереди задач в Python и Django. Если вы пишете веб-приложения, рано или поздно столкнётесь с ситуацией, когда пользователь нажал «Оформить заказ», а ваш сервер ушёл в таймаут — вот тут-то и пригодятся фоновые задачи. План такой: сначала разберём, зачем это всё нужно, потом эволюцию подходов — от cron до полноценных очередей, затем три основных инструмента — RQ, Django Tasks и Celery, примеры кода, и в конце — как это грамотно задеплоить в прод. Поехали.

---

## Слайд 2: Зачем нужны фоновые задачи

Классический сценарий: пользователь нажал «Оформить заказ» или «Скачать отчёт». Если вы делаете всё синхронно — то есть в том же HTTP-запросе отправляете письмо, генерируете PDF, общаетесь с платёжкой — то запрос может висеть две, пять, десять секунд. Воркер Gunicorn занят, другие пользователи ждут в очереди, получают таймауты и 502. UX убит. Идея простая: отдать пользователю ответ сразу — «заказ принят», «отчёт ставится в очередь» — а тяжёлую работу выполнить в фоне, в отдельном процессе. Пользователь получил ответ за сто миллисекунд, а письмо уйдёт через пару секунд. Вся магия — в асинхронности выполнения.

---

## Слайд 3: Синхронный vs фоновый сценарий

Сравним два подхода. Синхронно: запрос пришёл, отправили письмо, ответили. Воркер занят несколько секунд. При нагрузке — таймауты, 502. И если воркер упал посередине — повтора нет, письмо потеряно. В фоне: запрос пришёл, положили задачу в очередь, ответили. Воркер освободился за миллисекунды. Latency стабильная. А если воркер упал — задача осталась в очереди, можно сделать retry. Один раз настроили — и живите спокойно. Это базовая модель, на ней строится всё остальное.

---

## Слайд 4: Типичные кандидаты в фон

Что обычно выносят в фон? Во-первых, отправка email — приветствие, подтверждение заказа, сброс пароля. Во-вторых, генерация отчётов — PDF, Excel, выгрузки. В-третьих, обработка загруженных файлов — ресайз картинок, конвертация. В-четвёртых, интеграции с внешними API — платёжки, службы доставки, где есть таймауты и нужен retry. И наконец, периодические задачи — очистка старых записей, напоминания. Всё, что занимает больше пары сотен миллисекунд и не требует немедленного ответа пользователю — кандидат в очередь.

---

## Слайд 5: Эволюция: от cron к очередям

Раньше всё делали через cron: раз в минуту скрипт запускается, что-то делает. Проблемы: нет очереди, нет приоритетов, а если скрипт висит дольше минуты — получаем дубли, два процесса выполняют одно и то же. Очередь решает это: задачи кладутся в список, воркеры забирают по одной. Контроль нагрузки, retry, приоритеты. Брокер — Redis, RabbitMQ или даже БД — хранит очередь и метаданные. Это следующий уровень абстракции после cron.

---

## Слайд 6: Модель «очередь задач»

Три участника. Producer — код приложения, view или сервис. Он кладёт задачу в очередь. Queue — хранилище, в нашем случае Redis. Worker — отдельный процесс, который забирает задачи и выполняет их. Может быть несколько воркеров. Схема простая: Django ставит задачу в Redis, воркер забирает, выполняет. Классическая модель producer-consumer. Запомните эти три термина — они будут встречаться везде.

---

## Слайд 7: Redis как брокер для очередей

Redis — быстрый key-value store в памяти. У него есть списки, LPUSH, RPOP — идеально подходит для очередей. Плюс TTL, персистентность при необходимости. RQ, Redis Queue, использует Redis напрямую: одна очередь — один список. Простота — главный плюс. Но при очень высокой нагрузке и сложных сценариях смотрят на RabbitMQ и Celery. Redis хватает для большинства проектов.

---

## Слайд 8: Что такое RQ

RQ — библиотека для фоновых задач на Python. Задачи — обычные функции. Аргументы должны быть сериализуемы, pickle или JSON. Один брокер — Redis. Запуск воркера: rq worker. Для Django есть пакет django-rq: очереди в настройках, админка, management-команда. Минимум магии, максимум понятности. Если вы только начинаете с очередями — RQ идеальный старт.

---

## Слайд 9: Redis vs RQ — в чём разница

Частая путаница. Redis — это хранилище, «движок». Key-value store с операциями SET, GET, LPUSH. Используют для кэша, сессий, pub/sub, очередей. RQ — Python-библиотека, которая использует Redis как брокер. Она строит поверх Redis модель «очередь плюс воркеры». Redis работает сам по себе; RQ без Redis не работает. Аналогия: PostgreSQL — это БД, SQLAlchemy — ORM, который её использует. То же самое здесь.

## Слайд 10: Django 6 Tasks — встроенный контракт

С Django 6.0 в ядре появился Tasks API. Декоратор `@task`, метод `enqueue`, объект результата `TaskResult`. Важно понимать границу ответственности: Django даёт **контракт** — как описать задачу и как её поставить в «очередь». Но **отдельного процесса-воркера**, который бы сидел в фоне и вытаскивал задачи из Redis или RabbitMQ, в коробке **нет**. В ядре только два встроенных бэкенда — Immediate и Dummy. Реальный фон — через пакеты сообщества. На следующем слайде разберём это подробно, потому что это частый вопрос на собеседованиях: «а где же воркер?»

---

## Слайд 10a: Контракт, Immediate, Dummy и почему нет воркера

Простыми словами. Django говорит вам: «вот как писать задачи и как их запускать через `enqueue`». Это правила игры. А вот **отдельной программы**, которая крутится в фоне и сама забирает задачи из Redis или другой очереди, Django **не даёт**. Её нужно подключать отдельно.

**Immediate** — это настройка по умолчанию. Вы написали `enqueue` — и ваша функция **сразу выполняется**, прямо там же, где вызвали. Как будто вы просто вызвали обычную функцию, только через другой метод. **Фона нет**: пользователь ждёт, пока всё доделается. Зато ничего не ставить: ни Redis, ни отдельного воркера. Удобно учиться и писать код дома. На боевом сайте с тяжёлыми задачами так обычно не оставляют.

**Dummy** — для тестов. Вы вызвали `enqueue`, а код задачи **не запускается**. Запоминается только факт: «задачу хотели поставить, вот с какими данными». В тестах можно проверить: правильно ли мы вообще вызвали `enqueue`. На реальном сервере от этого толку нет — письма не уйдут.

**Почему в самом Django нет «настоящего» воркера.** У всех проектов разные желания: одним нужен Redis, другим — только база, третьим — RabbitMQ. Django делает сайты, а не заменяет Celery. Плюс уже есть готовые библиотеки. Поэтому в ядре — только простые варианты (Immediate и Dummy), а «настоящую очередь» подключают пакетами.

**Пакеты от сообщества** делают три шага: сохранить задачу куда-то снаружи (в БД или в Redis), запустить **отдельную программу-воркер**, она забирает задачи и выполняет. Примеры — `django-tasks-database` и `django-tasks-celery`. **Ваш код с `@task` и `enqueue` не меняется.** Меняется настройка в `settings` — и вы переходите с «сразу в том же запросе» на работу в фоне. На слайде таблица: Immediate и Dummy — без фонового воркера; database и celery — уже с очередью и воркером. Запомните: после `pip install django` **не появится** волшебная программа, которая сама разгребает очередь — её нужно выбрать и настроить.

## Слайд 11: Django Tasks — пример кода

Смотрите на экран. Пишем обычную функцию — `send_welcome_email`. Выше неё ставим декоратор `@task`. Это говорит Django: «эту функцию можно ставить в очередь». В view или сервисе вызываем `enqueue` — и передаём аргумент. Важный момент: передаём не объект `User`, а его `id` — число. Почему? Потому что аргументы сериализуются: превращаются в текст или байты, чтобы положить в очередь. Объект из базы с открытым соединением так не передать — будет ошибка. Поэтому всегда: `user.pk` или `order.id`. `result` — это объект, у него есть `id`. Его можно сохранить в БД и потом, когда пользователь обновит страницу, по этому `id` достать статус или результат. Красота в том, что этот же код работает и с Immediate — выполнится сразу, и с Celery — уйдёт в очередь. Одна программа, разная настройка.

## Слайд 12: Django Tasks — бэкенды

На слайде — сводная таблица. Четыре варианта. Мы уже разобрали Immediate и Dummy. Два других — от сообщества. `django-tasks-database` хранит задачи в таблице PostgreSQL. Запускаете воркер отдельной командой — он читает из этой таблицы. `django-tasks-celery` — под капотом используется Celery, Redis или RabbitMQ. В коде вы ничего не меняете: всё тот же `@task` и `enqueue`. Меняете только настройку в `settings` — какой бэкенд использовать. С Immediate на database или celery — одна строка. Сразу появляется настоящая очередь. Это называется «подменяемость бэкендов».

---

## Слайд 12a: django-tasks-database

Простыми словами: очередь — это просто таблица в вашей базе PostgreSQL. Миграции создают её при установке пакета. Когда вы вызываете `enqueue`, в эту таблицу пишется строка: какая функция, какие аргументы. Отдельная программа — воркер — запускается командой `python manage.py run_tasks`. Он периодически смотрит в таблицу: «есть новые задачи?» Забирает, выполняет, помечает готово. Плюс — не нужен Redis. Всё в одной базе. Минус — база нагружается ещё и очередью. При большом потоке задач может тормозить. Подходит для небольших проектов: интернет-магазин, админка, рассылки. Если не хотите ставить Redis отдельно — этот вариант для вас.

---

## Слайд 12b: django-tasks-celery

Здесь под капотом работает Celery. Redis или RabbitMQ — как «почтовый ящик» для задач. Вы вызываете `enqueue` — Django Tasks передаёт вызов в Celery. Celery кладёт сообщение в Redis. Celery worker забирает и выполняет. Плюсы: высокая скорость, можно крутить много воркеров, встроенные retry и расписание (beat). Минусы: нужно настроить Celery и брокер. Больше движущихся частей. Когда использовать: уже есть Celery в проекте или нужна серьёзная нагрузка — тысячи задач в час. Главное: ваш код с `@task` и `enqueue` не меняется. Только в `settings` указываете другой бэкенд.

---

## Слайд 13: django-rq — установка

Дальше работаем с django-rq — это самый простой способ добавить очереди в Django. Ставим два пакета: `django-rq` и `redis`. В `settings` добавляем `django_rq` в `INSTALLED_APPS` и прописываем `RQ_QUEUES` — где хост Redis, порт, какая база. Обычно localhost, порт 6379. Redis должен быть уже запущен на машине — локально или по сети. И всё. Никаких миграций для очереди, никакого Celery. Можно сразу писать задачи. Для джунов это идеальный старт: меньше движущихся частей, чем в Celery.

## Слайд 14: django-rq — определение задачи

Задача — это обычная Python-функция. Кладёте её в файл `tasks.py` приложения или в отдельный модуль. Например, функция `send_order_confirmation`. Она принимает `order_id` — число. Внутри по этому `id` загружает заказ из базы, формирует текст письма, вызывает `send_mail`. Почему передаём именно `id`, а не объект заказа? Потому что воркер — это другой процесс. Ему нужно передать данные «по проводу»: в Redis кладётся сериализованная задача. Объект Django с открытым соединением к БД так не передать. Поэтому правило: передаём `order.id`, `user.pk` — примитивы. В самой задаче делаем `Order.objects.get(pk=order_id)` и работаем с объектом. Это золотое правило для всех очередей.

## Слайд 15: django-rq — постановка в очередь

В view создаём заказ, вызываем `get_queue('default')` — получаем объект очереди. Далее `enqueue` — первый аргумент сама функция, второй — то, что ей передаём. Например, `queue.enqueue(send_order_confirmation, order.id)`. Сразу после этого возвращаем ответ пользователю. Задача уже в очереди, воркер её подхватит. Есть ещё опция `delay` — задача выполнится не сразу, а через N секунд. Например, напоминание через пять минут. Код очень простой. Никаких `celery_app.task`, никаких `.delay()` — просто `enqueue`. Если раньше работали с Celery, здесь почувствуете облегчение.

---

## Слайд 16: Сериализация аргументов

Остановлюсь на этом подробнее — это частая причина багов у новичков. Задача должна «уехать» из веб-процесса в воркер. Для этого аргументы сериализуются: превращаются в строку или байты. Что можно передавать: числа, строки, списки, словари, `id` моделей. Что нельзя: объекты `User`, `Order` с открытыми соединениями к БД, файловые объекты, что угодно несериализуемое. В задаче по `id` заново делаем `Model.objects.get(pk=...)`. Один раз передали объект — на воркере получите ошибку при десериализации. Я видел это на code review десятки раз. Запомните: только примитивы и id.

## Слайд 17: Django ORM в задачах

Воркер — отдельный процесс. У него своё соединение к базе. Бывает так: задача долго работала — минуту общалась с внешним API. За это время соединение к PostgreSQL могло «протухнуть» — сервер его закрыл. И вы получаете «connection already closed» в середине задачи. Что делать: в начале функции, если она долгая, можно вызвать `connection.close()`. Django при следующем обращении к БД откроет новое. Либо не держите соединение открытым на время внешних вызовов — делайте короткие транзакции. Это типичная ошибка при переносе кода из view в задачу.

---

## Слайд 18: django-rq — запуск воркера

Воркер — это отдельная программа, которая крутится и обрабатывает задачи. В разработке запускаете её вручную: `python manage.py rqworker default`. Для нескольких очередей — через пробел: `default high low`. Воркер будет брать задачи сначала из high, потом из default, потом из low. В проде воркер нельзя «запустить и забыть». Если он упадёт — задачи будут копиться, никто не выполнит. Поэтому воркер держат через systemd или supervisor: при падении он автоматически перезапустится. Это обязательно.

---

## Слайд 19: django-rq — несколько очередей

Можно завести несколько очередей. Например: `default`, `high` для срочных задач, `low` для фоновой подчистки. В настройках указываете разные Redis DB или разные ключи. В коде: `get_queue('high')` и туда кладёте критичную задачу. Воркеры можно запускать с разным приоритетом: один воркер только на high, другой — на default и low. Или разное количество воркеров на каждую очередь. Для старта хватит одной очереди — default.

## Слайд 20: django-rq — результат и статус

Когда вы вызываете `enqueue`, он возвращает объект `job`. У него есть `job.id` — уникальный идентификатор задачи. `job.get_status()` — статус: в очереди, выполняется, готово, упало. `job.result` — результат работы функции. Зачем это нужно? Пользователь нажал «Скачать отчёт». Вы поставили задачу, сохранили `job.id` в сессию или в базу. Сразу показали «отчёт готовится». Пользователь обновил страницу — вы по `job.id` проверили статус, если готово — отдали ссылку. Всё через этот идентификатор.

---

## Слайд 21: django-rq — обработка ошибок и retry

Если в задаче вылетело исключение, RQ помечает её как failed. Задачу видно в интерфейсе, можно разобраться. Можно настроить автоматический retry при постановке: `retry=Retry(max=3)` — RQ попытается выполнить до трёх раз. Либо в самой функции — try-except, логирование, при необходимости поставить задачу заново. Важно: retry — не панацея. Если ошибка в данных — неверный id — задача будет падать снова и снова. Нужна логика: временная ошибка сети — retry имеет смысл. Постоянная ошибка — залогировать и не retry.

## Слайд 22: Отложенное выполнение (scheduled)

Иногда нужно выполнить задачу не сейчас, а в конкретное время. Для RQ есть пакет rq-scheduler. Он позволяет ставить задачу на определённый момент — `enqueue_at` с датой и временем. Для периодических задач — «каждый день в полночь» — RQ сам по себе так не умеет. Вариант: cron раз в минуту запускает management-команду, которая ставит нужные задачи в RQ. Или одна задача «обработать всё накопившееся за день». У Celery есть встроенный beat — планировщик. В RQ — внешний триггер: cron или rq-scheduler.

---

## Слайд 23: Celery — кратко

Celery — «тяжёлая артиллерия» среди очередей. Несколько брокеров, встроенный планировщик beat, цепочки задач, группы, приоритеты. Декоратор `@celery_app.task`, вызов через `.delay()`. Подходит для больших проектов с высокой нагрузкой. Минус — больше настроек: broker, backend для результатов, отдельно beat, отдельно worker. Если RQ — «седан»: просто и по делу, то Celery — «грузовик»: мощно, но нужно разбираться. Для простой задачи «отправить письмо» Celery может быть избыточен. Но когда нужны цепочки задач, DAG, сложные пайплайны — Celery в тему.

---

## Слайд 24: Сравнение

Сводная таблица. Django Tasks — максимум по простоте, минимальная инфраструктура, подходит для малых и средних проектов. django-rq — средняя сложность, нужен Redis, средний масштаб. Celery — минимум по простоте, зато максимум по возможностям: Redis или RabbitMQ, высокий масштаб, DAG, workflows. Django Tasks встроен в Django с версии 6.0. Расписание: у Django Tasks — через бэкенд, у RQ — rq-scheduler или cron, у Celery — встроенный beat. Выбор зависит от размера проекта.

---

## Слайд 25: Когда что выбирать

Практические рекомендации. Небольшой проект, хотите быстро — Django Tasks с django-tasks-database или сразу django-rq. Уже есть Redis в проекте — django-rq логичный выбор. Сложные пайплайны, расписание из коробки, много воркеров — Celery. Хотите обойтись без Redis — Django Tasks с бэкендом на БД. Главное: не гонитесь за Celery, если задача простая — письмо при заказе, превью картинки. RQ или Django Tasks с database хватит с головой. Celery — когда масштаб и сложность оправдывают настройку.

## Слайд 26–28: Примеры — письмо, PDF, превью (один слайд в HTML)

Три типичных сценария. **Письмо после заказа:** view создаёт заказ в базе, ставит задачу `send_order_confirmation` с `order_id`, сразу возвращает пользователю «заказ принят». Задача в фоне по этому id загружает заказ, формирует письмо, вызывает `send_mail`. Пользователь не ждёт. **Экспорт в PDF:** тяжёлая операция — выборка из БД, генерация через reportlab или weasyprint. В view только постановка задачи и ответ «экспорт запущен, ссылка придёт на почту». Задача генерирует файл, сохраняет в media, шлёт письмо или пишет ссылку в модель. **Превью:** пользователь загрузил картинку. View сохраняет файл, ставит `generate_thumbnails` с `image_id`. Задача создаёт превью разных размеров, обновляет модель. Во всех трёх случаях — не блокируем HTTP. Тяжёлая работа — в фоне.

## Слайд 29: Мониторинг очереди

Нужно следить за очередью. Сколько задач ждут — `queue.count` или команда Redis `LLEN`. django-rq даёт веб-интерфейс: подключаете urls, открываете в браузере — видите очереди, джобы, упавшие задачи. Удобно. Для прода — метрики в Prometheus: число задач в очереди, время выполнения. При росте длины очереди — либо добавлять воркеры, либо оптимизировать задачи. Мониторинг обязателен. Иначе узнаете о проблеме, когда очередь в десять тысяч задач и пользователи жалуются. Лучше заранее видеть тренд.

## Слайд 30: Прод — запуск воркера (systemd)

В проде воркер должен работать как служба. Пишете unit-файл для systemd. ExecStart — путь к venv и команда `rqworker`. WorkingDirectory — папка проекта. Restart always — при падении перезапустится. After redis — запускать после Redis. Команды: `systemctl enable`, `start`, `status`. Логи — `journalctl -u rq-worker -f`. Без systemd или аналога воркер умрёт при перезагрузке сервера или при сбое. В проде так не живут. Обязательная настройка.

## Слайд 31–32: Прод — Redis и масштабирование (один слайд в HTML)

Про Redis. Включите персистентность — RDB или AOF. Иначе при перезапуске Redis потеряете все задачи в очереди. Память: настройте `maxmemory` и политику вытеснения. При переполнении Redis может выкинуть ключи — в том числе задачи. Сеть: ограничьте доступ только с нужных хостов. Redis без пароля на публичном интерфейсе — риск. Масштабирование: несколько воркеров на одной очереди — задачи сами распределяются. Можно на одной машине или на разных при общем Redis. Но смотрите на узкие места: CPU, диск, лимиты БД, лимиты внешних API. Слишком много воркеров без тюнинга — перегрузите БД или почтовый сервер. Масштабировать нужно с умом.

## Слайд 33–38: Типичные ошибки и практики (один слайд в HTML)

Шесть пунктов. **Ошибки:** передавать объект ORM вместо id — при сериализации сломается. Долгие задачи без таймаута — воркер зависнет, очередь вырастет. Игнорировать failed jobs — накопятся, никто не разберётся. Не закрывать соединения в долгих задачах — получите «connection already closed». **Практики:** в тестах использовать DummyBackend или тестовый Redis, мокать почту и внешние API. Безопасность — не передавать пароли и токены в аргументах, только id. Таймауты — настроить под задачу. Идемпотентность — повторное выполнение не должно давать дубли: два письма, два списания. И важно логировать job.id и аргументы — иначе при сбое в проде не воспроизведёте. Эти вещи — основа надёжности.

## Слайд 41: Деплой без Docker (HTML: первый слайд блока деплой)

Простейший вариант — виртуальная машина без контейнеров. Ставите ОС, Redis, Python, проект. Воркер запускаете через systemd — как службу. В unit-файле прописываете ExecStart — путь к venv и команду `rqworker` или `celery worker`. WorkingDirectory — папка проекта. Environment — переменные окружения: REDIS_URL, RQ_QUEUES или для Celery CELERY_BROKER_URL и CELERY_RESULT_BACKEND. Restart always — при падении воркер перезапустится. Redis обычно на той же машине — localhost. Если используете Celery и расписание — отдельный unit для beat. Beat должен быть в единственном экземпляре на весь кластер, иначе будут дубли задач. Конфиг — из переменных окружения, не хардкодить в коде. Классика. Многие проекты так живут годами.

## Слайд 43: Деплой с Docker (HTML: второй слайд блока деплой)

С Docker — всё упаковываете в образ. docker-compose: сервисы web, rq-worker, redis. Можно добавить celery-worker, celery-beat. Один образ для приложения — web и воркеры отличаются только command. Web — gunicorn, rq-worker — rqworker. depends_on: worker зависит от redis и web, чтобы сначала поднялась очередь. Redis — host в compose равен имени сервиса: `redis://redis:6379`. Beat — один контейнер, replicas один. Переменные — env_file или environment. Важно: RQ и Celery обрабатывают SIGTERM — graceful shutdown. Docker останавливает контейнер, посылает SIGTERM, воркер заканчивает текущую задачу и выходит. Никаких оборванных задач. Стандартная схема для средних проектов.

## Слайд 45: Деплой с Kubernetes (HTML: третий слайд блока деплой)

В Kubernetes — Deployment для воркера. replicas: 2 или больше — несколько подов обрабатывают очередь. REDIS_URL берёте из Secret, не пишете в коде. Celery — аналогично. Beat — replicas один, обязательно. Redis — либо свой Service в кластере, либо managed Redis снаружи. HPA — horizontal pod autoscaler — можно настроить автомасштабирование воркеров по длине очереди или CPU. Probes: liveness и readiness — Kubernetes поймёт, жив ли воркер. Один образ для web и workers, в Deployment разный command. Подставьте свой образ, секреты, лимиты ресурсов — и готово. Для крупных проектов — нормальный выбор.

## Слайд 47: Сводка деплой по средам (HTML: четвёртый слайд блока деплой)

Сводка. Без Docker: systemd, Redis на localhost, один сервер или несколько с общим Redis. Просто, но масштабировать вручную. С Docker: compose, redis в контейнере, web и workers — отдельные контейнеры. Удобно поднимать локально и на VPS. С Kubernetes: Deployment, HPA, масштабирование автоматом. Конфиг — env, ConfigMap, Secret. Общее правило: beat — один unit, один контейнер, один replica. Везде. Иначе дубли периодических задач. Выбор среды зависит от инфраструктуры и команды. Принципы одни: воркер как отдельный процесс, автоперезапуск, мониторинг.
