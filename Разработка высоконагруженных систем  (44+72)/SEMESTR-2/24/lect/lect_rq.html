Фоновые задачи и очереди

RQ, Django Tasks, Celery

Тема: как не блокировать HTTP-запрос тяжёлой работой и как устроены очереди задач в Python/Django.

План: зачем нужны фоновые задачи → эволюция подходов → RQ, Django Tasks, Celery → примеры → прод и практика.

Зачем нужны фоновые задачи

Пользователь нажал «Оформить заказ» или «Скачать отчёт».

Если всё делать в том же HTTP-запросе: отправка письма 2 сек, генерация PDF 10 сек, запрос висит, воркер занят, таймаут, плохой UX.

Идея: отдать ответ пользователю сразу («заказ принят», «отчёт ставится в очередь»), а тяжёлую работу выполнить в фоне в отдельном процессе.

Синхронный vs фоновый сценарий

СинхронноВ фоне (очередь)
Запрос → отправить письмо → ответЗапрос → положить задачу в очередь → ответ
Воркер занят 2–10 секВоркер свободен за <100 мс
Таймауты, 502 при нагрузкеСтабильная latency
Нет повтора при сбоеПовтор (retry) при падении воркера

Типичные кандидаты в фон

  • Отправка email (приветствие, заказ, сброс пароля)
  • Генерация отчётов (PDF, Excel), выгрузки
  • Обработка загруженных файлов (ресайз, конвертация)
  • Интеграции с внешними API (платёж, доставка) с таймаутами и retry
  • Периодические задачи (очистка, напоминания) — по расписанию

Эволюция: от cron к очередям

Раньше: cron раз в минуту/час запускает скрипт. Нет очереди, нет приоритетов, дубли при долгой работе.

Очередь: задачи кладутся в список, воркеры забирают по одной. Контроль нагрузки, retry, приоритеты.

Брокер: Redis, RabbitMQ, БД — хранят очередь и метаданные.

Модель «очередь задач»

[ Django / API ]  →  кладёт задачу  →  [ Redis / очередь ]
                                         ↓
[ Воркер 1 ]  ←  забирает задачу  ←  [ Redis ]
[ Воркер 2 ]  ←  забирает задачу  ←  [ Redis ]

Producer — код приложения (view, сервис). Queue — хранилище (Redis). Worker — процесс, который выполняет задачи.

Redis как брокер для очередей

Redis: быстрый key-value store в памяти. Подходит для очередей: списки (LPUSH/RPOP), TTL, персистентность при необходимости.

RQ (Redis Queue) использует Redis напрямую: одна очередь = один список. Простота — плюс для старта; при очень высокой нагрузке — RabbitMQ и Celery.

Что такое RQ

RQ (Redis Queue) — библиотека для фоновых задач на Python. Задачи — обычные функции; аргументы должны быть сериализуемы.

  • Один брокер — Redis
  • Запуск воркера: rq worker
  • Интеграция с Django: пакет django-rq

Redis vs RQ — в чём разница

Redis — хранилище данных в памяти. Key-value store с операциями SET, GET, LPUSH, RPOP.

RQ — Python-библиотека, которая использует Redis как брокер для очередей задач.

RedisRQ
Что этоХранилище (сервер)Библиотека для фоновых задач
РольХранит данныеОркестрирует задачи, воркеры, retry
Без другогоРаботает сам по себеТребует запущенный Redis

Итого: Redis — низкоуровневый сервис, RQ — слой очередей поверх него.

Django 6 Tasks — встроенный контракт

С Django 6.0 в ядре появился Tasks API: декоратор @task, метод .enqueue(), объект результата TaskResult.

Django не поставляет продакшен-воркер: только контракт и бэкенды Immediate (в том же потоке) и Dummy (для тестов). Реальный фон — через бэкенды сообщества.

Django Tasks — контракт, Immediate, Dummy

Контракт: @task, .enqueue(), TaskResult. Отдельного воркера в ядре нет.

Immediate: enqueue вызывает функцию сразу в том же потоке. Нет фона, нет retry. Dev и простые кейсы.

Dummy: задача не выполняется — только запись для тестов. Без Redis в CI.

Почему нет воркера в ядре: разная инфраструктура, масштаб, фокус Django — веб, не очереди; есть RQ/Celery.

БэкендВоркерОчередьПрод
ImmediateНетНетНет
DummyНетНетНет
tasks-databaserun_tasksБДДа
tasks-celeryCeleryRedis/RabbitДа

Сообщество: сохраняет задачи во внешнее хранилище + отдельный процесс. Код один — бэкенд в settings.

Django Tasks — пример кода

from django.tasks import task

@task()
def send_welcome_email(user_id: int) -> str:
    user = User.objects.get(pk=user_id)
    send_mail(...)
    return "sent"

# В view или сервисе:
result = send_welcome_email.enqueue(user.pk)

Аргументы должны быть JSON-сериализуемы.

Django Tasks — бэкенды

БэкендПоведение
ImmediateBackendВыполняет задачу сразу в текущем потоке
DummyBackendНе выполняет, для тестов
django-tasks-databaseОчередь в PostgreSQL, воркер отдельно
django-tasks-celeryИспользует инфраструктуру Celery

Итог: один API в коде, смена бэкенда — в настройках.

django-tasks-database

Очередь в PostgreSQL. Миграции создают таблицу; воркер run_tasks забирает задачи.

Плюсы: без Redis/RabbitMQ, одна БД, простая установка.

Минусы: нагрузка на БД, ниже производительность, polling.

Когда: небольшие проекты, минимум инфраструктуры.

django-tasks-celery

Использует Celery: брокер Redis или RabbitMQ. .enqueue() → Celery → worker.

Плюсы: высокая пропускная способность, retry, beat, масштабирование.

Минусы: нужны Celery и брокер, сложнее настройка.

Когда: уже есть Celery или нужна серьёзная нагрузка. Код один — бэкенд в settings.py.

django-rq — установка

pip install django-rq redis
INSTALLED_APPS += ['django_rq']

RQ_QUEUES = {
    'default': {'HOST': 'localhost', 'PORT': 6379, 'DB': 0},
}

Redis должен быть запущен локально или указан URL.

django-rq — определение задачи

# shop/tasks.py
def send_order_confirmation(order_id: int) -> None:
    order = Order.objects.get(pk=order_id)
    send_mail(subject=f"Заказ #{order.id}", ...)

В очередь передают примитивы, id моделей, а не объекты ORM.

django-rq — постановка в очередь

from django_rq import get_queue

def order_create_view(request):
    order = Order.objects.create(...)
    queue = get_queue('default')
    queue.enqueue(send_order_confirmation, order.id)
    return JsonResponse({"order_id": order.id})

# С задержкой:
queue.enqueue(send_order_confirmation, order.id, delay=300)

Сериализация аргументов

В очередь попадают сериализованные аргументы. Передавать нужно:

  • Примитивы (int, str, float, bool), списки, словари
  • ID моделей, а не объекты User, Order

Нельзя: объекты с открытыми соединениями. В задаче по id заново делают Model.objects.get(pk=...).

Django ORM в задачах

Воркер — отдельный процесс. У каждого своё соединение к БД. После долгой работы соединение может разорваться.

from django.db import connection
connection.close()  # в начале функции при необходимости

django-rq — запуск воркера

python manage.py rqworker default
python manage.py rqworker default high low

В проде — systemd или supervisor.

django-rq — несколько очередей

RQ_QUEUES = {
  'default': {...}, 'high': {'URL': 'redis://...6379/1'},
  'low': {'URL': 'redis://...6379/2'},
}
get_queue('high').enqueue(critical_task, ...)

django-rq — результат и статус

job = queue.enqueue(send_order_confirmation, order.id)
job.id  # job.get_status()  # queued, started, finished, failed
job.result  # после завершения

django-rq — retry

from rq import Retry
queue.enqueue(..., retry=Retry(max=3))

Отложенное выполнение (scheduled)

rq-scheduler: scheduler.enqueue_at(datetime(...), task, ...)

Периодические: cron → management-команда → постановка задач в RQ.

Celery — кратко

Мощная система: брокеры, beat, цепочки, группы. @celery_app.task, send_email.delay(user_id)

Когда что выбирать

  • Небольшой проект: Django Tasks или django-rq
  • Уже есть Redis: django-rq
  • Сложные пайплайны: Celery
  • Без Redis: Django Tasks + БД

Примеры: письмо, PDF, превью

Письмо после заказа: view создаёт заказ → enqueue(send_order_confirmation, order.id)

Экспорт PDF: view ставит задачу → «ссылка придёт на почту»

Превью: загрузка → enqueue(generate_thumbnails, image_id)

Мониторинг очереди

  • django-rq веб-интерфейс — очереди, джобы, failed
  • Метрики в Prometheus

Прод: Redis и масштабирование

Redis: RDB/AOF, maxmemory. Несколько воркеров на одну очередь — задачи распределяются.

Типичные ошибки и практики

  • ORM-объект вместо id
  • Долгие задачи без таймаута
  • Игнорировать failed jobs
  • Тесты: DummyBackend или тестовый Redis
  • Безопасность: не передавать пароли в аргументах
  • Идемпотентность — повторное выполнение не даёт дублей
