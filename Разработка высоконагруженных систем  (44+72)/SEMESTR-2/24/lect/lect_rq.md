# Лекция: Фоновые задачи и очереди (RQ, Django Tasks, Celery)

*Презентация для студентов. Ориентир: 2–3 минуты на слайд.*

---

## Слайд 1: Фоновые задачи и очереди

**Тема:** как не блокировать HTTP-запрос тяжёлой работой и как устроены очереди задач в Python/Django.

**План:** зачем нужны фоновые задачи → эволюция подходов → RQ, Django Tasks, Celery → примеры → прод.

---

## Слайд 2: Зачем нужны фоновые задачи

Пользователь нажал «Оформить заказ» или «Скачать отчёт».

Если всё делать **в том же HTTP-запросе**: отправка письма 2 сек, генерация PDF 10 сек, запрос висит, воркер занят, таймаут, плохой UX.

**Идея:** отдать ответ пользователю сразу («заказ принят», «отчёт ставится в очередь»), а тяжёлую работу выполнить **в фоне** в отдельном процессе.

---

## Слайд 3: Синхронный vs фоновый сценарий

| Синхронно | В фоне (очередь) |
|-----------|-------------------|
| Запрос → отправить письмо → ответ | Запрос → положить задачу в очередь → ответ |
| Воркер занят 2–10 сек | Воркер свободен за &lt;100 мс |
| Таймауты, 502 при нагрузке | Стабильная latency |
| Нет повтора при сбое | Повтор (retry) при падении воркера |

---

## Слайд 4: Типичные кандидаты в фон

- Отправка email (приветствие, заказ, сброс пароля).
- Генерация отчётов (PDF, Excel), выгрузки.
- Обработка загруженных файлов (ресайз, конвертация).
- Интеграции с внешними API (платёж, доставка) с таймаутами и retry.
- Периодические задачи (очистка, напоминания) — по расписанию.

---

## Слайд 5: Эволюция: от cron к очередям

**Раньше:** cron раз в минуту/час запускает скрипт. Нет очереди, нет приоритетов, дубли при долгой работе.

**Очередь:** задачи кладутся в список (очередь), воркеры забирают по одной. Контроль нагрузки, retry, приоритеты (в зависимости от системы).

**Брокер:** Redis, RabbitMQ, БД — хранят очередь и метаданные.

---

## Слайд 6: Модель «очередь задач»

```
[ Django / API ]  →  кладёт задачу  →  [ Redis / очередь ]
                                              ↓
[ Воркер 1 ]  ←  забирает задачу  ←  [ Redis ]
[ Воркер 2 ]  ←  забирает задачу  ←  [ Redis ]
```

**Producer** — код приложения (view, сервис). **Queue** — хранилище (Redis). **Worker** — процесс, который выполняет задачи.

---

## Слайд 7: Redis как брокер для очередей

Redis: быстрый key-value store в памяти. Подходит для очередей: списки (LPUSH/RPOP), TTL, персистентность при необходимости.

**RQ (Redis Queue)** использует Redis напрямую: одна очередь = один список. Простота — плюс для старта; при очень высокой нагрузке и сложных сценариях смотрят на RabbitMQ и Celery.

---

## Слайд 8: Что такое RQ

**RQ (Redis Queue)** — библиотека для фоновых задач на Python. Задачи — обычные функции; аргументы должны быть сериализуемы (pickle или JSON в зависимости от настроек).

- Один брокер — Redis.
- Запуск воркера: `rq worker`.
- Интеграция с Django: пакет **django-rq** (очереди в настройках, админка, management-команда).

---

## Слайд 9: Django 6 Tasks — встроенный контракт

С Django 6.0 в ядре появился **Tasks API**: декоратор `@task`, метод `.enqueue()`, объект результата `TaskResult`.

Django **не поставляет** продакшен-воркер: только контракт и встроенные бэкенды **Immediate** (выполнение в том же потоке) и **Dummy** (для тестов). Реальный фон — через бэкенды сообщества (django-tasks-database, django-tasks-celery и др.).

---

## Слайд 10: Django Tasks — пример кода

```python
from django.tasks import task

@task()
def send_welcome_email(user_id: int) -> str:
    user = User.objects.get(pk=user_id)
    send_mail(...)
    return "sent"

# В view или сервисе:
result = send_welcome_email.enqueue(user.pk)
# result.id — идентификатор для get_result() (если бэкенд поддерживает)
```

Аргументы должны быть **JSON-сериализуемы**.

---

## Слайд 11: Django Tasks — бэкенды

| Бэкенд | Поведение |
|--------|-----------|
| **ImmediateBackend** (по умолчанию) | Выполняет задачу сразу в текущем потоке |
| **DummyBackend** | Не выполняет, сохраняет результат для инспекции (тесты) |
| **django-tasks-database** | Очередь в PostgreSQL, воркер отдельно |
| **django-tasks-celery** | Использует инфраструктуру Celery |

Итог: один API в коде, смена бэкенда — в настройках.

## Слайд 12: django-rq — установка

```bash
pip install django-rq redis
```

**settings.py:**

```python
INSTALLED_APPS += ['django_rq']

RQ_QUEUES = {
    'default': {
        'HOST': 'localhost',
        'PORT': 6379,
        'DB': 0,
    },
}
```

Redis должен быть запущен локально или указан URL.

## Слайд 13: django-rq — определение задачи

Задача — обычная функция (или метод). Размещают в `tasks.py` приложения или в отдельном модуле.

```python
# shop/tasks.py
def send_order_confirmation(order_id: int) -> None:
    order = Order.objects.get(pk=order_id)
    send_mail(
        subject=f"Заказ #{order.id}",
        message=...,
        recipient_list=[order.email],
        fail_silently=False,
    )
```

Аргументы сериализуются (по умолчанию pickle). В очередь передают примитивы, id моделей, а не объекты ORM.

---

## Слайд 14: django-rq — постановка в очередь

```python
from django_rq import get_queue

def order_create_view(request):
    order = Order.objects.create(...)
    queue = get_queue('default')
    queue.enqueue(send_order_confirmation, order.id)
    return JsonResponse({"order_id": order.id})
```

**Вариант с задержкой:**

```python
queue.enqueue(send_order_confirmation, order.id, delay=300)  # через 5 мин
```

---

## Слайд 15: django-rq — запуск воркера

Воркер — отдельный процесс. В разработке:

```bash
python manage.py rqworker default
```

Для нескольких очередей:

```bash
python manage.py rqworker default high low
```

В проде воркер держат через systemd или supervisor, чтобы при падении он перезапускался.

---

## Слайд 16: django-rq — несколько очередей

Разделение по срочности или типу нагрузки:

```python
RQ_QUEUES = {
    'default': {...},
    'high':    {'URL': 'redis://localhost:6379/1'},
    'low':     {'URL': 'redis://localhost:6379/2'},
}
```

В коде: `get_queue('high').enqueue(critical_task, ...)`. Воркеры можно запускать с разным приоритетом или количеством на каждую очередь.

---

## Слайд 17: django-rq — результат и статус

```python
job = queue.enqueue(send_order_confirmation, order.id)
job.id          # идентификатор
job.get_status()  # queued, started, finished, failed, ...
job.result       # после завершения (если возврат значения нужен)
```

Для долгих задач: сохранить `job.id` в БД или сессию и по нему отдавать статус на фронт (или проверять через django-rq API).

---

## Слайд 18: django-rq — обработка ошибок и retry

При исключении в задаче RQ помечает задачу как failed. Можно настроить **retry**:

```python
from rq import Retry

queue.enqueue(send_order_confirmation, order.id, retry=Retry(max=3))
```

Либо в самой функции оборачивать вызов в try/except и логировать; при необходимости ставить задачу заново.

---

## Слайд 19: Celery — кратко

**Celery** — мощная система распределённых задач: несколько брокеров (Redis, RabbitMQ), расписание (beat), цепочки, группы, приоритеты.

Подходит для больших проектов и сложных сценариев. Минус — больше настроек и сущностей (broker, backend для результатов, beat, worker).

```python
@celery_app.task
def send_email(user_id):
    ...
send_email.delay(user_id)
```

## Слайд 20: Сравнение: Django Tasks, django-rq, Celery

|                  | Django Tasks   | django-rq | Celery       |
|------------------|----------------|-----------|--------------|
| Простота         | ⭐⭐⭐⭐⭐          | ⭐⭐⭐       | ⭐            |
| Инфраструктура   | минимальная    | Redis     | Redis/Rabbit |
| Масштаб          | низкий–средний | средний   | высокий      |
| DAG / workflows  | ❌              | ❌         | ✅            |

| Расписание       | через бэкенд   | rq-scheduler / cron | beat |

## Слайд 21: Когда что выбирать

- **Небольшой проект, быстрый старт:** Django 6 Tasks + бэкенд (например django-tasks-database) или сразу django-rq.
- **Уже есть Redis, нужны только фоновые задачи:** django-rq.
- **Сложные пайплайны, расписание, много воркеров:** Celery.
- **Минимум инфраструктуры (без Redis):** Django Tasks с бэкендом на БД.

---

## Слайд 22: Сериализация аргументов

В очередь попадают **сериализованные** аргументы. Передавать нужно:

- Примитивы (int, str, float, bool), списки, словари.
- ID моделей (например `user_id`, `order_id`), а не объекты `User`, `Order`.

**Нельзя:** объекты с открытыми соединениями, несериализуемые типы. В задаче по `id` заново делают `Model.objects.get(pk=...)`.

## Слайд 23: Django ORM в задачах

Воркер — отдельный процесс. У каждого процесса своё соединение к БД. После долгой работы соединение может разорваться.

Практика: в начале функции задачи при необходимости закрыть старые соединения:

```python
from django.db import connection
connection.close()
```

Либо использовать короткие транзакции и не держать соединение на время внешних вызовов.

---

## Слайд 24: Отложенное выполнение (scheduled)

**RQ:** пакет **rq-scheduler**. Задача ставится на время:

```python
from rq_scheduler import Scheduler
scheduler = Scheduler(connection=redis_conn)
scheduler.enqueue_at(datetime(2025, 4, 1, 12, 0), send_report, user_id=1)
```

**Периодические задачи:** cron раз в минуту вызывает management-команду, которая ставит задачи в RQ (или одну задачу «обработать всё накопившееся»).

---

## Слайд 25: Мониторинг очереди

- **Длина очереди:** `queue.count` или Redis `LLEN`.
- **django-rq:** встроенный веб-интерфейс (подключить urls) — очереди, джобы, failed.
- **Метрики:** экспорт в Prometheus (например через django-prometheus или кастомный endpoint) — число задач в очереди, время выполнения.

При росте длины очереди — масштабировать воркеры или оптимизировать задачи.

---

## Слайд 26: Прод: запуск воркера (systemd)

Воркер должен работать как служба и перезапускаться при сбое.

**Пример unit-файла (systemd):**

```ini
[Unit]
Description=RQ worker for Django
After=network.target redis.service

[Service]
User=www-data
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/venv/bin/python manage.py rqworker default
Restart=always

[Install]
WantedBy=multi-user.target
```

## Слайд 27: Прод: Redis

- **Персистентность:** RDB или AOF, чтобы при перезапуске Redis не потерять очередь (если приемлемо для задач).
- **Память:** лимит `maxmemory` и политика вытеснения; при переполнении возможна потеря задач.
- **Сеть:** воркеры и приложение подключаются к одному Redis (или кластеру); безопасность и доступ только с нужных хостов.

---

## Слайд 28: Прод: масштабирование воркеров

Несколько процессов воркера на одной очереди — задачи распределяются между ними. Запуск нескольких инстансов на одной машине или на разных (при общем Redis).

Ограничения: CPU/IO машины, лимиты БД, внешних API. Слишком много воркеров без тюнинга может перегрузить БД или почтовый сервер.

---

## Слайд 29: Типичные ошибки

- Передавать в задачу объект ORM вместо `id` — сериализация/десериализация и соединения с БД могут привести к ошибкам.
- Долгие задачи без таймаута — воркер «висит», очередь растёт.
- Игнорировать failed jobs — накапливаются, нужно разбирать причину и чинить код или данные.
- Не закрывать соединения к БД в долгих задачах — утечка или «connection already closed».

## Слайд 30: Пример: письмо после заказа

**Сценарий:** после создания заказа отправить клиенту письмо.

**View:** создаёт заказ, ставит в очередь задачу `send_order_confirmation(order_id)`, возвращает ответ.

**Задача:** по `order_id` загружает заказ, формирует текст, вызывает `send_mail`. При ошибке — логирование, при необходимости retry.

---

## Слайд 31: Пример: экспорт каталога в PDF

Тяжёлая операция: выборка из БД, генерация PDF (reportlab, weasyprint и т.д.). В view только постановка задачи и возврат «экспорт запущен, ссылка придёт на почту» или «статус по ссылке /job_id».

Задача генерирует файл, сохраняет в media/storage, отправляет письмо со ссылкой или кладёт ссылку в модель «экспорт» для скачивания по запросу.

---

## Слайд 32: Пример: генерация превью (thumbnail)

Загрузка изображения пользователем → в view сохраняем файл, ставим задачу `generate_thumbnails(image_id)`.

Задача: по `image_id` открывает файл, создаёт превью разных размеров, сохраняет, обновляет модель (поля с путями к превью). При ошибке — пометить запись как failed, логировать.

## Слайд 33: Тестирование задач

- **Юнит-тест:** вызывать функцию задачи напрямую с тестовыми аргументами (как обычную функцию); мокать почту, внешние API.
- **Интеграция с очередью:** в тестах использовать DummyBackend (Django Tasks) или поднимать тестовый Redis и проверять, что задача попадает в очередь и выполняется (при необходимости один воркер в потоке).

Не держать в CI тяжёлые воркеры по умолчанию — только при явной необходимости.

---

## Слайд 34: Отладка упавших задач

В django-rq в интерфейсе видно failed jobs: traceback, аргументы. Можно перезапустить задачу после исправления кода.

В логах воркера — полный traceback. Важно логировать `job.id` и аргументы при постановке в очередь, чтобы воспроизвести сбой.

---

## Слайд 35: Безопасность

- В аргументы задачи **не передавать** пароли, токены, персональные данные в открытом виде — только идентификаторы; чувствительные данные брать в задаче из БД/секретов.
- Ограничить доступ к Redis и к веб-интерфейсу django-rq (только для админов, по сети не светить).
- В проде не логировать тело писем и конфиденциальные поля в логах воркера.

---

## Слайд 36: Таймауты

Задача не должна висеть бесконечно. В RQ можно задать таймаут при постановке или глобально для воркера. При превышении задача помечается как failed, воркер переходит к следующей.

Долгие операции (большие отчёты, много файлов) разбивать на подзадачи или увеличивать таймаут осознанно и мониторить очередь.

---

## Слайд 37: Идемпотентность

Задача может быть выполнена повторно (retry после сбоя, ручной повтор). Желательно, чтобы повторное выполнение с теми же аргументами не приводило к дублям (два письма, два списания и т.д.).

Подходы: проверка «уже обработано» по флагу в БД, использование уникальных ключей, идемпотентные операции на стороне внешнего API.

---

## Слайд 38: Резюме

- Фоновые задачи разгружают HTTP-запрос и улучшают отказоустойчивость.
- Очередь = producer (приложение) + broker (Redis и др.) + worker.
- **Django 6 Tasks** — единый API в Django, бэкенд выбирается отдельно.
- **django-rq** — простой вариант на Redis для Django.
- **Celery** — для сложных сценариев и высоких нагрузок.
- В проде: воркер как служба, мониторинг очереди, обработка failed jobs, безопасность и таймауты.
