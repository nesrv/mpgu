## Слайд 1. RabbitMQ как брокер: что это за узел

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Давайте с нуля. Представьте две программы: одна что-то произвела, другая должна это получить, но не обязательно в ту же секунду. Если они будут звонить друг другу по HTTP и ждать ответа — упал второй сервис, и первый висит, таймауты, каскады. Брокер — это **посредник**: первая программа кладёт «письмо» в общий ящик, вторая забирает, когда готова. RabbitMQ как раз такой ящик с правилами сортировки.

Важно сразу отсечь заблуждение: это **не база данных**. В Postgres вы строите запросы, индексы, транзакции. В брокере — **очереди и маршруты**. Данные надолго хранить там не ваша основная цель.

Два порта запомнить наизусть: **5672** — это когда **программы** разговаривают с брокером по протоколу AMQP. **15672** — когда **вы** открыли браузер и смотрите картинку: очереди, графики, сколько сообщений висит. В лабе это Management-плагин; без него веб-морды нет, а «рабочий» порт 5672 всё равно есть.

На слайде схема: продьюсеры, консьюмеры, внутри узла — маршрутизация и хранилище. Erlang под капотом для нас сейчас не важен; важно, что узел **помнит** очередь, если консьюмер выключили. Закрывающая мысль: брокер — **буфер и почтальон**, не замена БД.

### Docker Compose

*К фрагменту YAML на слайде.*

- **Три отдельных контейнера** в одной сети Compose по умолчанию.
- Имя сервиса `rabbitmq` → **hostname** в сети; в URL из контейнера — `rabbitmq:5672`, не `localhost`.
- Образ **`rabbitmq:3-management`**: RabbitMQ 3.x + **Management** (веб **15672**).
- **`depends_on: [rabbitmq]`** — порядок старта, **не** готовность AMQP; в проде — healthcheck или retry в коде.
- **`build: .`** — сборка из Dockerfile; **`command`** — разные процессы из одного образа (`producer.py` / `consumer.py`).
- **`RABBITMQ_URL`** — соглашение **вашего** приложения, не стандарта AMQP. Подробнее — [приложение](#appendix-docker).

## Слайд 2. AMQP на «пальцах»: о чём договариваются клиент и брокер

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Программы не шлют друг другу «просто JSON в сокет» в хаотичном виде. Они говорят с брокером **строгим протоколом AMQP** — это бинарные кадры по TCP, как служебный язык. Вам в коде это спрячет библиотека: `basic_publish`, `basic_consume` — а под капотом идут кадры.

Аналогия **почта** с таблицы на слайде — рабочая на годы. **Сообщение** — письмо: тело и свойства. **Exchange** — сортировочный центр: сюда всё свалилось, дальше решают правила. **Binding** — правило «при таком ключе положи копию вот в эту очередь». **Queue** — ящик, где письма **ждут**. **Consumer** забирает; **ack** — «прочитал, выкинь из ящика».

Подчеркните: это **не HTTP**. Не обязательно JSON в теле — как договоритесь. И не путать с «положили строку в Redis»: у Redis нет той же модели **exchange + binding + маршрутизация** как у AMQP.

Студентам полезно один раз увидеть цепочку: приложение → TCP → брокер → внутри declare очередей, publish, consume, ack. Дальше по курсу вы это будете называть одними словами на ревью.

## Слайд 3. Virtual host: изоляция «песочниц»

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Один сервер RabbitMQ — как **дом**. **Vhost** — отдельные **квартиры**: в каждой свои очереди и exchange с **одинаковыми именами**, но это **разные** сущности. Очередь `tasks` в `/shop` и очередь `tasks` в `/billing` — две разные, они не пересекаются.

При подключении клиент указывает, **в какую квартиру** заходить — это кусок в URI после хоста. Неправильный vhost — как пришли в подъезд, а ключи не подходят: своих очередей не видите.

Зачем это в жизни? Одна железка брокера на компанию, а команды и среды **разделены**: dev, staging, prod — или отдельный vhost на сервис. Права выдают **на vhost**, и сосед случайно не сносит ваши очереди.

Не путать с Docker-контейнером: vhost — **логика внутри одного процесса брокера**, не отдельный контейнер. В лабе часто один vhost по умолчанию **`/`** — одной «квартиры» хватает. Переход: дальше про то, как внутри одной квартиры устроены **звонок** (connection) и **темы разговора** (channel).

### Docker Compose

*К фрагменту YAML на слайде.*

- Один контейнер **`rabbitmq`**; отличия только в **`environment`**.
- Пример URI: `amqp://guest:guest@rabbitmq:5672/shop` — логин/пароль, хост **`rabbitmq`**, порт **5672**, путь **`/shop`** = **имя vhost** `shop` (не путь на диске).
- Vhost по умолчанию в URI — **`/`**; синтаксис зависит от клиентской библиотеки.
- Vhost **`shop`** / **`billing`** должны **существовать** в брокере (UI или init), иначе ошибка доступа.

## Слайд 4. Соединение (connection) и канал (channel)

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

**Connection** — это реальный **TCP до брокера**, долгий «звонок». Его открывать дорого: рукопожатие, ресурсы. Поэтому в приложении обычно **один connection на процесс** (или немного), а не на каждое сообщение.

**Channel** — виртуальные **линии внутри того же звонка**. Хотите объявить очередь, опубликовать, подписаться — делаете это через канал. Несколько потоков или задач могут использовать **разные каналы** поверх одного TCP.

Если оборвали **TCP** — рвутся **все** каналы сразу. Если на каком-то канале жёсткая ошибка, в старых клиентах иногда падает весь connection — отсюда осторожность с многопоточностью и одним channel на всё.

В Docker Compose вы не «настраиваете канал в YAML» — каналы живут **в коде** внутри процесса `app` или `worker`. Практика: pika, aio-pika — создали connection при старте, дальше везде работаем через **channel**. Связка со следующим слайдом: по каналу вы как раз говорите «положи в exchange» и «дай из очереди».

## Слайд 5. Путь сообщения: publish → exchange → binding → queue

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Главная мысль, которую нужно дожать: в нормальной схеме вы **не кидаете сообщение «в очередь по имени»** так, будто это единственный адрес. Вы публикуете в **exchange** с **routing key** — а бroker по **bindings** решает, в **какие** очереди полетит копия.

**Exchange** — не склад на полку. Он **маршрутизирует** и отпускает. **Долго ждать** сообщение будет уже в **queue**. Консьюмер читает **из очереди**; **ack** говорит брокеру: убери из очереди после успешной обработки (упрощённо).

На слайде вторая схема показывает силу модели: один exchange, несколько bindings с **разными ключами** — разные очереди для биллинга, логистики, аудита. Маршрутизация **на стороне брокера** при каждом publish.

Compose на слайде — каркас: rabbitmq, producer, consumer; сами exchange/queue/binding объявляются **в коде** при старте, чтобы не было гонки «consumer раньше producer». Итог фразой: запомните стрелки **publish → exchange → binding → queue → consumer → ack**.

### Docker Compose

*К фрагменту YAML на слайде.*

- **Два сервиса** (`producer`, `consumer`) — на схеме видны **разные роли**; в репо может быть один пакет и один `build: .`.
- RabbitMQ **один**, очередь **общая** для обоих контейнеров.
- Старт в любом порядке ок, если в коде есть **declare** очередей/exchange.
- Compose **не создаёт** очереди и exchange — только **приложение** по AMQP.

## Слайд 6. Типы exchange: как брокер выбирает очереди

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Теперь «как сортировочный центр решает». **Direct**: routing key должен **точно совпасть** с ключом в binding — как штамп на конверте. **Fanout**: не смотрим на ключ — **копия в каждую** привязанную очередь, как газета каждому подписчику. **Topic**: ключи как путь, в binding — **шаблоны** с `*` (одно слово) и `#` (хвост). **Headers** на вводном курсе редко — запомните, что бывает. **Default exchange** `""` — ленивый режим для прототипов: ключ считается **именем очереди**.

На экзамене и на работе вас будут спрашивать: «почему не default везде?» Ответ простой: в росте системы **явный именованный exchange и bindings** проще объяснить на доске и в документации, чем магия пустого exchange.

Mermaid на слайде — три картинки в одной: direct, fanout, topic. Не надо зубрить картинку; надо уметь **нарисовать от руки** для своей задачи. Дальше — что происходит с **очередью** как буфером и что смотреть в UI.

## Слайд 7. Очередь, устойчивость сообщений и Management UI

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

**Очередь** — буфер. Воркер медленный — хвост растёт, как пробка. **Durable** у очереди значит: после перезапуска брокера **метаданные** очереди восстановятся. **Persistent** у сообщения — «тело постарайся дописать на диск»; это не абсолютная страховка без нормальной эксплуатации и политики, но лучше, чем только память.

**Exclusive** и **auto-delete** — для временных штук; в проде без понимания легко выстрелить себе в ногу. Management UI — первая линия обороны: **ready** (ждут), **unacked** (отдали воркеру, ack ещё не пришёл), скорости publish/consume.

Схема prefetch на слайде вводит мост: сообщение может висеть «у воркера в буфере» до ack. Пока не ack’нули — в типичной настройке **другому** воркеру то же сообщение не отдадут. Quorum queues — тема уровня продвинутого курса; для нас достаточно **classic** и понимания компромиссов.

### Речь

Сейчас коротко зафиксируем цель. После пары часов вы должны уметь **нарисовать на доске** поток: кто шлёт, в какой exchange, какие очереди, кто читает, зачем **две** очереди, а не одна. Это общий язык с тимлидом и на защите лабы.

## Слайд 9. Что считаем «потоковым приложением» здесь

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

«Потоковое» здесь — не обязательно видео по UDP. Это когда **события идут непрерывно или пачками**, а обработка **отвязана** от одного HTTP-запроса: запрос быстро ответил «приняли», дальше жизнь идёт в фоне. Лента, уведомления, чат, статусы заказа, обработка загрузок — всё из одной кухни.

Примеры с рунета на слайде — **иллюстрация класса задач**, не утверждение про внутренний стек ВК или Озона. Нам важно **куда смотреть глазами архитектора**: очередь задач, подписчики, асинхронщина.

Граница: кодеки WebRTC и RTP — другая лекция; здесь — **сообщения в брокере** и кто их ест. Плавный переход: если поток такой — почему не крутим всё через прямой HTTP между сервисами?

## Слайд 10. Зачем между сервисами брокер, а не прямой HTTP

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

HTTP между сервисами — как **синхронный звонок**: ты не кладёшь трубку, пока сосед не ответил. Сосед упал — ты ждёшь таймаут, нагрузка расползается каскадом. Брокер — как **ящик у двери**: положил письмо и пошёл дальше; сосед заберёт, когда оживёт.

Три выгоды одной фразой: **декуплинг по времени** (не ждём свободного воркера), **декуплинг по доступности** (буфер, пока consumer лежит — с оговорками про диск и лимиты), **масштаб чтения** (несколько воркеров на одну очередь или разные очереди с одного exchange).

Цена: **ещё один компонент** в инфраструктуре — мониторинг, бэкапы политики, алерты. И семантика: с ручным ack по умолчанию вы живёте в мире **at-least-once** — повторы возможны, к этому вернёмся. Итог: брокер покупают **за буфер и развязку**, платят **операционкой**. Дальше — резюме цепочки в четырёх словах перед паттернами.

## Слайд 11. Модель маршрутизации: резюме перед паттернами

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Запомните наизусть четыре опоры: **exchange → bindings → queue → consumer**. Producer публикует в exchange (кроме случая default). Binding связывает exchange с очередью и задаёт правило. Consumer читает из **очереди**. Ack убирает сообщение из очереди после обработки.

Типы exchange — direct, topic, fanout, headers. Схема на слайде — общая: несколько сервисов пишут, несколько воркеров читают; стрелки **в exchange** слева, **из очередей** справа.

Отсылка назад: детали узла, AMQP, vhost, channel, типы exchange — слайды 1–7. Впереди — **паттерны**: одни и те же кубики, разные схемы сборки. Первый паттерн — буфер и сглаживание пиков.

## Слайд 12. Паттерн: буфер — одна очередь, сглаживание пиков

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Пользователь нажал «загрузить сторис» — API **не должен** крутить тяжёлый ffmpeg в том же запросе. Быстро ответили «приняли», задачу **в очередь**, клиент спокоен. Воркеры забирают с другой скорости. Это **тикеты в жёлоб**: кассир один, очередь растёт в пик, но HTTP не висит.

Риск честно проговорить: если воркеров мало, **очередь раздувается** — нужны метрики и масштабирование. Примеры ВК, ОК, Дзен на слайде — про **класс задачи**.

Compose: отдельно `api` и `encoder` — часто один образ, разные команды. Это базовый producer–consumer; от него отталкиваются все остальные. Следующий шаг: **несколько** воркеров на **одну** очередь.

### Docker Compose

*К фрагменту YAML на слайде.*

- **`api`** — публикует задачи в Rabbit; **`encoder`** — только консьюмер (отдельный процесс).
- **`ports: ["8000:8000"]`** — `хост:контейнер`; с ноутбука → `http://localhost:8000`.
- У **`encoder`** наружу портов нет — только исходящий коннект к `rabbitmq:5672`.
- Один **`RABBITMQ_URL`** — один логический хост брокера в сети Docker.
- Общий **`build: .`**, разный **`command`** — один образ, две роли.

<a id="s13"></a>

## Слайд 13. Паттерн: work queue — несколько воркеров, одна очередь

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Одна очередь `transcode.tasks`, три воркера. Rabbit раздаёт сообщения **конкурирующим** подписчикам: кто освободился — взял следующее. Масштаб не «разгоняем один CPU», а **добавляем воркеров**.

Prefetch — сколько неack’нутых сообщений воркер держит у себя; от этого зависит, насколько **честно** делится нагрузка. Идемпотентность — отдельный слайд; здесь только намёк: при сбоях одно сообщение **могут доставить снова**.

Порядок **глобально** между воркерами не гарантирован — если нужен строгий порядок по сущности, дизайн другой (partition, одна очередь на ключ, один consumer). Рутуб и Okko на слайде — иллюстрация **однотипных тяжёлых задач**. В Compose три одинаковых `worker_N` — в Kubernetes это обычно **replicas**.

### Docker Compose

*К фрагменту YAML на слайде.*

- **`worker_1` … `worker_3`** — три контейнера, одна команда `python transcode_worker.py`.
- В коде — **одна очередь**; Rabbit раздаёт сообщения конкурирующим консьюмерам.
- В проде то же часто как **replicas** одного сервиса (Swarm/Kubernetes).
- Три сервиса = три **процесса**, не три потока в одном процессе.
- **`uploader`** — только producer в очередь.

## Слайд 14. Паттерн: pub/sub через fanout — своя очередь на каждый сервис

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Событие одно — **пост создан** — а ленте, поиску и рекомендациям нужна **каждому своя копия**. Если все три сервиса подписаны на **одну** очередь, они **делят** сообщения: один получил, двое не увидели. Это не pub/sub, это **конкуренция за работу**.

Правильно: **fanout exchange**, у каждого подписчика **своя очередь**, в каждую прилетает **полная копия**. Медленный поиск **не блокирует** ленту — у них разные хвосты.

Типичная ошибка на экзамене — нарисовать fanout в одну общую очередь «для всех». Запомните картинку с тремя очередями. Compose: `posts` публикует, три `*_worker` читают разные очереди.

### Docker Compose

*К фрагменту YAML на слайде.*

- Один **`posts`** (producer) + три consumer с **разными** `command` / скриптами.
- У каждого consumer — **своя очередь** на одном fanout exchange.
- Один **`build: .`** ≠ одна логика: меняются **`command`** и имена очередей в коде.

<a id="s15"></a>

## Слайд 15. Паттерн: direct exchange — маршрутизация по точному ключу

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Не «всем копию», а **разным отделам разные типы событий**. Ключ `order.paid` — строго в очередь биллинга, `order.shipped` — в логистику. Без шаблонов и звёздочек: **точное совпадение** routing key с binding.

Удобно, когда типов событий **перечислимо** и без глубокой иерархии. Когда ключей сотни и появляются уровни `a.b.c` — смотрят в сторону **topic**. Озон и Маркет на слайде — про **статусы заказа** как именованные потоки.

<a id="s16"></a>

## Слайд 16. Паттерн: topic exchange — иерархия и шаблоны

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Ключи как путь: `chat.room123.message`. Подписчик говорит брокеру: «мне всё `chat.#`» или «мне `*.paid`». Звёздочка — одно слово, решётка — хвост любой длины. Гибче direct, но нужна **дисциплина имён** и ревью схемы, иначе через год никто не поймёт, почему сообщение ушло не туда.

Связка с лабой: ваши `chat.message.created` и `persisted` — как раз topic-мышление. Compose — один backend публикует, несколько consumer’ов с разными шаблонами.

### Docker Compose

*К фрагменту YAML на слайде.*

- Структура файла как у fanout: один publisher, несколько reader-сервисов.
- Отличие — в **bindings** (topic, шаблоны ключей), не в синтаксисе Compose.

<a id="s17"></a>

## Слайд 17. Паттерн: конвейер (pipeline) — стадии и новые события

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Один воркер «съел всё» — для учёбы иногда ок, в проде чаще **цепочка**: приняли → очередь → записали в БД → **новое** событие «сохранено» → следующий этап (например, раздача в WebSocket). Каждая стадия заканчивается **осмысленным событием** для следующей очереди.

На слайде — ваша учебная схема: web публикует `created`, worker читает incoming, пишет Postgres, публикует `persisted`, web потребляет и шлёт в комнату. Зачем: **web не тащит** тяжёлую запись в БД на каждый чих в том же стиле, что приём; границы ответственности читаются с доски.

UI обновляют после **persisted** — меньше ситуации «в чате видно, в БД пусто». Следующий слайд — **зачем отдельный контейнер worker** именно в лабе.

### Docker Compose

*К фрагменту YAML на слайде.*

- Четыре сервиса: **`rabbitmq`**, **`postgres`**, **`web`**, **`worker`**.
- **`web`** и **`worker`**: часто один **`build: .`**, разный **`command`** (uvicorn vs `worker.py`); у worker обычно **нет** проброса HTTP наружу.
- **`depends_on`** — старт контейнеров, не «Postgres уже принимает коннекты» (без healthcheck).
- **`DATABASE_URL`**: хост **`postgres`** = имя сервиса; префикс `postgresql+asyncpg://` — договорённость SQLAlchemy/async.
- **`ports: ["8100:8000"]`** — с хоста **8100**, в контейнере слушает **8000**.
- **`POSTGRES_*`** — стандартный entrypoint образа `postgres`; пароли в git для учебы, в проде — **secrets**.
- **Зависимости:** `web` и `worker` тянут и **брокер**, и **БД**; `rabbitmq` с `postgres` **ни от кого** в YAML не зависят. См. [§1а](#adx1b) про порядок старта и `localhost` vs имя сервиса.

<a id="s18"></a>

## Слайд 18. Роль отдельного worker в лабе

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

`worker.py` — отдельный процесс: сидит на **incoming**, коммитит в Postgres, шлёт **persisted**. Web в этой модели **не обязан** в одном потоке с HTTP крутить всю запись в БД для каждого сообщения чата.

Зачем учебно: показать **очередь между приёмом и записью**, масштаб (несколько worker, один web), устойчивость (очередь переживает рестарт web), явные стадии **created** vs **persisted**.

Честность: в БД писать прямо из `main.py` **можно** — тогда для этой задачи отдельный worker не нужен как в инженерии, но в курсе он нужен, чтобы **увидеть паттерн** на диаграмме и в Docker Compose. Отсылка к `about.md` и `containers-architecture.mmd`.

### Docker Compose

*К фрагменту YAML на слайде.*

- Фрагмент **укорочен**; полный пример — [слайд 17](#s17).
- Сервис **`worker`** в `services:` = отдельный контейнер и процесс; в логах — и uvicorn, и worker.
- Остановили только **`web`** — **`worker`** может продолжать читать очередь и писать в БД (WS подождёт).

<a id="s19"></a>

## Слайд 19. Слой доставки в браузер: WebSocket и брокер

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Браузер **не подключается** к RabbitMQ. С брокером говорят **серверы**. Браузер держит WebSocket с **вашим** backend; backend читает очередь и решает, в какой сокет пушнуть.

Если инстансов web **два**, а событие пришло на тот, где **нет** сокета пользователя — получите классическую проблему. Варианты: sticky sessions, общий pub/sub между инстансами (Redis и т.д.), отдельный сервис уведомлений. На вводном курсе достаточно **назвать проблему** и не продавать иллюзию «Rabbit сам доставит в браузер».

### Docker Compose

*К фрагменту YAML на слайде.*

- **`web1`** / **`web2`**: один образ и `command`, разные **`ports`** на хосте (**8101**, **8102**) — имитация масштаба.
- Оба читают одну очередь **`persisted`** — для **прода** спорно (сообщение получит **один** инстанс); на лекции — обсудить sticky / общую шину.
- Комментарий про **nginx** + sticky — типичный прод: один **443**, один пользователь → тот же backend.

<a id="s20"></a>

## Слайд 20. Паттерн: RPC поверх очередей

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Вместо «вызвал функцию — сразу ответ» кладём **заявку** в очередь. В заявке указываем **куда ответить** (`reply_to`) и **номер заказа** (`correlation_id`). Исполнитель кладёт ответ в очередь ответов; клиент сопоставляет по id.

Плюс — гасим пики и отвязываемся по времени. Минусы — **латентность**, таймауты, сложнее отладка. Публичное API чаще HTTP/gRPC; RPC через Rabbit — **внутренняя** тяжёлая обработка, антиспам, очередь к ML. Не перепутать с «всё в мире должно быть RPC в Rabbit».

### Docker Compose

*К фрагменту YAML на слайде.*

- **`rpc_client`** и **`rpc_server`** — два процесса; очереди в **коде** при старте.
- DNS по именам сервисов внутри сети Compose.
- **`ports`** не нужны, если с хоста в эти сервисы не ходят — «API» = очередь.

<a id="s21"></a>

## Слайд 21. Семантика доставки и подтверждения (ack)

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

**Auto ack**: сообщение считается «съеденным», как только его доставили в память клиента. Воркер упал через секунду — **потеря**. Для денег и заказов так нельзя.

**Manual ack**: подтверждаем **после** успешной обработки. Упали до ack — брокер **отдаст снова** — это **at-least-once** по попыткам. Волшебного exactly-once «только настройкой брокера» нет: нужна **идемпотентность** в коде, уникальные ключи, иногда outbox.

Закрывающая мысль: проектировать обработчик с мыслью «это выполнится **один или два раза** — и оба раза ок».

<a id="s22"></a>

## Слайд 22. Dead Letter Exchange (DLX) и DLQ

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Сообщение битое, внешний API не отвечает, схема не та — крутить вечно в основной очереди **больно**: засоряете метрики и тратите CPU. **DLX/DLQ**: отказ, TTL, лимит повторов — сообщение уезжает на **полку проблемных**. Там его разбирает человек или скрипт.

Без DLQ типичный исход: либо вечный цикл, либо тихая потеря. Настройка через аргументы очереди `x-dead-letter-exchange` — в YAML не видно, видно в **коде** declare. UGC и модерация на слайде — про **реальные битые payload’ы**.

### Docker Compose

*К фрагменту YAML на слайде.*

- Упрощённо: только **`rabbitmq`** и **`worker`** — фокус на идее DLX.
- **`x-dead-letter-exchange`** — в **`queue_declare`** (код), **не** в `docker-compose.yml`.
- Отдельный контейнер для DLQ не обязателен: второй consumer или скрипт.

<a id="s23"></a>

## Слайд 23. Повторные попытки (retry)

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

«Упало — повтори» звучит просто. Если упало **всё** (например, база), retry превращается в **шторм**: брокер и воркеры захлёбываются. Нужны **лимиты**, **задержки**, **экспоненциальная** стратегия, **метрики** на глубину retry-очереди.

Связать с идемпотентностью: повтор — норма, не паника. На дашборде обязательно видеть retry и DLQ.

<a id="s24"></a>

## Слайд 24. Идемпотентность и дедупликация

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Одно событие доставили **дважды** — не баг брокера, а **ожидаемый** мир при requeue и рестартах. Обработчик как кнопка лифта: нажали дважды — поехали один раз. Инструменты: **event_id**, уникальный индекс в БД, таблица «уже обработано».

Ловушка: записали в Postgres и **потом** упали до publish в Rabbit — классика для **transactional outbox** на старших курсах; здесь — только указать на проблему. Без идемпотентности «ровно один раз» вам никто не подарит.

<a id="s25"></a>

## Слайд 25. Порядок сообщений

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Одна очередь, один consumer — порядок обычно **как клали**. Несколько воркеров на одну очередь — кто быстрее обработал, тот и «впереди» в бизнес-смысле: **глобальный порядок не гарантирован**.

Если порядок нужен по **user_id** или **room_id** — partition: своя очередь на ключ, или один consumer на ключ, или один writer в БД для комнаты. Для чата часто важен порядок **внутри комнаты**, не глобально по всему кластеру.

<a id="s26"></a>

## Слайд 26. Prefetch (QoS) и backpressure

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Prefetch — «не кидай мне сразу сотню сообщений; держи на брокере, пока я не ack’ну первые N». Иначе один жадный воркер забирает пачку, остальные **простаивают**. Слишком маленький prefetch — растёт латентность на round-trip.

Backpressure в голове: воркер **не должен** бесконечно копить необработанное в RAM; prefetch — один из предохранителей. Настраивать после замеров времени обработки, не «на глаз из туториала».

## Слайд 27. Наблюдаемость

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Без метрик вы слепы. Очередь растёт — воркеры не справились или мертвы. **Unacked** растёт — воркеры зависли или забыли ack. Management UI — бесплатный старт; дальше Prometheus-плагин, логи с **correlation_id** в сообщении.

Архитектура «с очередью» без графиков глубины очереди — подпись под будущий инцидент в пятницу вечером.

<a id="s28"></a>

## Слайд 28. RabbitMQ и Kafka

| | |
|:---|:---|
| **Ориентир** | 3–4 мин. |

### Речь

Не holy war. **Rabbit** — очереди, гибкая маршрутизация, хорош для **задач** и **интеграции** сервисов, consumer тянет сообщения в привычной картине курса. **Kafka** — **лог** с партициями, большой throughput, хранение истории и переигрывание, часто центральная **шина событий** при огромном объёме.

На курсе Rabbit — компактная модель для **паттернов**, которые потом мысленно переносятся на другие системы. Выбор — по требованиям к задержке, объёму, хранению, команде и операционке.

## Приложение. Подробно про Docker Compose в лекции

Фрагменты YAML в `lect-arch-mq.md` помечены как **«Примерно в Docker Compose»** — это не готовый прод, а **иллюстрация ролей** процессов. Ниже — что имеет смысл проговорить студентам «по строчкам» и какие оговорки не забыть.

**Содержание приложения:** [1. Файл](#adx1) · [2. RabbitMQ](#adx2) · [3. build / command / env](#adx3) · [4. Воркеры](#adx4) · [5. Web + БД](#adx5) · [6. Чего нет в YAML](#adx6) · [7. Таблица по слайдам](#adx7)

<a id="adx1"></a>

### 1. Что такое этот файл

**Docker Compose** читает YAML (часто `docker-compose.yml`). В корне — ключ **`services:`**: каждый подключённый блок — **именованный сервис** = по умолчанию **один контейнер** при `docker compose up` (если не масштабировать). Команда `docker compose up` создаёт **одну общую сеть**, куда попадают все сервисы из файла; внутри этой сети **DNS-имя** контейнера совпадает с **именем сервиса** (`rabbitmq`, `web`, `worker`…). Поэтому в URL пишут `@rabbitmq:5672`, а не `localhost`: с хоста ноутбука `localhost:5672` — это проброс на хост, а **между контейнерами** — логика «сервис как hostname».

<a id="adx2"></a>

### 2. Разбор типового куска с RabbitMQ

Пример как на ранних слайдах:

```yaml
services:
  rabbitmq:
    image: rabbitmq:3-management
    ports:
      - "5672:5672"
      - "15672:15672"
```

- **`services:`** — начало списка контейнеров приложения.
- **`rabbitmq:`** — имя сервиса (и hostname в сети). Можно назвать и `broker`, тогда в URL будет `@broker:5672`.
- **`image: rabbitmq:3-management`** — готовый образ с Docker Hub: major-версия **3**, тег **`management`** добавляет **веб-UI** на порту **15672**. Без `-management` был бы только AMQP **5672** — для лабы с Management удобнее с тегом.
- **`ports: "5672:5672"`** — формат **`хост:контейнер`**. Левая часть — порт на вашей машине (Windows/Mac/Linux), правая — что слушает процесс **внутри** контейнера. Так вы с хоста открываете браузер `http://localhost:15672` и клиент `localhost:5672`. **Внутри** сети Compose другие контейнеры ходят на **`rabbitmq:5672`**, проброс им не обязателен.

<a id="adx3"></a>

### 3. Приложения: `build`, `command`, `environment`

```yaml
  producer:
    build: .
    command: python producer.py
    depends_on: [rabbitmq]
    environment:
      RABBITMQ_URL: amqp://guest:guest@rabbitmq:5672/
```

- **`build: .`** — собрать образ из `Dockerfile` в текущей директории (контекст сборки). Альтернатива — только **`image: myregistry/producer:1.0`** без сборки локально.
- **`command:`** — заменяет **CMD** (и иногда взаимодействует с ENTRYPOINT) образа. Один образ — разные сервисы с разным `command` дают **разные процессы** (producer vs consumer).
- **`depends_on: [rabbitmq]`** — контейнер `producer` стартует **после** контейнера `rabbitmq` в смысле порядка запуска Compose. **Не гарантирует**, что RabbitMQ уже принимает соединения: возможна гонка на секунды. В учебных примерах часто достаточно; в проде добавляют **healthcheck** и `depends_on: condition: service_healthy` (в зависимости от версии Compose) или **retry** в коде при коннекте.
- **`environment:`** — переменные внутри контейнера. **`RABBITMQ_URL`** — соглашение **вашего** приложения (читаете в `os.environ`); сам RabbitMQ эту переменную не требует для своей работы.
- **`amqp://guest:guest@rabbitmq:5672/`** — схема AMQP, логин/пароль по умолчанию у образа для гостевого доступа, хост **`rabbitmq`**, порт **5672**, путь **`/`** — vhost по умолчанию. Для учебной сети Docker **guest** с удалённых хостов часто отключён политикой образа — с хоста ноутбука иногда нужны доп. настройки; **между контейнерами** обычно работает.

**Безопасность (одна фраза лектору):** `guest:guest` в репозитории — норм для локалки и пары; в проде — отдельные пользователи, пароли из secrets, vhost и TLS.

<a id="adx4"></a>

### 4. Несколько воркеров в одном файле

Три сервиса `worker_1`, `worker_2`, `worker_3` с одинаковым `build` и `command` — это **три контейнера**, три процесса Python. В оркестраторах то же самое часто делают **replicas** одного сервиса. Смысл для лекции: горизонтальное масштабирование **читателей** очереди.

<a id="adx5"></a>

### 5. Веб + БД + брокер (pipeline лабы)

- **`postgres:`** с `POSTGRES_USER`, `POSTGRES_PASSWORD`, `POSTGRES_DB` — стандартный entrypoint образа `postgres` создаёт пользователя и БД при первом запуске (данные в **volume**, если его добавили; в коротких примерах на слайде volume может быть опущен — тогда при `docker compose down -v` данные пропадут).
- **`web`** пробрасывает порт наружу — к нему ходит браузер. **`worker`** порт наружу не пробрасывает — он только коннектится к `rabbitmq` и `postgres`.
- **`DATABASE_URL`** и **`RABBITMQ_URL`** должны использовать **имена сервисов** `postgres` и `rabbitmq`, а не `127.0.0.1`, потому что **localhost внутри контейнера** — это сам контейнер, а не хост ноутбука.

<a id="adx6"></a>

### 6. Что в Compose не живёт

Имеет смысл явно сказать:

- **Очереди, exchange, bindings** создаёт **код приложения** (или отдельный init-job), а не YAML Compose — если только вы специально не гоняете `rabbitmqadmin`/конфиг через init-контейнер.
- **Prefetch, ack, DLX** — настройки **клиента AMQP** или **аргументы очереди** в коде, не директивы `docker-compose`.
- **Каналы AMQP** — внутри процесса; в Compose вы описываете только **сколько процессов** и с какой командой они стартуют.
