1. RabbitMQ — посредник-почтальон

Одна программа кладёт сообщения, другая забирает, когда готова. Брокер помнит очередь, если получатель временно выключен.

  • Протокол AMQP 0-9-1 поверх TCP; движок — Erlang VM
  • Порт 5672 — клиентский; 15672 — Management UI (веб-интерфейс)
  • Не Postgres: брокер — для очередей и маршрутов, а не сложных запросов

2. AMQP — язык клиента и брокера

Не HTTP и не «просто JSON в Redis» — бинарный протокол поверх TCP. Ваш код вызывает channel.basic_publish — библиотека переводит в AMQP-кадры.

AMQP-идеяНа пальцах
СообщениеПисьмо: тело (payload) + свойства (routing, content-type…)
ExchangeСортировочный центр — маршрутизирует, сам не хранит
BindingПравило «ключ → очередь»
QueueЯщик с письмами — ждут получателя
ConsumerЗабирает из ящика; ack — «прочитал, убери»

3. Virtual host — изоляция «квартир»

  • Один RabbitMQ — как дом; vhost — отдельные квартиры; tasks в /shop и tasks в /billing — это разные очереди
  • URI: amqp://user:pass@rabbit:5672/shop
  • Изоляция прав: dev / prod, команда A / команда B. Не Docker-контейнер — логика внутри одного брокера

4. Connection и Channel

Connection — долгий TCP-звонок (открытие дорого, держат постоянным). Channel — виртуальные линии в одном TCP. Обрубили TCP — рвутся все каналы.

Одно TCP на процесс — много каналов на потоки / asyncio-задачи.

7. Очередь, устойчивость и Management UI

  • durable — очередь переживёт перезапуск брокера (метаданные)
  • delivery_mode = persistent — тело сохраняется на диск
  • Management UI :15672 — ready / unacked, rate publish/consume, привязки
  • Quorum queues (RabbitMQ 3.8+) — для критичных сценариев в проде

Пока ack не отправлен — сообщение unacked, другому consumer'у не отдаётся.

8. Тема и результат занятия

После лекции вы умеете нарисовать на доске: кто шлёт, куда падает, кто читает, зачем две очереди а не одна. Это язык общения с коллегами при разборе фич и ревью архитектуры.

  • Разложить потоковую систему на exchange, очереди и сервисы
  • Назвать типовые схемы, которые узнают ревьюеры
  • Связать с учебным проектом: FastAPI + RabbitMQ + worker + PostgreSQL

9. Что считаем «потоковым» здесь

Не обязательно видеопоток по UDP. Главное — события идут непрерывно или пачками, обработка асинхронна от HTTP-запроса пользователя.

  • Лента и уведомления → ВКонтакте, Одноклассники
  • Чат стрима → Рутуб, VK Live
  • Статусы заказа → Озон, Яндекс Маркет
  • Фоновая обработка сторис → ВК, ОК

Достаточно очереди задач и событий между сервисами — это уже «поток». Граница лекции — сообщения в брокере; кодеки и RTP для WebRTC — отдельная тема.

10. Зачем брокер, а не прямой HTTP

HTTP: если сосед занят или упал — вы ждёте, падаете по таймауту, давите каскадом. Брокер — ящик у двери: положили письмо и пошли дальше, сосед заберёт когда проснётся.

ПроблемаЧто даёт брокер
Продьюсер ждёт ConsumerДекуплинг по времени
Consumer упалБуферизация сообщений в очереди
Нужно масштабировать чтениеНесколько воркеров на одну очередь
Разные SLA у потребителейНезависимые очереди и приоритеты

Цена: новая инфраструктура, семантика at-least-once, операционка.

11. Модель маршрутизации — резюме перед паттернами

Четыре слова: publish → exchange → binding → queue → consumer. Всё остальное — вариации этой цепочки.

12. Паттерн: буфер — сглаживание пиков

Рунет: загрузка сторис ВК/ОК, постобработка контента в Дзен

API быстро отвечает «принято» — тяжёлая работа встаёт в хвост. Воркер обслуживает не всех сразу. Как тикет в жёлоб: кассир (воркер) работает в своём темпе.

Риск: рост очереди при перекосе скоростей — нужны метрики и масштабирование consumer'ов.

13. Паттерн: work queue — конкурирующие подписчики

Рунет: транскодинг видео — Рутуб, Okko, Кинопоиск

  • Одна общая куча задач, много рабочих рук — кто освободился, взял следующий тикет
  • Масштаб: запустить второй воркер, а не ускорять один
  • Prefetch (QoS) — сколько unacked держит воркер; влияет на честность нагрузки
  • Идемпотентность — одно и то же сообщение могут попробовать дважды при сбоях

17. Паттерн: конвейер (pipeline)

Рунет: чат стрима Рутуб, мессенджер VK — принято → сохранено → доставлено

Не один воркер «сделал всё», а цепочка: «приняли» → очередь → «сохранили в БД» → новое событие «сохранено» → следующий этап (пуш в браузер). Каждая станция заканчивает работу и передаёт дальше по ленте.

  • Веб-слой не обязан писать в БД сам — разгрузка и чёткие границы
  • UI обновляется после факта персистенции — меньше рассинхрона
  • Учебный эталон — containers-architecture.mmd

18. Роль отдельного worker.py

  • Разгрузка web — сообщения в durable-очереди, web не блокируется тяжёлой работой
  • Масштаб — несколько worker при одном web
  • Устойчивость — рестарт web не теряет сообщения, worker доработает
  • Явная модель — «создано» vs «сохранено»; UI видит факт после persisted

Писать в БД из main.py можно — тогда отдельный worker.py не нужен. В лабе он нужен, чтобы увидеть паттерн на диаграмме и в коде.

19. Слой доставки в браузер: WebSocket + брокер

RabbitMQ не лезет в браузер — с ним говорят серверы. Браузер держит WebSocket с вашим backend; backend читает очередь и решает, кому из открытых сокетов отправить обновление.

Проблема: событие пришло на инстанс 1, но сокет пользователя — на инстансе 2. Решение: Redis pub/sub или sticky sessions.

20. Паттерн: RPC поверх очередей

Рунет: тяжёлый скоринг, внутренние ML-сервисы с очередью задач

В заявке — reply_to (куда ответить) и correlation_id (номер заказа). Исполнитель кладёт ответ в очередь ответов, клиент сопоставляет по id.

Минусы: задержка, сложность таймаутов. Для публичного API чаще HTTP/gRPC.

21. Семантика доставки: ack и at-least-once

РежимЧто происходит при падении workerКогда
auto ackСообщение сгорело при доставке → потеряУчёба, некритичные данные
manual ackБрокер отдаст снова → at-least-onceПродакшен, деньги, состояние

Нет волшебного exactly-once без идемпотентности обработчика. Повтор — норма, а не авария. Проектируйте обработчик устойчивым к повтору.

22. Dead Letter Exchange (DLX) — полка «проблемных»

Рунет: модерация UGC — Рутуб, ВК, Дзен; битый payload, таймаут внешнего API

Без DLQ битые сообщения крутятся вечно или теряются молча — оба варианта хуже. Очередь настраивается аргументом x-dead-letter-exchange.

23. Повторные попытки — осторожно с нагрузкой

«Упало — попробуй ещё раз» звучит просто. Если упало всё сразу (база лежит), retry превращается в шквал повторов и может положить и брокер.

  • Простая retry-очередь с TTL и возвратом → возможен retry-шторм
  • Лучше: exponential backoff на уровне приложения или планировщика
  • Метрики глубины retry и DLQ — обязательны в дашборде
  • Согласовать с идемпотентностью: повтор — норма, не авария

24. Идемпотентность и дедупликация

Одно событие дважды доставили — нормальная ситуация при ack/restart. Обработчик — как кнопка лифта: нажали два раза — едете один раз.

  • idempotency_key в сообщении + уникальный индекс в БД
  • «Уже обработано — пропускаю» по event_id
  • Запись в БД + публикация в Rabbit → классическая ловушка двойной операции
  • Продвинутое решение: transactional outbox

25. Порядок сообщений — иллюзии и реальность

  • Один воркер, одна очередь → порядок обычно как в очереди
  • Три воркера из одной очереди → порядок обработки не гарантирован
  • Нужен порядок по user_id → отдельная очередь на ключ или один consumer на ключ

Для чатов достаточно порядка внутри комнаты при одном writer'е в БД или одном partition'е. Глобальный порядок — очень дорого; спросите, нужен ли он на самом деле.

26. Prefetch (QoS) и обратное давление

prefetch_count = N — брокер не шлёт воркеру больше N неподтверждённых сообщений. Иначе один жадный воркер заберёт пачку, а два других простоят.

ПроблемаСледствие
Слишком большой prefetchОдин воркер перегружен, остальные простаивают
Слишком маленькийРост latency на round-trip за каждым сообщением

Настраивать после профилирования времени обработки одного сообщения.

27. Наблюдаемость: что метрить

  • Глубина очереди: ready, unacked, возраст head-of-line
  • Rate: publish/s, consume/s, ack/s
  • Корреляция: trace_id / event_id в заголовках или теле
  • Management UI :15672 — первый бесплатный датчик
  • Prometheus plugin — экспорт метрик в Grafana

Очередь растёт → воркеры не успевают или упали.
unacked растёт → воркеры зависли или не шлют ack.

29. Анти-паттерн: одна «универсальная» очередь

Смешали лёгкие уведомления и тяжёлые видео-джобы. Очередь забита тяжестью — лёгкие стоят. Как одна касса на магазин и опт: все в одной змейке, раздражение гарантировано.

  • Разные SLA и скорости → head-of-line blocking
  • Сложно масштабировать избирательно один тип нагрузки
  • Решение: разнести по очередям, мониторить каждую отдельно

30. Анти-паттерн: fanout в одну общую очередь

Хотели «всем разослать», а сделали одну очередь на троих → конкуренция. Сообщение одно — заберёт кто-то один. Остальные два не увидят эту копию.

  • Подписчики конкурируют — каждый получает другую подвыборку потока
  • Для широковещания нужны отдельные очереди на каждого (см. слайд 14)
  • Типичная ошибка студенческих схем — проверяйте себя на экзамене

31. Безопасность и границы доверия

  • Доступ к брокеру: отдельные учётки по сервисам, vhost на команду/среду
  • TLS в проде — трафик между сервисами и брокером шифруется
  • В сообщениях не гоняйте пароли и токены открытым текстом
  • PII (персональные данные) — политика хранения и логирования, 152-ФЗ в РФ

Брокер общий для сервисов — инженерная гигиена: каждый сервис видит только свои очереди в своём vhost. Не «admin/admin» на весь кластер.

32. Чеклист проектирования потока на RabbitMQ

  3. Сколько воркеров, нужен ли порядок по ключу?
