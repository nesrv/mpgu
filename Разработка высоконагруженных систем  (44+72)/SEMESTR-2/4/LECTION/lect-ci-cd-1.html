CI/CD: Методология Непрерывной Интеграции и Доставки

Введение

Многие разработчики слышат про CI/CD как про что-то обязательное, но не понимают, как это работает на практике. В лекции простым языком — как устроены пайплайны, зачем автоматизация и почему без CI/CD современная разработка превращается в хаос.

Что такое CI/CD и какие проблемы он решает?

CI/CD — автоматизация цепочки от коммита до работающего кода на сервере.

  • Страх деплоя → «деплоим 10 раз в день и не боимся»
  • Человеческие ошибки — забыл миграцию, не обновил конфиг — с CI/CD такого не будет
  • Всегда работает — если тесты прошли в пайплайне, код рабочий

  • Поймёте пайплайны, стейджи, джобы
  • Напишете конфигурации для GitLab CI и Jenkins
  • Разберётесь с Docker в контексте CI/CD
  • Узнаете инструменты российских компаний
  • Получите базу для Kubernetes, ArgoCD, мониторинга

Как работает современная разработка?

В реальных проектах при команде больше 3–5 человек всё сложнее: планирование, разработка, тестирование, развёртывание происходят одновременно и непрерывно.

Рост и масштабирование

Команда 10 разработчиков, тестировщики, менеджер, микросервисы, БД — простая схема превращается в сложную систему. Релизы должны быть частыми, качество — высоким.

Ключевые характеристики современного подхода

  • Командная работа — десятки людей, синхронизация
  • Частые релизы — обновления несколько раз в день
  • Высокие требования к качеству — пользователи не прощают багов

Типичный день в жизни кода

День 1: Планирование
├─ Менеджер создаёт задачу в Jira
├─ Команда обсуждает
└─ Разработчик берёт задачу в работу

День 2-3: Разработка
├─ Ветка в Git, код, локальные тесты, коммиты

День 4: Код-ревью
├─ Pull Request, проверка коллег, подтверждение 2+

День 5: Тестирование
├─ Мердж в develop, автотесты, ручное тестирование

День 6-7: Подготовка к релизу
├─ Исправление багов, мердж в release, стейджинг

День 8: Релиз
├─ Деплой на продакшн, мониторинг, откат или успех

Восемь дней на одну фичу — и это оптимистичный сценарий.

Инструменты современной разработки

GitHub, GitLab, Bitbucket
Jira, Trello
Microsoft Teams, Mattermost, Telegram

Git — стандарт контроля версий. Таск-трекеры организуют задачи. Системы коммуникации — уведомления о коммитах, тестах, деплоях. Плюс инструменты тестирования под каждую платформу.

Реальный пример процесса разработки

Средняя продуктовая компания, маркетплейс, 30 человек. Дейлики, демо раз в две недели. Проблемы при релизе: фронт/бэкенд/мобилка не синхронизированы, баги, разные версии на стейджинге и проде. Координация — главная проблема; сложность растёт экспоненциально с размером команды.

Проблемы координации и деплоя

Тестирование вручную — дни. Деплой — лотерея. Попытки без автоматизации: жёсткие код-ревью, фризы, чек-листы на 50 пунктов — костыли, люди ошибаются, процессы замедляются.

Начало автоматизации

Что если сервер сам будет:
- Проверять код при каждом коммите?
- Запускать тесты?
- Деплоить на тестовый сервер?
- Откатывать изменения при проблемах?

Это и есть CI/CD — подход, который решает большинство описанных проблем. Автоматизация требует времени на настройку, зато релизы становятся предсказуемыми и безопасными.

Ручной vs автоматический процесс

РУЧНОЙ ПРОЦЕСС                    АВТОМАТИЧЕСКИЙ
14:00  Чек-лист 30 пунктов         14:00  git push → пайплайн
14:15  SSH, забыл пароль           14:01  ✓ Тесты прошли
14:25  git pull — конфликт!         14:03  ✓ Деплой на staging
14:40  npm install — нет места      14:06  ✓ Деплой в продакшн
...
18:30  Починил. Уже поздно         14:07  ✓ Healthcheck OK

Цена ошибки в ручном процессе — человеческий фактор; найти причину сложно.

Проблема масштабирования

Время на ручной деплой:
├─ 1 сервер:     30 минут
├─ 5 серверов:   2.5 часа
├─ 20 серверов:  10 часов
└─ 100 серверов: Невозможно

Время на ручное тестирование:
├─ 10 тест-кейсов:   1 час
├─ 200 тест-кейсов:  3 дня
└─ 1000:             Никто не будет делать

Команды срезают углы — тестируют только критичное, откладывают обновления → риск катастрофы.

Реальная история из российской практики

E-commerce, 20 человек. Релизы по четвергам, целый день:
Утром — 200 тест-кейсов, 3 тестировщика
Обед — 5-7 багов, разработчики чинят
15:00 — повторное тестирование
17:00 — деплой 8 серверов по чек-листу
На 5-м сервере другая версия PHP — откат
20:00 — не проверили мобильное приложение, API изменился
22:00 — релиз закончен, все измотаны

Через месяц — деплой не той ветки, откат 3 часа, потери для бизнеса. Ручные процессы создают стресс и «героическую» культуру.

Решение: Continuous Integration

Continuous Integration (CI) — практика, при которой
разработчики регулярно интегрируют код в общий репозиторий,
и каждая интеграция автоматически проверяется сборкой и тестами.

«Continuous» = непрерывный, «Integration» = объединение кода. Ключевая идея: находить проблемы как можно раньше. Баг через 5 минут — исправление за минуты; баг через месяц в проде — дни.

Минимальный пример CI (GitLab)

test:
  script:
    - npm install
    - npm test

При каждом пуше GitLab запускает эти команды. Упали тесты — сразу видно.

Как работает CI: пошаговый разбор

Цикл обычно 5–15 минут. Уведомление о результате; при падении — какой тест, на какой строке.

Основные компоненты CI

1. Триггеры — когда запускаться (пуш, MR и т.д.)
2. Окружение выполнения — сервер, Docker, раннер (чистое, воспроизводимое)
3. Набор проверок — тесты, метрики, стандарты кода
4. Оповещения — чат, почта, чтобы команда реагировала

Принципы эффективного CI

1. Частая интеграция — пушить хотя бы раз в день
2. Автоматизация всего — сборка, тесты, линт, документация
3. Быстрая обратная связь — пайплайн 5–10 минут идеально

Чем дольше код в изолированной ветке, тем сложнее мердж. Если ждать час — контекст потерян.

CI меняет культуру разработки

До CI:                     После CI:
"Работает у меня"    →     "Проходят все тесты"
"Потом починим"      →     "Чиним сразу"
"Кто-то сломал"      →     "CI показывает кто и когда"
"Не знаю почему упало" →   "Вот лог с ошибкой"
"Боюсь мержить"      →     "CI проверил, всё ок"

«Зелёная сборка» = можно деплоить. Красная = сначала починить.

Метрики эффективности CI

Время пайплайна — 5–10 мин хорошо, 30+ тормозит
Частота запусков — десятки раз в день = норма
Процент успешных сборок — 85–95% идеал
  (100% — возможно, тесты слишком мягкие)

CI в российских реалиях

Санкции — GitHub Actions может быть недоступен, Travis — карты.
Альтернативы: self-hosted GitLab, Jenkins, TeamCity (JetBrains).

В небольших компаниях — не хватает инфраструктуры.
Контейнеризация помогает запускать CI на том же железе.

Тренд: все двигаются к автоматизации, скорость — ключевой фактор.

Путь к зрелому CI

Уровень 0: Нет CI — всё вручную
Уровень 1: Сборка при пуше, уведомления
Уровень 2: Unit-тесты, проверка стиля
Уровень 3: Интеграционные тесты, безопасность, покрытие
Уровень 4: Автодеплой на тестовые среды, параллельность
Уровень 5: Кеширование, выборочные тесты, ML

Большинство на 2–3. CI — про культуру постоянной интеграции, не только про инструменты.

Что такое Continuous Delivery (CD)?

Continuous Delivery (CD) — код автоматически
подготавливается к релизу после всех проверок.
Ключевое: подготавливается. Решение о релизе — человек.

CI: «работает ли код?» CD: «готов ли к релизу?» Цель CD — сделать релизы скучными: без стресса, одним кликом.

Минимальный пример CD (GitLab)

stages: [ test, build, deploy ]
test:   stage: test; script: [ npm test ]
build:  stage: build; script: [ npm run build, docker build..., docker push... ]
deploy_staging:  stage: deploy; script: [ ssh ... docker pull/run ]; environment: staging
deploy_production: stage: deploy; when: manual  # Ручное подтверждение!

После авто-стейджей код готов к прод-релизу, кнопку нажимает человек.

Как CD расширяет CI

CI: клонирование, зависимости, линтинг, unit/integration тесты, безопасность
       ↓ все проверки прошли
CD: сборка, Docker-образ, реестр, деплой dev → тесты → staging → тесты
    [MANUAL] Деплой на продакшн

CI — качество кода. CD — подготовка к релизу. Вместе — быстрая и безопасная доставка.

Артефакты — основа CD

Артефакт = готовый к деплою пакет (Docker, JAR, ZIP...)
Принцип "Build once, deploy many":
  Один артефакт (версия по commit SHA) → Dev, Test, Staging, Prod
  Тот же артефакт везде = гарантия идентичности

Никакой пересборки для продакшна — в прод попадает то, что тестировали на стейджинге.

Окружения в CD

Development — нестабильное, каждый коммит
Testing/QA — для тестировщиков, стабильнее
Staging — копия продакшна (железо, данные, настройки)
Production — боевое, только проверенный код

Стратегии деплоя

Blue-Green: два окружения, переключение трафика 100% с Blue на Green
Canary: 5% → 25% → 50% → 100% трафика, метрики под контролем
Rolling: обновление реплик по одной, приложение доступно

Автоматизация релиз-процессов

Заметки релиза — из коммитов и PR автоматически
Версионирование — Semantic Versioning по типам коммитов
Конфигурация — переменные, секреты в пайплайне, без ручного правления на сервере

Метрики и мониторинг в CD

После деплоя — интеграция с мониторингом.
Если метрики ухудшаются (ошибки, производительность) —
автоматический откат (automated rollback).

Delivery vs Deployment

Continuous Delivery — деплой в прод с ручным подтверждением
Continuous Deployment — деплой в прод полностью автоматический
  (все проверки прошли → код сам в проде)

Delivery подходит большинству. Deployment — высокая зрелость процессов.

Что такое Continuous Deployment?

Continuous Deployment — каждое изменение, прошедшее
все автопроверки, автоматически попадает в продакшн
без ручного вмешательства. Push → тесты → 15–30 мин → у пользователей.

В Delivery решение принимает человек, в Deployment — машина по результатам тестов.

Пример Continuous Deployment

stages: [ test, build, deploy ]
test: unit + integration + e2e
build: docker build & push с $CI_COMMIT_SHA
deploy_production: kubectl set image... ; check_deployment_health.sh
  only: [ main ]   # Без when: manual!

Никаких кнопок — код сам доезжает до продакшна при зелёных проверках.

Полный цикл Continuous Deployment

09:30  push → 09:31 unit → 09:33 integration → 09:36 e2e
→ 09:41 образ → 09:43 деплой 1% → мониторинг
→ 09:50 10% → 10:00 50% → 10:10 100%
Итого: ~40 минут от пуша до продакшна у всех

Amazon — деплой каждые 11 сек, Netflix — каждые 16 мин.

Предпосылки для Continuous Deployment

1. Культура качества — код может попасть в прод через полчаса
2. Покрытие тестами — unit, integration, e2e, контракты, производительность
3. Мониторинг и observability — метрики, логи, трейсы

Progressive Rollout

Canary: 1–2% трафика → мониторинг 10–15 мин
Early Adopters: 5–10%
Partial: 25–50% → A/B старая/новая версия
Full Rollout: 100%
На каждом шаге — откат при проблемах

Feature Flags

// Код в проде, но выключен; включение — конфиг без деплоя
if (featureFlags.isEnabled('new-payment-system'))
  return newPaymentProcessor.process(order);
else
  return legacyPaymentProcessor.process(order);

// Конфиг: percentage, users (beta-testers)

Инфраструктура для CD

Kubernetes — rolling updates, перезапуск подов
Service Mesh (Istio) — маршрутизация трафика (1% на новую версию), метрики
Prometheus + Grafana — мониторинг в реальном времени

Когда НЕ нужен Continuous Deployment

Регулируемые индустрии (банки, медицина) — ручной контроль, комплаенс
B2B с крупными клиентами — клиенты хотят знать об изменениях заранее
Embedded и десктоп — обновления скачивает пользователь; IoT может быть офлайн

Continuous Deployment в России

Яндекс, VK, Авито используют для части сервисов. Большинство — на Continuous Delivery. Препятствия — менталитет («а вдруг сломается?»), недостаток опыта. Молодые стартапы строят автоматизацию сразу; крупные компании пробуют CD на внутренних сервисах.

Преимущества CI/CD для команды и бизнеса

Время обнаружения проблемы:
                Без CI/CD    С CI/CD
Синтаксическая ошибка  2 ч      30 сек
Сломанный тест        1 день    2 мин
Проблема интеграции   1 нед    10 мин
Баг в продакшне       2 нед    30 мин
Стоимость исправления растёт со временем!

Зелёный пайплайн = уверенность в коде. Регрессию делают автотесты; тестировщики занимаются исследовательским тестированием и проектированием тест-кейсов.
