Слайд 0a. Глоссарий: слова, которые встретятся в лекции

  • Дальше будут аббревиатуры и термины. Ниже — одним экраном, «что имели в виду»; к документации и прошлым лекциям по k8s можно вернуться позже.
Слово Как говорят / сленг Простыми словами
SPOF Эс-пи-о-эф; в чатах часто пишут «сингл поинт оф фейл», говорят «узкое горлышко», «одна точка — и всё легло». Одна точка, от которой всё рухнуло: один сервер, один человек с доступом. Пример: одна VPS — если она легла, всё легло.
HPA Эйч-пи-эй (или коротко «аш-па»); сленг: «автоскейл реплик», «подкрутить реплики по CPU/очереди». Автоматически добавить или убрать копии приложения (реплики) по нагрузке. Не делает из маленького сервера большой по памяти.
PVC Пи-ви-си; сленг: «заявка на том», «перзистент» (от *persistent volume*), «диск из стореджа». Заявка приложению: «дай постоянный диск такого размера». Сами данные на диске не спасаются волшебством — нужны бэкапы.
Helm chart Хелм + чарт (как *chart*); сленг: «пакет ямлов», «шаблоны с values», «поставить приложение одной командой». Пакет шаблонов для Kubernetes: подставили значения — получили готовые YAML. Удобно ставить одно и то же приложение в разные среды.
Kustomize overlay Кастомайз (от названия инструмента); оверлей — почти всегда так и говорят; сленг: «база + патчи под среду», «без копипасты всех файлов». Слой правок поверх общей папки с манифестами: для dev одни цифры, для prod другие — без дублирования всех файлов.
IAM Ай-ай-эм; сленг: «права в облаке», «кто куда лезет», «сервисный аккаунт + роли» (в духе AWS/GCP). В облаке: кто (человек или сервис) что может. Pod в кластере может работать от роли облака, чтобы не хранить долгоживущий ключ в Secret.
LB (*load balancer*) Эл-би или «лоад-балансер»; сленг: «балансир», «внешка перед кластером» (vs Ingress — «роутинг внутри по HTTP»). Снаружи кластера: принять трафик и разделить по нескольким целям. Ingress — уже внутри, по HTTP(S) направляет на нужный сервис по адресу и пути.

Слайд 0b. DevOps и GitOps: как это связано

  • DevOps — широко: культура и практики (dev + ops вместе): автоматизация, CI/CD, мониторинг, общая ответственность за продукт, быстрая обратная связь. Это не один инструмент и не обязательно Kubernetes.
  • GitOps — узко и конкретно: как держать желаемое состояние кластера/приложений в Git и подтягивать его агентом (pull), чтобы прод совпадал с репозиторием (ревью, откат, аудит — как у кода).
  • Связь: GitOps — один из способов реализовать идеи DevOps вокруг доставки и конфигурации в cloud-native мире; DevOps без GitOps — норма (например, push из CI: kubectl apply из пайплайна, Ansible, другие модели).
  • Запомнить на собесе: DevOps отвечает на «как мы вообще работаем и автоматизируем»; GitOps — на «где истина для кластера и кто её применяет».
DevOps GitOps
Уровень Подход, процессы, культура Паттерн управления состоянием (часто k8s)
Типичный фокус Скорость, качество, надёжность цикла разработки Git = источник правды, синхронизация с кластером
Обязательно друг друга? GitOps не заменяет весь DevOps Можно не делать GitOps и всё равно «в DevOps»

Слайд 1. С чего боль всего в проде

  • Дежурный ночью чинит прод в обход Git: kubectl edit или правка YAML на сервере. В Git одна версия манифестов, в кластере уже другая — расхождение никто не закоммитил.
  • Утром в корпоративном чате (в компаниях часто Slack, Microsoft Teams, Mattermost — канал, куда пишут дежурным) нельзя дать привычный ответ: «вот merge request, вот diff — построчное сравнение „было / стало“ в Git или в интерфейсе ревью, вот как откатить». Изменения не в истории репозитория → их не воспроизвести и не аудировать. Сервис может быть жив — но сломан процесс учёта; это инцидент процесса (не обязательно даунтайм).
  • Configuration drift: на стенде один replicas, на проде другой; в Wiki третья версия — классика технического долга операционки.
  • Новый разработчик спрашивает: «Где истина?» Если ответ «спроси Васю» — нет инженерной системы, есть героизм.
  • Связка: GitOps делает истину адресуемой — репозиторий, а не голова и не чат.

Слайд 2. Git как источник истины (single source of truth)

  • Желаемое состояние — не «как сейчас в голове у дежурного», а то, что записано в репозитории: манифесты Kubernetes (какие Deployment, сколько реплик, какие Service/лимиты и т.д.). После apply/sync кластер должен к этому состоянию сойтись; Git — каноническая копия этого описания.
  • Такие описания в Git кладут в разных форматах: «голый» YAML — обычные файлы .yaml, как во вводной лекции по k8s, без шаблонов; Helm — чарт (шаблоны + values.yaml, на выходе всё равно манифесты); Kustomize — база + оверлеи (патчи под dev/prod без копирования всего дерева файлов). Суть одна: источник правды — коммиты в Git.
  • Изменение = MR/PR: ревью, обсуждение, blame по строкам — привычный процесс из разработки ПО.
  • Откат релиза инфраструктуры: git revert или откат тега — предсказуемее, чем «Вася помнит, что откатить».
  • Оговорка: Git хранит желаемое; секреты в открытом виде туда не кладут — отдельные механизмы (см. слайд про секреты).

Слайд 3. Агент синхронизации (сердце GitOps)

  Git (ветка / тег)     агент в кластере      Kubernetes API
        |                      |                    |
        +---- clone / watch ---+---- apply/sync ----+
  • Агент (Argo CD, Flux и др.) работает внутри кластера (или с доверенным доступом) и непрерывно или по событию подтягивает манифесты из Git.
  • Нормальный поток: вы мержите в ветку → агент видит новый коммит → синхронизирует объекты Kubernetes.
  • Культура: «я не накатываю с ноута» — я меняю Git, кластер следует заявленному состоянию.
  • Технически агент использует тот же Kubernetes API, что и kubectl; меняется кто инициирует и откуда берётся намерение.

Слайд 4. Дрифт: когда «в кластере» и «в Git» разошлись

  • Ручной kubectl scale или смена образа: факт кластера разошёлся с Git — классический дрифт.
  • После sync GitOps либо подтянет кластер к Git (перезапишет ручные правки), либо только покажет рассинхрон (OutOfSync) и будет ждать человека — сценарий задают настройки агента и договорённости команды (например, на проде запретить авто-исправление без явного «ОК»).
  • В Argo CD часто настраивают: prune — при синхронизации удалять из кластера объекты, которых уже нет в Git (чтобы не копились «осиротевшие» ресурсы); selfHeal — при дрифте сам снова применить манифесты из Git (откатить ручной kubectl scale и т.п.). В Flux те же идеи — другими полями в YAML, смысл похожий.
  • Hotfix ночью допустим, но завтра тот же фикс должен попасть в Git, иначе снова «истина только в проде».
  • Метрика зрелости: за минуту ответить, совпадает ли прод с конкретным коммитом в репо манифестов.

Слайд 5. GitOps — это не то же самое, что «у нас есть CI»

  • CI отвечает за качество кода и сборку образа: тесты, линтеры, docker build, push в registry.
  • GitOps отвечает за согласование кластера с репозиторием манифестов — другой контур ответственности.
  • Типичная связка: CI собрал образ myapp:<git-sha>, затем либо обновила версию в Helm values и закоммитила, либо триггернула процесс → агент подтянул изменения.
  • Если в конце только «CI делает kubectl apply», это push-модель CD, не обязательно GitOps в строгом смысле (см. слайд про pull/push).

Слайд 6. Pull-модель vs push-модель

  • Push CD: пайплайн имеет kubeconfig/токен и толкает изменения в API. Проще начать; секреты CI — критичное звено; компрометация job — риск.
  • Pull GitOps: кластер сам забирает желаемое; CI не обязан держать ключи к прод-кластеру, если роли разделены.
  • Чеклист GitOps: (1) Git = истина; (2) агент в кластере; (3) видимость дрифта; (4) секреты не plaintext в публичном репо; (5) прод отделён от dev политикой, не только «договорились».

Слайд 7. Зачем столько имён: контейнер ≠ оркестратор

  • Контейнер (упрощённо): процесс в ограниченном окружении с образом ФС — удобно переносить между средами.
  • Оркестрация: планирование на ноды, перезапуск при падениях, service discovery, rolling-обновления, масштаб — то, что один Docker на одной машине не обещает.
  • Типичная лестница: локально Podman/Docker → стенд Compose → прод малый k3s → организационный k8s.
  • Фраза для собеседования: «Понимаю, на каком уровне абстракции решается задача: упаковка, одна машина или кластер».

Слайд 8. Podman — контейнеры без обязательного демона

  • Без обязательного dockerd (daemonless), удобный rootless; команды близки к Docker — образы те же (OCI).
  • Для курса — локалка и сборка, как Docker; оркестратором кластера Podman реже (детали — слайд 8a).

Слайд 8a. Podman: происхождение, архитектура, отличия от Docker

  • Откуда взялся: развитие экосистемы Red Hat вокруг OCI-образов — проект libpod (ныне ядро Podman), рядом в том же мире Buildah (сборка образов), Skopeo (копирование образов между registry без демона). Идея — дать инструменты без обязательного центрального демона и с упором на rootless и политики Enterprise Linux; название Podman отражает поддержку pod’ов в духе Kubernetes (несколько контейнеров с общими namespace).
  • Архитектура (упрощённо): CLI podman обращается к libpod; при запуске контейнера нет единого «вечного» dockerd — процессы стартуют через conmon + runc или crun (OCI runtime). Образы и слои — совместимы с Docker Hub / обычным registry (формат OCI).
  Docker (классика):  docker CLI → dockerd → containerd → runc
  Podman:             podman CLI → libpod → conmon → runc / crun
                      (нет одного обязательного демона на все run)
  • Отличия от Docker: нет обязательного демона; удобный rootless; на RHEL/Fedora часто по умолчанию; Dockerfile и podman build обычно совместимы с привычным workflow; сеть/тома могут отличаться в деталях — при переносе скриптов читают доки. Docker Compose ↔ podman compose / podman kube play — разные обёртки, смысл похожий.
  • Где используется: рабочие станции и CI в Red Hat / OpenShift-стеке; среды, где важны rootless и аудит без root-демона; обучение и поставки, завязанные на RPM-стек. В «голом» Kubernetes на нодах обычно containerd/CRI-O, а не Podman как runtime — но образы те же.
  • Перспективы: доля растёт в корпоративном Linux-мире и в Podman Desktop; Docker по-прежнему де-факто стандарт у разработчиков и в туториалах. Для карьеры полезно уметь оба CLI и понимать, что контейнер = OCI, а «докер» vs «подман» — в основном способ запуска и политика безопасности, а не другой формат образа.

Слайд 9. Docker — эталон «образ + run»

  • docker build → образ; docker run → контейнер. Под капотом цепочка: dockerd → containerd → runc (образы и запуск делегируются containerd, не «один монолит»).
  • В Kubernetes на ноде kubelet говорит с runtime по CRI; чаще это containerd или CRI-O, а не docker CLI. Имя containerd в логах — норма.
  • На пуле машин без оркестратора сами решаете, как пережить падение хоста; дальше — Compose на одной машине.

Слайд 10. Docker Compose — оркестрация «одной команды»

  • Один compose.yaml — сервисы, сети, тома на одной машине; docker compose up быстро поднимает стенд.
  • Не заменяет k8s, если нужны много нод, зоны HA и единые политики на весь кластер: в k8s это RBAC (кто что может в API) и Quota (потолки CPU/RAM/Pod на namespace) — через API принудительно для всех; в Compose — по сути один хост и лимиты вручную.
  • Compose на VPS — нормально; k8s на одной машине «для галочки» — часто лишняя боль.

Слайд 11. HashiCorp Nomad — другой вид оркестратора

  • Планировщик задач на агентов; Docker — один из драйверов, бывают и другие типы workload.
  • Часто проще вход для части команд, чем полный zoo k8s; другая модель сети и сервисов.
  • В резюме/собесе: «Nomad — альтернативный оркестратор, не путаю с Compose».
  • Если стек Nomad-only, знания k8s концептуально переносимы: желаемое состояние, планирование, отказоустойчивость.

Слайд 11a. Nomad подробнее: где, зачем, когда

  • Nomad — планировщик job’ов на пул машин (серверы + агенты); не «k8s под другим именем»; часто рядом Consul / Vault. Драйверы: не только Docker, но и exec, Java и др. — можно мешать контейнеры и legacy без полной упаковки в образ.
  • Где: стек HashiCorp, edge, команды, которым тяжёлый вход в k8s; доля на рынке меньше, чем у k8s, но имя в резюме встречается. Когда k8s: нужен стандарт индустрии, Helm, managed в облаке.
  • Лицензии HashiCorp (в т.ч. BSL) — сверять с актуальными условиями перед продом.

Слайд 12. k3s — Kubernetes облегчённый

  • Один бинарь, вшитые компоненты, меньше RAM/CPU — влезает на маленькую ВМ.
  • Совместимость с Kubernetes API: те же Deployment/Service/Ingress — навыки переносятся на managed k8s.
  • Типичные кейсы: внутренние панели, боты, небольшие API, edge, homelab.
  • Сравнение с «полным» k8s — не «хуже», а другой компромисс цены и операционки.

Слайд 13. «Полный» Kubernetes (k8s)

  • Ожидается масштаб: выделенный control plane, worker-пулы, CNI/CSI, RBAC, admission, интеграции с облаком (LB, диски, IAM).
  • В проде «полный» k8s обычно приходят тремя путями: managed-сервис в облаке, свой кластер на арендованных ВМ (IaaS), bare metal (своё железо / colo) — детальный разбор на следующем слайде.
  • Фраза: «k8s — это платформа; приложение всё равно нужно проектировать».

Слайд 13a. Managed Kubernetes и «bare metal»: где граница ответственности

  • Managed (EKS, GKE, AKS, Yandex Managed Kubernetes и др.): провайдер ведёт control plane (etcd, апгрейды, HA), вы подключаете ноды; облако даёт LB, диски, IAM — детали и цены разные, идея одна.
  • На вас почти всегда: манифесты, образы, Ingress, лимиты, данные и бэкапы — провайдер не «пишет» ваш сервис и не отвечает за безопасность вашего YAML.
  • Bare metal — k8s на своём железе/colo: полный контроль и часто регуляторика; всё сами: сеть, хранилище, HA мастеров, etcd, патчи. Свой k8s на ВМ (kubeadm, kubespray…) — без строки managed CP, но с ночными рисками на мастерах.
  • Грубо: нет платформенной команды → managed; жёсткая привязка к своей площадке → bare metal / private cloud; учёба → kind/k3d.

Слайд 14. Сравнение: одна машина → кластер

  Compose              Nomad / k3s / k8s
     |                        |
  одна ВМ              несколько нод / HA
  docker compose       kubectl / API
  «поднял стенд»       «задекларировал кластеру»
  • Слева — быстро и дёшево в операционке; справа — цена входа выше, потолок по масштабу и стандартизации выше.
  • YAML в обоих мирах, но семантика разная: Compose — docker-сервисы; Kubernetes — объекты API с контроллерами.
  • Переход не обязан быть линейным по времени: можно долго жить на Compose и один день перейти на k3s при росте требований.

Слайд 15. Таблица: чем дышит каждый инструмент

Инструмент Типичная цель Масштаб «Прод» часто
Podman локально, CI, rootless 1 хост реже как оркестратор
Docker образы, run 1 хост да (с обвязкой)
Compose много контейнеров 1 хост (обычно) стенды, малые сайты

k3s совместимый k8s легче от 1 ноды да, VPS/edge

  • Для резюме: «Docker + Compose» и отдельно «k3s/kind, деплой через kubectl/Helm» — разный вес на рынке.

Слайд 16. Что выбрать для пет-проекта

  • Веб + БД на одной VPS: Compose — минимум трения; k3s — если нужно потрогать Ingress, Deployment, HPA с боевым API.
  • Для строки «Kubernetes»: лучше k3s/k3d/kind + пара сервисов + Ingress, чем «поставил minikube и забыл».
  • Полный k8s на одной машине без задачи — часто учебная боль без выхлопа; лучше маленький реальный сценарий на k3s.
  • Для размышления: ваш пет-проект — где он на оси «одна машина / кластер» и что мешает сдвинуться вправо?

Слайд 17. Почему Kubernetes «дружит» с GitOps

  • В Kubernetes вы не «крутите ручки в панели навсегда», а говорите: «хочу три таких пода, такой сервис, такой лимит памяти». Это описание — обычный текст (YAML). Текст удобно хранить в Git: как лабораторная или код — коммит, ветка, откат, «кто последний менял строку».
  • Зачем Git, если есть API кластера? API — для машин и для kubectl; Git — для людей: привычный merge request, коллега прочитал изменения, обсудили, потом смержили. То же самое дисциплина, что с исходниками приложения, только предмет — как должен выглядеть кластер.
  • GitOps в этой картинке — не новая магия, а связка: «желаемое состояние лежит в репозитории» + «агент в кластере подтягивает его и приводит кластер в соответствие». Kubernetes уже умеет применять YAML; GitOps добавляет привычный рабочий процесс вокруг этого YAML.
  • Плохой компромисс (для понимания): выкладывать конфиги только в отдельное хранилище Helm-чартов, минуя привычные ветки и ревью в Git — лучше, чем хаос в чате, но слабее, чем полноценный процесс с merge request и историей, как у кода.

Слайд 18. Argo CD — роли компонентов (обзор)

  • Argo CD — программа внутри кластера, которая смотрит на Git и говорит API Kubernetes: «сделай как в репозитории». Ниже — три привычных куска (имена можно не зубрить, важно смысл).
  • Application (в терминах Argo) — одна запись-настройка: «возьми этот репозиторий, эту папку или ветку, эту ревизию (коммит/тег) и применяй в этот кластер / этот namespace». То есть одна карточка = один связанный деплой.
  • Repo Server — как кладовщик: скачивает Git (и держит кэш), чтобы всегда можно было сравнить: «что в репо» vs «что уже в кластере».
  • Application Controller — «мозг»: смотрит на разницу (дрифт), по правилам решает, подтянуть ли кластер к Git (sync) или только показать проблему. В веб-интерфейсе это видно как «не совпадает» (OutOfSync), diff и история — удобно не только «админам кластера», но и разработчикам и менеджерам: видно, что выкатилось.

Слайд 19. Flux — семейство контроллеров

  • Flux — тоже GitOps-агент, но устроен набором маленьких контроллеров в Kubernetes, каждый делает свой шаг. Опять же: имена подмножеств можно не заучивать, важно цепочка.
  • Сначала контроллеры Source (в манифестах это типа GitRepository, HelmRepository…) — подтянуть код или список Helm-чартов извне, «как git pull / указатель на репозиторий», но автоматически и по расписанию или событию.
  • Потом контроллеры Kustomize и Helm — собрать из этого готовые YAML (накладывают Kustomize-патчи или рендерят Helm-шаблоны) и положить в кластер так, как задумано в Flux.
  • Идея Flux в одном предложении: настройки GitOps — сами обычные объекты Kubernetes (свои CRD); весь пайплайн можно описать одним набором YAML «в стиле кластера». Argo CD vs Flux — часто вопрос привычки команды и лицензий/поддержки, а не «кто правильнее»; на экзамене достаточно понять роль агента и отличие pull-модели от «CI жмёт kubectl».

Слайд 20. Namespace и несколько сред (dev/stage/prod)

  • В одном кластере: namespace dev, stage, prod + квоты и RBAC.
  • В разных кластерах: выше изоляция и цена операционки.
  • GitOps: отдельные Application на ветку/путь или отдельные репозитории; важно запретить случайный cross-sync.
  • Практика: прод-манифесты в отдельной ветке или репо с жёстким ревью.

Слайд 21. Паттерн: образ из CI, версия в Git

  • CI публикует образ myapp:<git-sha> в registry.
  • Версия в манифестах обновляется коммитом (вручную, скриптом CI, ботом вроде Renovate).
  • При sync Kubernetes тянет новый образ, если imagePullPolicy и кэш ноды позволяют (типичный вопрос: «почему не обновилось»).
  • Правило: прод всегда на конкретном теге, не на «плавающем» latest.

Слайд 22. CI: build, test, artifact

  • Линт и тесты до сборки образа экономят время и стоимость registry.
  • Артефакт — immutable образ; тег = ссылка на конкретную сборку.
  • Популярные registry: GHCR, GitLab, Docker Hub, облачные ECR/GCR/YC.
  • SBOM / сканирование уязвимостей — следующий слой зрелости (одна фраза в резюме уже плюс).

Слайд 23. CD без GitOps: push из пайплайна

  • Job получает kubeconfig или токен и выполняет kubectl apply / helm upgrade.
  • Плюс: быстрый старт, меньше компонентов в кластере.
  • Минус: каждый раннер/репо с ключом — поверхность атаки; сложнее ответить «кто менял прод в 3:12».
  • Гибрид: CI обновляет Git, дальше GitOps — часто золотая середина.

Слайд 24. CD с GitOps: promotion

  • Dev-ветка смержена → агент обновил dev-кластер автоматически.
  • Prod — отдельный MR или release tag с апрувом владельца сервиса.
  • Снижается риск класса ошибок «случайно apply с prod credentials на dev кластер» (другие ошибки всё ещё возможны).
  • Культура: релиз виден в истории Git — артефакт для аудита.

Слайд 25. Секреты: не класть пароли в открытый Git

  • Sealed Secrets: в Git — зашифрованный манифест, расшифровка только в целевом кластере.
  • External Secrets Operator: в Git ссылки, значения приезжают из Vault, AWS Secrets Manager и т.д.
  • SOPS + age/PGP — распространённый вариант в командах.
  • На курсовом: не коммитить реальные пароли даже в приватный репо «на время».

Слайд 26. Антипаттерн: один человек с kubeconfig

  • Если только один человек знает, как откатить прод — SPOF по людям.
  • GitOps: права на merge в прод-репо = контролируемый круг лиц.
  • RBAC в кластере ограничивает, кто может что менять через API даже при утечке токена с узким scope.
  • Цель: дежурный может следовать runbook, а не «звонить единственному знающему».

Слайд 27. k3s на одной VPS: зачем

  • Одна VPS — дёшево; реальный цикл: образ → Deployment → Service → Ingress.
  • Подходит внутренним сервисам, лендингам, ботам, где SLA не контрактный 99.99%.
  • Рост: упёрлись в машину — больше железа, managed k8s или вынести часть в serverless.
  • Честность: одна VPS с k3s не заменяет HA-кластер в банке.

Слайд 28. Установка k3s (идея, не копипаста на экзамене)

  • Обычно — официальный скрипт или пакет дистрибутива; в методичке курса могут быть свои пины версий.
  • kubeconfig с сервера копируют себе безопасно (scp, ssh), настраивают контекст.
  • Ingress-контроллер — отдельное решение или встроенное; сверяться с документацией вашей сборки k3s.
  • Безопасность: API не вывешивать в интернет без TLS и ACL/VPN.

Слайд 29. kubeconfig и доступ

  • Файл ~/.kube/config: секции clusters, users, contexts; текущий контекст — поле current-context.
  • Контекст = какой кластер + какими учётными данными + (иногда) namespace по умолчанию.
  • Ротация сертификатов/токенов — операционная рутина; при «внезапно отвалилось» смотрят срок и правильный контекст.
  • Связка с GitOps: для рутины деплоя kubeconfig менее критичен, если всё через Git; для дебага — ограниченный доступ всё равно нужен.

Слайд 30. Цепочка деплоя на VPS с GitOps

  Git (main)
      |
      v
  Argo CD / Flux  (Pod в k3s на той же VPS или отдельно)
      |
      v
  Kubernetes API  (apply / sync)
      |
      v
  Deployment / Service / Ingress
  • Агент имеет service account с правами в этом кластере — ключевая доверенная связка.
  • Разработчик не пушит напрямую в API; он пушит в Git.
  • Аудит: «какой коммит сейчас в проде» — вопрос к Git и к Argo/Flux.

Слайд 31. Если GitOps «ещё рано»: ручной apply с ноутбука

  • kubectl apply -f после git pull — рабочий способ для учебы и микрокоманд.
  • Правило гигиены: применили руками — сразу закоммитьте тот же YAML в Git, иначе дрифт.
  • GitOps добавляется, когда болит масштаб команды или аудит.
  • Переходный этап: CI кладёт артефакты, человек мержит манифесты — лучше, чем «знает только один человек».

Слайд 32. Ограничения одной VPS

  • SPOF: упал хостинг — сервис недоступен; резервирование — второй сервер или облако.
  • HPA не создаст CPU/RAM на ноде: если нет места, поды в Pending.
  • Вертикаль VPS упирается в потолок; горизонталь на одной машине ограничена.
  • План роста: БД в managed, статика в CDN, кластер в managed k8s.

Слайд 33. Бэкапы (напоминание)

  • PVC с данными приложения требуют политики снапшотов или dump БД.
  • Для k3s — официальные рекомендации по состоянию control plane; не копировать /var «на глаз» без гайда.
  • Восстановление прогнать учебным прогоном хотя бы раз — иначе это не бэкап, а надежда.
  • Кратко: правило 3-2-1 (три копии, два носителя, одна off-site).

Слайд 34. Ingress на маленьком k3s

  • Один Ingress с несколькими rules экономит порты и сертификаты.
  • Let's Encrypt через cert-manager — типичный путь; нужны корректный DNS и 80 для HTTP-01 (или DNS-01).
  • «502» — цепочка: Service endpoints → Ingress backend → логи контроллера.
  • Для учебы: начать с HTTP, затем TLS.

Слайд 35. «Полный» k8s в проде: картина

  • Несколько узлов control plane или managed control plane — отказоустойчивость «пульта».
  • Worker pools: отдельные пулы под GPU, memory-optimized, spot и т.д.
  • Продуктовые команды чаще отдают Helm chart или Kustomize overlay; платформа держит политики и кластеры.
  • Связь с VPS: там вы все роли сразу; здесь — разделение труда.

Слайд 36. Managed Kubernetes (операционный взгляд)

  • Провайдер держит control plane; в биллинге обычно ноды, диски, LB, трафик, registry — сверяйтесь с калькулятором облака.
  • Плюс managed: LB, диски и IAM для Pod’ов «из коробки»; на bare metal то же чаще собирать самим.
  • Ваш слой (манифесты, лимиты, Ingress, GitOps, образы) — см. 13a. Lock-in: API k8s переносим, облачные LB/аддоны — привязка; часто managed дешевле по людям, чем круглосуточная платформа на железе.

Слайд 37. LoadBalancer и Ingress в облаке

  • Service LoadBalancer просит облако выдать внешний адрес (если cloud controller настроен).
  • Ingress + контроллер принимает HTTP(S) и режет по host/path на внутренние сервисы.
  • Типичная цепочка: DNS → LB → Ingress controller → Service → Pod.
  • Путаница: Ingress «сам по себе» не откроет порт без контроллера и правил.

Слайд 38. Чем прод k3s на VPS отличается от облачного k8s

Аспект k3s на VPS Managed / свой большой кластер
Отказоустойчивость одна машина HA, зоны
Операционка вы сами частично провайдер
Стоимость входа низкая выше / сложнее биллинг
Обучение k8s отлично отлично
  • Один сервер прощает ошибки дешевле, но цена простоя растёт с выручкой.
  • Облако добавляет счёт за трафик, LB, диски — зато предсказуемые паттерны масштаба.
  • Оба пути валидны в разных фазах жизни продукта.
