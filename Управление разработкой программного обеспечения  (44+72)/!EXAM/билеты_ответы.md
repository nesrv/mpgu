## Билет 1: Введение в управление разработкой ПО. Основные понятия и терминология

### Теоретическая часть

#### 1. Определение управления разработкой программного обеспечения и основные цели и задачи

**Определение:** Управление разработкой программного обеспечения — это комплексная дисциплина, включающая планирование, организацию, контроль и координацию процессов создания, тестирования, внедрения и поддержки программных продуктов.

**Основные цели:**
- Обеспечение своевременной разработки ПО с требуемым качеством
- Оптимизация использования ресурсов (временных, финансовых, человеческих)
- Управление рисками в процессе разработки
- Обеспечение соответствия продукта требованиям заказчика
- Поддержание эффективной работы команды разработки

**Основные задачи:**
- Планирование и оценка трудозатрат
- Управление требованиями и изменениями
- Организация процессов разработки и тестирования
- Контроль качества и метрик проекта
- Управление командой и коммуникациями
- Управление рисками и проблемами
- Документирование и отчетность

#### 2. Основные различия между управлением проектами и управлением разработкой ПО

| Критерий | Управление проектами | Управление разработкой ПО |
|----------|---------------------|---------------------------|
| **Сфера применения** | Любые проекты (строительство, мероприятия, IT и др.) | Специфично для разработки ПО |
| **Фокус** | Общее управление проектом, сроками, бюджетом | Технические аспекты, процессы разработки, качество кода |
| **Жизненный цикл** | Инициация → Планирование → Выполнение → Контроль → Завершение | Требования → Проектирование → Кодирование → Тестирование → Внедрение → Поддержка |
| **Метрики** | Бюджет, сроки, объем работ | Скорость разработки, покрытие тестами, качество кода, производительность |
| **Инструменты** | MS Project, Jira, Trello | Git, CI/CD, инструменты разработки, системы управления версиями |
| **Риски** | Общие проектные риски | Технические долги, уязвимости безопасности, проблемы производительности |

**Вывод:** Управление разработкой ПО является специализацией управления проектами с фокусом на технических аспектах, процессах разработки и особенностях программных продуктов.

### Практическая часть

**Кейс неуспешного проекта:** Разработка CRM-системы для крупной компании (срок 6 месяцев, команда 10 человек, бюджет $500K)

**Основные причины неудачи:**

1. **Отсутствие четких требований** — требования менялись еженедельно без контроля изменений
2. **Нереалистичные сроки** — недооценка сложности на 40%, отсутствие буферного времени
3. **Отсутствие тестирования на ранних этапах** — дефекты обнаружены только на этапе внедрения
4. **Плохая коммуникация** — отсутствие регулярных встреч с заказчиком
5. **Недостаток технической экспертизы** — неправильный выбор технологий, накопление технического долга

**Меры по предотвращению:**

1. **Формализация требований** — создание детального технического задания с версионированием, процесс управления изменениями требований
2. **Реалистичная оценка** — использование нескольких методов оценки (story points, экспертная оценка), добавление 20-30% буферного времени
3. **Shift-left тестирование** — начало тестирования с первых спринтов, автоматизация тестирования, CI/CD
4. **Регулярная коммуникация** — ежедневные стендапы, еженедельные демо, ретроспективы
5. **Технический аудит** — code review, архитектурные решения на ранних этапах, метрики качества кода

## Билет 2: Процессы разработки ПО. Классификация и моделирование процессов

### Теоретическая часть

#### 1. Основные процессы разработки ПО

**1. Анализ требований (Requirements Analysis):**
- Сбор и анализ требований заказчика
- Выявление функциональных и нефункциональных требований
- Валидация и согласование требований
- Создание спецификаций требований

**2. Проектирование (Design):**
- Архитектурное проектирование (выбор архитектуры, паттернов)
- Детальное проектирование (диаграммы классов, последовательности)
- Проектирование базы данных
- Проектирование пользовательского интерфейса

**3. Кодирование (Implementation/Coding):**
- Написание кода по спецификациям
- Следование стандартам кодирования
- Code review и рефакторинг
- Интеграция компонентов

**4. Тестирование (Testing):**
- Модульное тестирование
- Интеграционное тестирование
- Системное тестирование
- Приемочное тестирование

#### 2. Взаимосвязь процессов и важность формализации

**Взаимосвязь процессов:**
- Процессы имеют входы и выходы: выход одного процесса является входом другого
- Итеративность: процессы могут повторяться в циклах (Agile)
- Параллельность: некоторые процессы выполняются одновременно (тестирование во время разработки)
- Обратная связь: результаты тестирования влияют на требования и проектирование

**Почему важно формализовать процессы:**

1. **Предсказуемость** — команда знает, что делать на каждом этапе
2. **Стандартизация** — единые подходы повышают качество
3. **Контроль** — возможность отслеживать прогресс и метрики
4. **Масштабируемость** — процессы можно применять в разных проектах
5. **Обучение** — новые члены команды быстрее включаются
6. **Автоматизация** — формализованные процессы легче автоматизировать
7. **Соответствие стандартам** — ISO, CMMI требуют формализованных процессов

### Практическая часть

**Диаграмма процессов разработки веб-приложения для управления задачами:**

```
[Требования заказчика]
         ↓
[Анализ требований] → [Спецификация требований]
         ↓                      ↓
[Архитектурное проектирование] → [Архитектура системы]
         ↓                              ↓
[Детальное проектирование] → [Дизайн-документы]
         ↓                      ↓
[Кодирование фронтенда] ← → [Кодирование бэкенда]
         ↓                      ↓
[Интеграция компонентов] → [Интегрированное ПО]
         ↓                      ↓
[Модульное тестирование] ← → [Интеграционное тестирование]
         ↓                              ↓
[Системное тестирование] → [Протестированное ПО]
         ↓                      ↓
[Исправление дефектов] ← → [Документирование]
         ↓                      ↓
[Внедрение] → [Эксплуатация]
```

**Входы и выходы:**

| Процесс | Входы | Выходы |
|---------|-------|--------|
| Анализ требований | Требования заказчика, интервью | Спецификация требований, User Stories |
| Проектирование | Спецификация требований | Диаграммы архитектуры, дизайн-документы |
| Кодирование | Дизайн-документы, спецификации API | Исходный код, документация кода |
| Тестирование | Исходный код, тест-кейсы | Отчеты о тестировании, список дефектов |

## Билет 3: Жизненный цикл разработки ПО. Этапы инициации, планирования и разработки

### Теоретическая часть

#### 1. Этап инициации жизненного цикла разработки ПО

**Задачи этапа инициации:**
- Определение целей и задач проекта
- Идентификация стейкхолдеров
- Оценка предварительной осуществимости проекта
- Определение основных требований и ограничений
- Выбор методологии разработки
- Формирование команды проекта

**Артефакты:**
- Устав проекта (Project Charter)
- Описание целей и задач проекта
- Список стейкхолдеров
- Предварительная оценка бюджета и сроков
- Риски и предположения проекта
- Методология разработки (выбранная)

#### 2. Этап планирования и разработки, взаимосвязь этапов

**Этап планирования:**
- Детальное планирование работ (WBS)
- Оценка трудозатрат и ресурсов
- Составление календарного плана
- Планирование рисков
- Определение метрик успеха

**Этап разработки:**
- Реализация спроектированного решения
- Написание кода по спецификациям
- Code review и рефакторинг
- Интеграция компонентов
- Непрерывная интеграция (CI)

**Взаимосвязь этапов:**
- Линейная зависимость: результаты одного этапа — входы для следующего
- Итеративная связь: в Agile этапы повторяются в циклах (спринтах)
- Обратная связь: результаты тестирования влияют на доработку кода
- Параллельность: планирование следующих спринтов во время разработки текущего

### Практическая часть

**Диаграмма жизненного цикла для учебного проекта "Система управления библиотекой":**

```
ИНИЦИАЦИЯ
├─ Устав проекта
├─ Цели: автоматизация учета книг
├─ Команда: 5 человек
└─ Методология: Scrum (спринты 2 недели)
         ↓
ПЛАНИРОВАНИЕ
├─ Product Backlog (30 user stories)
├─ Sprint 1 планирование (5 stories)
├─ Оценка: 40 story points
└─ Календарный план (6 спринтов)
         ↓
РАЗРАБОТКА (Sprint 1-6)
├─ Разработка функций
│  ├─ Модуль каталога книг
│  ├─ Модуль выдачи/возврата
│  └─ Модуль отчетности
├─ Code Review (еженедельно)
├─ CI/CD (автоматические тесты)
└─ Инкрементальные релизы
         ↓
ТЕСТИРОВАНИЕ
├─ Unit тесты (покрытие >80%)
├─ Integration тесты
├─ E2E тесты
└─ User Acceptance Testing
         ↓
ВНЕДРЕНИЕ
├─ Развертывание на сервере
├─ Миграция данных
├─ Обучение пользователей
└─ Поддержка
```

**Точки контроля:**
- Sprint Review (каждые 2 недели)
- Retrospective (каждые 2 недели)
- Code Review (каждые PR)
- QA Gate (перед релизом)
- UAT Gate (перед внедрением)

## Билет 4: Жизненный цикл разработки ПО. Этапы тестирования, внедрения и поддержки

### Теоретическая часть

#### 1. Этап тестирования в жизненном цикле разработки ПО

**Виды тестирования:**

1. **Модульное тестирование (Unit Testing)** — тестирование отдельных функций/классов
2. **Интеграционное тестирование (Integration Testing)** — тестирование взаимодействия компонентов
3. **Системное тестирование (System Testing)** — тестирование системы в целом
4. **Приемочное тестирование (UAT)** — тестирование пользователями
5. **Регрессионное тестирование** — проверка, что новые изменения не сломали существующий функционал
6. **Нагрузочное тестирование (Performance Testing)** — проверка производительности
7. **Тестирование безопасности (Security Testing)** — выявление уязвимостей
8. **Usability Testing** — тестирование удобства использования

**Этап тестирования включает:**
- Планирование тестирования
- Разработку тест-кейсов
- Выполнение тестов
- Отчетность о дефектах
- Проверку исправлений

#### 2. Этап поддержки ПО и важность планирования поддержки на ранних этапах

**Этап поддержки включает:**
- Устранение ошибок и дефектов
- Обновление и модификация функций
- Техническая поддержка пользователей
- Мониторинг производительности
- Обновление документации

**Почему важно планировать поддержку на ранних этапах:**

1. **Архитектурные решения** — выбор модульной архитектуры упрощает поддержку
2. **Документация** — создание документации во время разработки легче, чем после
3. **Мониторинг** — внедрение логирования и мониторинга на этапе разработки
4. **Бюджет** — планирование ресурсов на поддержку (обычно 15-20% бюджета)
5. **Команда** — определение команды поддержки заранее
6. **Процессы** — настройка процессов обработки инцидентов

### Практическая часть

**План внедрения и поддержки для системы управления заказами:**

**1. Стратегия развертывания: Blue-Green Deployment**

- **Blue (текущая версия):** Прод-сервер
- **Green (новая версия):** Тестовый сервер → проверка → переключение трафика

**Этапы развертывания:**
1. Развертывание на тестовом окружении (1 день)
2. Тестирование функциональности (2 дня)
3. Миграция на продакшн (Blue-Green, без downtime)
4. Мониторинг в течение 48 часов
5. Откат при необходимости (в течение 1 часа)

**2. План миграции данных:**

- **Подготовка:** Бэкап текущей БД, создание скриптов миграции
- **Миграция:** Инкрементальная миграция по модулям (заказы → клиенты → продукты)
- **Валидация:** Проверка целостности данных после миграции
- **Откат:** Возможность отката к старой версии БД

**3. Процедуры поддержки пользователей:**

- **L1 (1-й уровень):** Обработка типовых вопросов, роутинг (8x5)
- **L2 (2-й уровень):** Решение технических проблем (8x5)
- **L3 (3-й уровень):** Разработчики для критических багов (24/7 on-call)

**SLA:**
- Критические инциденты: реакция 1 час, решение 4 часа
- Высокий приоритет: реакция 4 часа, решение 1 день
- Средний приоритет: реакция 1 день, решение 3 дня

**Инструменты поддержки:**
- Система тикетов (Jira Service Management)
- База знаний (Confluence)
- Мониторинг (Datadog, Alerts)

## Билет 5: Роли и ответственность в команде разработки. Современные роли

### Теоретическая часть

#### 1. Современные роли 2026 года и их отличие от традиционных

**Platform Engineer:**
- **Обязанности:** Создание и поддержка внутренних платформ разработки (IDP), инфраструктура как код, self-service инструменты
- **Отличие от DevOps:** Фокус на создании платформы для разработчиков, а не только на автоматизации

**SRE (Site Reliability Engineer):**
- **Обязанности:** Обеспечение надежности систем, SLI/SLO, мониторинг, автоматизация операций
- **Отличие от системного администратора:** Применение принципов разработки к операциям, автоматизация рутинных задач

**DevSecOps Engineer:**
- **Обязанности:** Интеграция безопасности в CI/CD, сканирование зависимостей, анализ уязвимостей
- **Отличие от Security Engineer:** Безопасность встраивается в процесс разработки, а не проверяется отдельно

**ML Engineer:**
- **Обязанности:** Разработка ML-моделей, MLOps, интеграция AI в продукты
- **Отличие от Data Scientist:** Фокус на продакшн-развертывании и эксплуатации моделей

#### 2. Team Topologies и Conway's Law

**Team Topologies (4 типа команд):**

1. **Stream-aligned teams** — команды, работающие над потоком бизнес-ценности (продуктовые команды)
2. **Platform teams** — команды, создающие внутренние платформы для других команд
3. **Enabling teams** — команды, помогающие другим командам развиваться (коучинг, экспертиза)
4. **Complicated-subsystem teams** — команды, работающие над сложными техническими подсистемами

**Conway's Law:**
"Организации проектируют системы, которые копируют структуру коммуникации этих организаций."

**Влияние:**
- Архитектура системы отражает структуру команды
- Монолитная команда → монолитная архитектура
- Микросервисы требуют независимых команд
- Необходимо согласовать организационную структуру с желаемой архитектурой

### Практическая часть

**Распределение ролей в команде из 8 человек для микросервисной платформы:**

| Роль | Количество | Обязанности |
|------|------------|-------------|
| Tech Lead / Архитектор | 1 | Архитектура системы, технические решения |
| Backend Developer | 2 | Разработка микросервисов (API Gateway, User Service, Order Service) |
| Frontend Developer | 1 | Веб-интерфейс администратора и пользователя |
| DevOps / Platform Engineer | 1 | Kubernetes, CI/CD, мониторинг |
| QA Engineer | 1 | Тестирование, автоматизация тестов |
| Product Owner | 1 | Требования, приоритизация, коммуникация с бизнесом |
| Scrum Master | 1 | Фасилитация процессов, удаление препятствий |

**RACI матрица (для модуля "Управление заказами"):**

| Задача | Tech Lead | Backend | Frontend | DevOps | QA | PO | SM |
|--------|-----------|---------|----------|--------|----|----|----|
| Архитектура модуля | A | R | C | C | I | I | I |
| Разработка API | C | A | I | I | I | R | I |
| Разработка UI | C | I | A | I | I | R | I |
| Развертывание | C | C | I | A | I | I | I |
| Тестирование | I | C | C | I | A | R | I |

**Легенда:** R - Responsible, A - Accountable, C - Consulted, I - Informed

**Организация по Team Topologies:**

- **Stream-aligned team (6 человек):** Backend (2) + Frontend (1) + QA (1) + PO (1) + SM (1)
- **Platform team (1 человек):** DevOps / Platform Engineer
- **Enabling role (1 человек):** Tech Lead / Архитектор (помогает с техническими решениями)

## Билет 6: Коммуникации в проектах разработки ПО. Remote и hybrid работа

### Теоретическая часть

#### 1. Async-first подход и виды коммуникаций

**Async-first подход:**
Приоритет асинхронной коммуникации над синхронной. Команда не должна ждать реального времени для получения информации.

**Принципы:**
- Документирование важной информации
- Использование инструментов с историей сообщений
- Минимизация синхронных встреч
- Четкие сроки ответа на сообщения

**Виды коммуникаций:**

**Синхронная:**
- Встречи в реальном времени (Zoom, Teams)
- Видеозвонки
- Чаты с моментальным ответом
- **Недостатки:** Требует присутствия всех, проблемы с timezone

**Асинхронная:**
- Email
- Slack/Teams сообщения (без требования моментального ответа)
- Документация (Wiki, Confluence)
- Code comments, PR reviews
- **Преимущества:** Гибкость времени, меньше прерываний, лучше для удаленных команд

- Фиксированное окно для встреч (перекрытие часовых поясов)

- Async-first для большинства коммуникаций

## Билет 9: Методологии разработки ПО. Классификация: традиционные и гибкие методологии

### Теоретическая часть

#### 1. Классификация методологий и сравнение

**Традиционные методологии:**
- Waterfall, V-Model, RUP (Rational Unified Process)
- Последовательная разработка
- Жесткое планирование, формальная документация

**Гибкие методологии:**
- Agile, Scrum, Kanban, XP (Extreme Programming)
- Итеративная разработка
- Гибкое планирование, минимальная документация

**Сравнение по критериям:**

| Критерий | Традиционные | Гибкие |
|----------|--------------|--------|
| **Гибкость** | Низкая — сложно изменить требования | Высокая — требования меняются в процессе |
| **Предсказуемость** | Высокая — четкие сроки и бюджет | Средняя — сроки гибкие, бюджет приблизительный |
| **Скорость разработки** | Медленная — долгое планирование | Быстрая — инкрементальная разработка |
| **Документация** | Обширная формальная | Минимальная, по требованию |
| **Команда** | Большая, формальная структура | Малая, самоорганизующаяся |
| **Риски** | Выявляются на поздних этапах | Выявляются рано, итеративно |

#### 2. Критерии выбора методологии и области применения

**Критерии выбора методологии:**

1. **Стабильность требований:**
   - Стабильные → Waterfall
   - Нестабильные → Agile

2. **Размер проекта:**
   - Большой (>50 человек) → Waterfall или SAFe
   - Малый (<10 человек) → Scrum/Kanban

3. **Опыт команды:**
   - Неопытная → Waterfall (четкие инструкции)
   - Опытная → Agile (самоорганизация)

4. **Требования к документации:**
   - Высокие (госсектор) → Waterfall
   - Низкие (стартап) → Agile

5. **Срочность:**
   - Срочный → Agile (быстрые релизы)
   - Не срочный → Waterfall (тщательное планирование)

**Области применения:**

**Традиционные (Waterfall, V-Model):**
- Государственные системы (требования к документации)
- Критические системы (авиация, медицина)
- Проекты с четкими требованиями
- Большие корпорации с формальными процессами

**Гибкие (Agile, Scrum, Kanban):**
- Стартапы с быстро меняющимися требованиями
- Веб-приложения и SaaS
- Продуктовые команды
- Проекты с неясными требованиями

## Билет 10: Методологии Waterfall и V-Model. Области применения

### Теоретическая часть

#### 1. Принципы методологии Waterfall

**Принципы:**
- Последовательное выполнение этапов
- Каждый этап завершается до начала следующего
- Формальное документирование на каждом этапе
- Минимальные изменения после завершения этапа

**Последовательность этапов:**

1. **Требования (Requirements)** — сбор и документирование требований
2. **Проектирование (Design)** — архитектурное и детальное проектирование
3. **Реализация (Implementation)** — кодирование
4. **Тестирование (Testing)** — тестирование системы
5. **Внедрение (Deployment)** — развертывание
6. **Поддержка (Maintenance)** — поддержка и обновления

**Преимущества:**
- Предсказуемость сроков и бюджета
- Четкая структура и процессы
- Хорошая документация
- Подходит для проектов с четкими требованиями

**Недостатки:**
- Низкая гибкость к изменениям
- Долгое ожидание результатов (первые результаты только после реализации)
- Риски обнаруживаются поздно
- Сложность исправления ошибок на поздних этапах

#### 2. V-Model и отличия от Waterfall

**Особенности V-Model:**
- Тестирование связано с каждым этапом проектирования
- Левая ветвь "V" — проектирование (требования → система → компоненты)
- Правая ветвь "V" — тестирование (компоненты → интеграция → система → приемка)
- Каждому уровню проектирования соответствует уровень тестирования

**Отличия от Waterfall:**

| Критерий | Waterfall | V-Model |
|----------|-----------|---------|
| **Тестирование** | В конце проекта | Параллельно с каждым этапом проектирования |
| **Связь тестов с дизайном** | Слабая | Сильная — тесты планируются на этапе проектирования |
| **Раннее выявление ошибок** | Поздно | Рано — тесты готовы до реализации |
| **Сложность** | Проще | Сложнее — больше документирования |

**Области применения (2026):**

**Waterfall и V-Model применяются в:**
- Государственном секторе (требования к документации, формальные процессы)
- Крупных корпорациях с жесткими процессами
- Критических системах (авиация, медицина, финансы)
- Проектах с четкими, неизменными требованиями
- Регулируемых отраслях (FDA, ISO стандарты)

**Почему до сих пор используются:**
- Требования законодательства (формальная документация)

## Билет 11: Методологии Agile, Scrum и Kanban. Практическое применение

### Теоретическая часть

#### 1. Принципы Манифеста Agile и артефакты Scrum

**Манифест Agile (4 ценности):**

1. **Люди и взаимодействие важнее процессов и инструментов**
2. **Работающее ПО важнее исчерпывающей документации**
3. **Сотрудничество с заказчиком важнее согласования условий контракта**
4. **Готовность к изменениям важнее следования первоначальному плану**

**12 принципов Agile (ключевые):**

- Удовлетворение заказчика за счет ранней и непрерывной поставки ценного ПО
- Изменения требований приветствуются (даже на поздних этапах)
- Работающее ПО поставляется часто (от 2 недель до 2 месяцев)
- Бизнес и разработчики работают вместе ежедневно
- Проекты строятся на мотивированных личностях
- Наиболее эффективный метод передачи информации — личный разговор
- Работающее ПО — главный показатель прогресса
- Agile процессы способствуют устойчивому развитию (постоянный темп)
- Постоянное внимание к техническому совершенству и хорошему дизайну
- Простота (искусство максимизации объема незавершенной работы) важна
- Самоорганизующиеся команды создают лучшие архитектуры, требования и дизайны
- Команда регулярно осмысливает, как стать эффективнее, и соответственно корректирует свое поведение

**Артефакты Scrum:**

**1. Product Backlog:**
- Упорядоченный список всех требований к продукту
- Приоритизирован Product Owner
- Живой документ (постоянно обновляется)
- Содержит User Stories с оценками

**2. Sprint Backlog:**
- Выбранные из Product Backlog задачи для текущего спринта
- Создается на Sprint Planning
- Команда отвечает за выполнение
- Может обновляться командой в течение спринта

**3. Increment:**
- Работающий продукт, созданный в результате спринта
- Должен соответствовать Definition of Done
- Инкрементально добавляется к предыдущим спринтам
- Демонстрируется на Sprint Review

#### 2. Роли и события Scrum

**Роли в Scrum:**

**1. Product Owner (PO):**
- Отвечает за максимизацию ценности продукта
- Управляет Product Backlog
- Приоритизирует требования
- Коммуникация с бизнесом и стейкхолдерами

**2. Scrum Master (SM):**
- Фасилитатор процессов Scrum
- Удаляет препятствия (impediments)
- Защищает команду от внешних отвлечений
- Обучает команду Scrum-практикам

**3. Команда разработки:**
- Самоорганизующаяся команда (5-9 человек)
- Кросс-функциональные навыки
- Отвечает за выполнение Sprint Backlog
- Совместно оценивает работу

**События Scrum:**

**1. Sprint:**
- Фиксированный период времени (1-4 недели, обычно 2 недели)
- В течение спринта не меняются цели и состав команды
- Результат — работающий Increment

**2. Sprint Planning:**
- В начале спринта (до 8 часов для 2-недельного спринта)
- Команда выбирает задачи из Product Backlog
- Планирует работу на спринт

**3. Daily Scrum (Daily Standup):**
- Ежедневно, 15 минут
- Три вопроса: что сделал, что буду делать, какие препятствия
- Не решение проблем, а выявление препятствий

**4. Sprint Review:**
- В конце спринта (до 4 часов)
- Демонстрация Increment стейкхолдерам
- Обратная связь и обновление Product Backlog

**5. Sprint Retrospective:**
- После Sprint Review, перед следующим Sprint Planning (до 3 часов)
- Команда анализирует процесс и улучшает его
- Формат: что хорошо, что плохо, что улучшить

## Билет 12: Extreme Programming (XP) и гибридные методологии. AI в Agile-процессах

### Теоретическая часть

#### 1. Использование AI в Agile-процессах

**AI для планирования спринтов:**

**1. Приоритизация задач:**
- AI анализирует исторические данные и предлагает приоритеты
- Учет бизнес-ценности, технического долга, зависимостей
- Предсказание влияния на метрики (velocity, качество)

**2. Оценка задач:**
- AI предлагает story points на основе описания задач
- Анализ похожих задач из истории проекта
- Учет сложности и рисков

**3. Оптимальное распределение задач:**
- AI предлагает распределение задач по разработчикам
- Учет экспертизы и загрузки
- Балансировка нагрузки команды

**Автоматизация ретроспектив и других Agile-процессов:**

**1. Автоматизация ретроспектив:**
- AI собирает данные из Jira, Git, Slack
- Анализ метрик: velocity, количество багов, time to resolve
- Генерация вопросов для ретроспективы на основе данных
- Выявление паттернов проблем

**2. Daily Standup автоматизация:**
- AI-боты собирают статус из Git commits, PR
- Автогенерация ежедневных отчетов
- Выявление блокеров на основе данных

**3. Sprint Review автоматизация:**
- AI генерирует демо-сценарии на основе реализованных фич
- Автоматические чендж-логи
- Анализ обратной связи стейкхолдеров

**4. Улучшение метрик:**
- Предсказание velocity на основе истории
- Выявление рисков срыва спринта
- Рекомендации по улучшению процессов

**Примеры инструментов:**
- Linear — AI для оценки задач
- Jira AI — автоматизация планирования
- ChatGPT — генерация вопросов для ретроспектив

#### 2. Гибридные методологии и их преимущества

**Гибридные методологии:**
Комбинация элементов разных методологий для адаптации к специфике проекта.

**Преимущества:**
- Гибкость — выбор лучших практик из разных методологий
- Адаптация — методология подстраивается под проект, а не наоборот
- Масштабируемость — можно начать с простого и усложнять
- Практичность — использование проверенных практик

**Scrumban:**
Комбинация Scrum и Kanban

**Элементы Scrum:**
- Sprint Planning
- Daily Standup
- Sprint Review
- Retrospective

**Элементы Kanban:**
- Визуализация потока (Kanban-доска)
- WIP Limits
- Continuous flow (не фиксированные спринты)
- Метрики: Lead Time, Cycle Time

**Когда использовать:**
- Поддержка продуктов (непрерывный поток работ)
- Неравномерная загрузка команды
- Нужна гибкость Kanban с структурой Scrum

**SAFe (Scaled Agile Framework):**
Фреймворк для масштабирования Agile на уровень портфеля

**Уровни SAFe:**
1. **Team Level** — отдельные команды используют Scrum/Kanban
2. **Program Level** — несколько команд работают вместе (Agile Release Train)
3. **Portfolio Level** — стратегическое планирование на уровне портфеля

**Ключевые практики:**
- Agile Release Trains (ART) — долгосрочные итерации (8-12 недель)
- PI Planning — планирование на уровне программы
- System Demo — демонстрация интегрированного решения
- Inspect & Adapt — ретроспективы на уровне программы

**Когда использовать:**
- Большие организации (50+ разработчиков)
- Несколько команд работают над одним продуктом
- Нужна координация на уровне программы
- Интеграция с корпоративными процессами

### Практическая часть

**Гибридная методология для крупного проекта (50+ человек, 5 команд):**

**Проект:** Разработка платформы электронной коммерции (микросервисная архитектура)

**Структура команд:**
- Команда 1: User Service (10 человек)
- Команда 2: Product Service (10 человек)
- Команда 3: Order Service (12 человек)
- Команда 4: Payment Service (8 человек)
- Команда 5: Platform Team (10 человек)

**Гибридная методология: SAFe + Scrum + Kanban**

**1. Team Level (Scrum для каждой команды):**

- **Sprint Duration:** 2 недели
- **Артефакты:** Product Backlog, Sprint Backlog, Increment
- **События:** Sprint Planning, Daily Standup, Sprint Review, Retrospective

**2. Program Level (SAFe элементы):**

- **PI (Program Increment):** 12 недель (6 спринтов)
- **PI Planning:** Планирование на 12 недель каждые 3 месяца
- **System Demo:** Каждые 2 недели, демонстрация интегрированного решения
- **Inspect & Adapt:** Ретроспектива на уровне программы после PI

**3. Platform Team (Kanban):**

- **Continuous flow:** Нет фиксированных спринтов
- **WIP Limits:** 3 задачи максимум в работе
- **Метрики:** Lead Time, Cycle Time для инфраструктуры

**Использование AI:**

**1. Sprint Planning:**
- AI помогает в оценке задач (Linear AI)
- Предложение приоритетов на основе бизнес-метрик

**2. Retrospectives:**
- AI анализирует метрики из Jira, Git, Datadog
- Генерация вопросов: "Velocity снизилась на 15% в последних 2 спринтах, почему?"

**3. Daily Standup:**
- AI-бот собирает статус из Git commits
- Автоматический отчет: "Команда 2 завершила 3 PR, есть блокер в задаче ORD-123"

**4. PI Planning:**
- AI помогает в оценке зависимостей между командами
- Предсказание рисков интеграции
