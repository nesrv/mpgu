## Билет 1: Введение в управление разработкой ПО

### Теоретическая часть

**1. Что такое управление разработкой ПО и зачем оно нужно?**

Управление разработкой ПО — планирование, организация и контроль процессов создания программных продуктов. 

**Зачем нужно:**
- Уложиться в сроки и бюджет
- Обеспечить качество продукта
- Управлять командой и рисками
- Соблюдать требования заказчика

**Пример:** В проекте разработки CRM-системы управление включает планирование спринтов, оценку задач, контроль качества кода и коммуникацию с заказчиком.

**2. Чем отличается управление разработкой ПО от обычного управления проектами?**

| Управление проектами | Управление разработкой ПО |
|---------------------|--------------------------|
| Любые проекты | Только разработка ПО |
| Фокус на бюджет и сроки | Фокус на качество кода, тестирование |
| Метрики: бюджет, сроки | Метрики: velocity, покрытие тестами, качество кода |
| Инструменты: Jira, MS Project | Инструменты: Git, CI/CD, системы управления версиями |

### Практическая часть

**Кейс неуспешного проекта:** CRM-система (6 месяцев, 10 человек, $500K)

**Причины неудачи:**
1. Нечеткие требования (менялись еженедельно)
2. Нереалистичные сроки (недооценка на 40%)
3. Позднее тестирование (дефекты на этапе внедрения)
4. Плохая коммуникация (нет регулярных встреч)

**Меры предотвращения:**
1. Формализация требований (ТЗ с версионированием)
2. Реалистичная оценка (story points, буфер 20-30%)
3. Shift-left тестирование (тесты с первых спринтов)
4. Регулярная коммуникация (daily standup, еженедельные демо)

## Билет 2: Процессы разработки ПО

### Теоретическая часть

**1. Какие основные процессы разработки ПО вы знаете?**

1. **Анализ требований** — сбор и документирование требований (User Stories, ТЗ)
2. **Проектирование** — архитектура системы, проектирование БД, API
3. **Кодирование** — написание кода, code review, рефакторинг
4. **Тестирование** — unit, integration, system, UAT

**Пример:** При разработке веб-приложения сначала собираем требования (User Stories), затем проектируем архитектуру (микросервисы или монолит), потом пишем код и тестируем.

**2. Почему важно формализовать процессы разработки?**

- Предсказуемость (команда знает, что делать)
- Стандартизация (единые подходы)
- Масштабируемость (применение в разных проектах)
- Автоматизация (формализованные процессы легче автоматизировать)

### Практическая часть

**Процессы для веб-приложения управления задачами:**

Требования → Проектирование → Кодирование → Тестирование → Внедрение

**Входы/выходы:**
- Анализ требований: вход — требования, выход — User Stories
- Проектирование: вход — User Stories, выход — архитектура, API
- Кодирование: вход — архитектура, выход — код
- Тестирование: вход — код, выход — отчеты, список багов

## Билет 3: Жизненный цикл разработки ПО

### Теоретическая часть

**1. Что включает этап инициации проекта?**

**Задачи:**
- Определение целей проекта
- Идентификация стейкхолдеров
- Оценка осуществимости
- Выбор методологии (Scrum, Waterfall)
- Формирование команды

**Артефакты:** Устав проекта, список стейкхолдеров, предварительная оценка

**2. Как связаны этапы планирования и разработки?**

В Agile этапы повторяются в спринтах: планирование → разработка → тестирование → демо. Результаты тестирования влияют на доработку кода и следующее планирование.

### Практическая часть

**Жизненный цикл для проекта "Система управления библиотекой":**

```
Инициация → Планирование (Product Backlog, Sprint Planning) 
→ Разработка (6 спринтов по 2 недели) 
→ Тестирование → Внедрение
```

**Точки контроля:** Sprint Review (каждые 2 недели), QA Gate (перед релизом)

## Билет 4: Этапы тестирования, внедрения и поддержки

### Теоретическая часть

**1. Какие виды тестирования применяются в разработке ПО?**

- **Unit тесты** — тестирование отдельных функций
- **Integration тесты** — тестирование взаимодействия компонентов
- **System тесты** — тестирование системы в целом
- **UAT** — приемочное тестирование пользователями

**2. Почему важно планировать поддержку на ранних этапах?**

- Архитектурные решения влияют на поддержку (модульность упрощает исправления)
- Документация легче создается во время разработки
- Мониторинг нужно внедрять на этапе разработки
- Бюджет на поддержку (обычно 15-20% от бюджета разработки)

### Практическая часть

**План внедрения системы управления заказами:**

**Развертывание:** Blue-Green (тестовый → проверка → переключение на прод)

**Миграция данных:** Инкрементальная (заказы → клиенты → продукты) с валидацией

**Поддержка:** L1/L2/L3 уровни, SLA: критические — реакция 1 час, решение 4 часа

## Билет 5: Роли в команде разработки

### Теоретическая часть

**1. Чем отличаются современные роли от традиционных?**

**Platform Engineer** — создает внутренние платформы для разработчиков (self-service инструменты), отличие от DevOps — фокус на платформе, а не только автоматизации

**SRE** — обеспечивает надежность систем (SLI/SLO), отличие от сисадмина — применение принципов разработки к операциям

**DevSecOps Engineer** — интегрирует безопасность в CI/CD, отличие от Security Engineer — безопасность встроена в процесс, а не проверяется отдельно

**ML Engineer** — продакшн-развертывание ML-моделей, отличие от Data Scientist — фокус на продакшн, а не исследовании

**2. Что такое Team Topologies и Conway's Law?**

**Team Topologies** — 4 типа команд:
- Stream-aligned (продуктовые команды)
- Platform (внутренние платформы)
- Enabling (коучинг, экспертиза)
- Complicated-subsystem (сложные подсистемы)

**Conway's Law:** Архитектура системы копирует структуру команды. Микросервисы требуют независимых команд.

### Практическая часть

**Распределение ролей (8 человек, микросервисная платформа):**

- Tech Lead (1) — архитектура
- Backend Dev (2) — микросервисы
- Frontend Dev (1) — UI
- DevOps/Platform (1) — инфраструктура
- QA (1) — тестирование
- PO (1) — требования
- SM (1) — процессы

**RACI (для модуля "Управление заказами"):**
- Архитектура: Tech Lead (A), Backend (R)
- Разработка API: Backend (A)
- Развертывание: DevOps (A)
- Тестирование: QA (A)

## Билет 6: Коммуникации в remote/hybrid командах

### Теоретическая часть

**1. Что такое async-first подход и виды коммуникаций?**

**Async-first** — приоритет асинхронной коммуникации (не ждем реального времени).

**Виды:**
- **Синхронная** — Zoom, чаты (требует присутствия всех, проблемы с timezone)
- **Асинхронная** — Email, Slack, документация (гибкость времени, лучше для remote)

**2. Почему важна культура документации для remote-команд и как решать проблемы timezone?**

**Важность:** Удаленные команды не могут "подойти и спросить", информация должна быть доступна всем.

**Правила:** "Documentation as code", README для каждого проекта, ADR для архитектурных решений.

**Проблемы timezone:** Разные часовые пояса (3-12 часов), ограниченное окно для встреч.

**Решения:** Фиксированное окно встреч (перекрытие timezone), async-first для большинства коммуникаций, SLA на ответы (24 часа).

### Практическая часть

**План коммуникаций (12 человек, 3 timezone):**

- **Каналы:** Slack (ежедневная работа), GitHub (code reviews), Confluence (документация)
- **Встречи:** Daily Standup (async в Slack до 10:00 UTC), Sprint Planning (синхронно 10:00-12:00 UTC)
- **Правила:** Архитектурные решения → ADR, API изменения → OpenAPI, важные обсуждения → резюме в Confluence

## Билет 7: Инструменты управления разработкой

### Теоретическая часть

**1. Как AI-инструменты влияют на разработку ПО?**

**GitHub Copilot** — автодополнение кода (ускоряет на 30-50%)

**ChatGPT/Claude** — генерация кода по описанию, рефакторинг, документация

**Cursor** — AI-powered редактор с интеграцией LLM

**Влияние:** Ускорение разработки на 20-40%, улучшение качества (best practices встроены), обучение разработчиков

**2. Что такое Platform Engineering и инструменты observability?**

**Platform Engineering** — создание внутренних платформ для разработчиков (self-service). Инструменты: Backstage (портал разработчика), Cortex (service ownership).

**Observability** — три столпа: метрики (Datadog, Prometheus), логи (Datadog, Grafana), трейсинг (OpenTelemetry, Datadog APM).

### Практическая часть

**Набор инструментов для SaaS-приложения (15 человек):**

- AI: GitHub Copilot, ChatGPT
- Проекты: Jira, Confluence
- CI/CD: GitHub Actions, Kubernetes
- Observability: Datadog (APM, метрики, логи), Sentry (ошибки)

**GitHub Actions пайплайн:** test → security scan → deploy (только при успехе test и security)

## Билет 8: AI в разработке ПО

### Теоретическая часть

**1. Как LLM влияют на процессы разработки и что такое DevEx метрики?**

**Влияние LLM:**
- Автоматизация кода (генерация функций)
- Code review (AI-анализ ошибок)
- Генерация документации и тестов

**DevEx метрики (Developer Experience):**
- Time to First Commit (< 1 дня)
- Time to First Deploy (< 1 недели)
- CI/CD Duration (< 30 минут)
- Developer Satisfaction (> 7/10)

**2. Какие этические аспекты AI в разработке?**

- **Лицензирование AI-кода** — проверка лицензий (GPL и др.)
- **Надежность и безопасность** — code review даже для AI-кода, проверка уязвимостей
- **Ответственность** — разработчик отвечает за AI-сгенерированный код
- **Предвзятость** — проверка на справедливость алгоритмов

### Практическая часть

**Генерация REST API эндпоинта (ChatGPT):**

**Промпт:** "FastAPI эндпоинт для списка пользователей с пагинацией"

**Результат:** Код корректный, но нужна проверка (обработка ошибок, подключение к БД)

**Этические аспекты:** Код проверен разработчиком, добавлена обработка ошибок, использование AI документировано

## Билет 9: Методологии разработки ПО

### Теоретическая часть

**1. Сравните традиционные и гибкие методологии.**

| Критерий | Традиционные (Waterfall) | Гибкие (Agile) |
|----------|-------------------------|----------------|
| Гибкость | Низкая | Высокая |
| Предсказуемость | Высокая | Средняя |
| Скорость | Медленная | Быстрая |
| Документация | Обширная | Минимальная |

**2. По каким критериям выбирать методологию?**

- **Стабильность требований** (стабильные → Waterfall, нестабильные → Agile)
- **Размер проекта** (большой → Waterfall/SAFe, малый → Scrum/Kanban)
- **Требования к документации** (высокие → Waterfall, низкие → Agile)
- **Срочность** (срочный → Agile, не срочный → Waterfall)

**Области применения:**
- Waterfall: госсектор, критические системы
- Agile: стартапы, веб-приложения

### Практическая часть

**Выбор методологии для трех проектов:**

- **Проект А (госсистема):** Waterfall — жесткие требования к документации
- **Проект Б (стартап):** Scrum — быстро меняющиеся требования, нужны быстрые релизы
- **Проект В (корпоративный):** Scrumban/SAFe — смешанные требования, большая команда

## Билет 10: Waterfall и V-Model

### Теоретическая часть

**1. Принципы Waterfall и его преимущества/недостатки.**

**Принципы:** Последовательные этапы, формальная документация, минимальные изменения.

**Этапы:** Требования → Проектирование → Реализация → Тестирование → Внедрение → Поддержка

**Преимущества:** Предсказуемость, четкая структура, хорошая документация

**Недостатки:** Низкая гибкость, долгое ожидание результатов, риски обнаруживаются поздно

**2. Отличия V-Model от Waterfall и области применения.**

**Отличие:** В V-Model тестирование связано с каждым этапом проектирования (левая ветвь — проектирование, правая — тестирование).

**Применение:** Госсектор, критические системы (авиация, медицина), крупные корпорации

### Практическая часть

**План проекта по Waterfall (система документооборота, 12 месяцев):**

- **1-2 месяц:** Требования (ТЗ, спецификации)
- **3-5 месяц:** Проектирование (архитектура, БД, интерфейсы)
- **6-9 месяц:** Реализация (разработка модулей)
- **10-11 месяц:** Тестирование (unit, integration, system, UAT)
- **12 месяц:** Внедрение (развертывание, миграция, обучение)

**Точки контроля:** Утверждение ТЗ, утверждение архитектуры, критерии приемки выполнены

## Билет 11: Agile, Scrum и Kanban

### Теоретическая часть

**1. Принципы Agile и артефакты Scrum.**

**Манифест Agile (4 ценности):**
1. Люди важнее процессов
2. Работающее ПО важнее документации
3. Сотрудничество с заказчиком важнее контракта
4. Готовность к изменениям важнее плана

**Артефакты Scrum:**
- **Product Backlog** — список всех требований (приоритизирован PO)
- **Sprint Backlog** — задачи на текущий спринт
- **Increment** — работающий продукт после спринта

**2. Роли и события Scrum.**

**Роли:** Product Owner (требования, приоритеты), Scrum Master (фасилитация процессов), команда разработки (5-9 человек)

**События:** Sprint (2 недели), Sprint Planning (начало спринта), Daily Scrum (15 минут ежедневно), Sprint Review (демо), Retrospective (улучшение процессов)

### Практическая часть

**Product Backlog для мобильного приложения (примеры User Stories):**

1. US-1: Зарегистрироваться (8 SP) - Must Have
2. US-2: Войти в аккаунт (5 SP) - Must Have
3. US-3: Отслеживать шаги (13 SP) - Must Have
4. US-4: Статистика за день (8 SP) - Must Have
5. US-5: Установить цели (5 SP) - Must Have

**Sprint 1 (2 недели, 5 разработчиков):** Выбрано US-1 до US-5 (39 SP)

**Kanban-доска:** To Do | In Progress (макс 3) | Review (макс 2) | Testing (макс 2) | Done

## Билет 12: XP и гибридные методологии

### Теоретическая часть

**1. Как AI используется в Agile-процессах?**

**AI для планирования:**
- Приоритизация задач (анализ истории)
- Оценка задач (story points на основе похожих задач)
- Распределение задач (по экспертизе и загрузке)

**Автоматизация ретроспектив:**
- AI собирает метрики из Jira, Git, Slack
- Генерация вопросов на основе данных ("Velocity снизилась на 15%, почему?")
- Выявление паттернов проблем

**2. Что такое гибридные методологии и их примеры?**

**Гибридные методологии** — комбинация элементов разных методологий для адаптации к проекту.

**Scrumban** — Scrum (спринты, роли) + Kanban (WIP limits, continuous flow)

**SAFe** — фреймворк для масштабирования Agile (Team Level → Program Level → Portfolio Level), Agile Release Trains (8-12 недель)

**Преимущества:** Гибкость, адаптация под проект, масштабируемость

### Практическая часть

**Гибридная методология для проекта (50+ человек, 5 команд):**

**Team Level:** Scrum (каждая команда — 2-недельные спринты)

**Program Level:** SAFe (PI Planning раз в 3 месяца, System Demo каждые 2 недели)

**Platform Team:** Kanban (continuous flow, WIP limits)

**AI поддержка:**
- Sprint Planning: AI помогает в оценке задач (Linear AI)
- Retrospectives: AI анализирует метрики и генерирует вопросы
- Daily Standup: AI-бот собирает статус из Git commits

**Схема взаимодействия:** Команды работают в Scrum, Platform Team в Kanban, координация через PI Planning и System Demo
