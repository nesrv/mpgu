"Управление разработкой - это искусство балансировать между хотелками заказчика и реальными возможностями команды"

Типичные ситуации в IT

  • Junior developer в стартапе: есть крутая идея, но нет понимания процессов
  • Team lead: нужно объяснить бизнесу, почему "просто добавить кнопочку" займет две недели
  • Основатель стартапа: думает, что можно нанять пару программистов и через месяц получить готовый продукт

Что такое управление разработкой ПО?

Формальное определение:

Комплекс процессов планирования, организации, контроля и координации деятельности по созданию программного продукта

На практике:

Искусство балансировать между противоречивыми требованиями и ограничениями

Основные балансы в управлении

  • Хотелки заказчика vs Реальные возможности команды
  • Качество кода vs Time-to-market
  • Технический долг vs Новые фичи
  • Креативность разработчиков vs Дисциплина процессов

Почему нельзя управлять программистами как строителями?

  • Нематериальность продукта - код нельзя потрогать
  • Высокая степень творчества - один алгоритм можно написать десятками способов
  • Быстрое изменение требований - заказчик часто не знает, что хочет
  • Сложность оценки прогресса - "90% готово" это мем

Проблема нематериальности

Пример из московского стартапа:
Founder: "Сколько процентов готова авторизация?"
Developer: "90%"
Через неделю: "90%"
Еще через неделю: "90%"

Оказалось, что осталось написать обработку edge cases, но это заняло еще месяц.

Творческий характер программирования

Последние 10% могут занять столько же времени, сколько первые 90%

Строительство дома vs Разработка приложения

Строительство дома:

  • Четкий план
  • Материальный результат
  • Стандартные технологии
  • Легко измерить прогресс
  • Предсказуемые риски

Разработка приложения:

  • Требования меняются
  • Результат виден после deploy
  • Новые фреймворки каждые полгода
  • Прогресс нельзя измерить в %
  • Непредсказуемые риски

Реальный кейс: МТС

МТС заказала мобильное приложение для управления тарифами

Изначально: 5 экранов, 3 месяца
Через месяц: + чат с поддержкой
Еще через месяц: + интеграция с соцсетями
Потом: + пуш-уведомления, аналитика, геолокация...

Итог: проект растянулся с 3 месяцев на год

Участники IT-проекта

IT-зоопарк:

  • Заказчик - тот, кто платит деньги
  • Пользователь - тот, кто будет пользоваться продуктом
  • Заинтересованные стороны - все, кто может повлиять на проект

Важно: Заказчик и пользователь - часто разные люди!

Особенности заказчиков

  • Часто не понимают, что именно хотят ("Сделайте красиво")
  • Любят менять требования в процессе разработки
  • Хотят все и сразу, но за копейки
  • Не понимают, почему "просто добавить кнопочку" займет неделю

Примеры конфликта интересов

  • Банк: IT-директор заказывает систему, операционисты работают
  • Образование: Министерство заказывает электронный дневник, учителя и родители пользуются
  • Продажи: Руководитель хочет CRM, менеджеры вбивают данные

Заинтересованные стороны

Внутренние:

  • Инвесторы

Внешние:

  • Регуляторы
  • Партнеры

Управление - это постоянный поиск компромиссов

Product Owner vs Project Manager

Product Owner:

Отвечает за ЧТО делаем

  • Видение продукта
  • Приоритеты фич
  • Требования

Project Manager:

Отвечает за КАК и КОГДА делаем

  • Процессы
  • Сроки
  • Ресурсы

Требования - самая болезненная часть

Функциональные:

  • Что система делает
  • "Пользователь может перевести деньги"
  • "Система отправляет SMS"

Нефункциональные:

  • Как система работает
  • "1000 транзакций в секунду"
  • "Время отклика < 2 сек"

Проблема с требованиями

Заказчик: "Нам нужна система как у Amazon"
Разработчик: "Amazon - это сотни микросервисов, тысячи разработчиков, миллиарды долларов..."
Заказчик: "Ну сделайте попроще, но чтобы работало так же"

Архитектура ПО

Высокоуровневая структура системы - как план дома

Типы архитектур:

  • Монолит - все в одном (проще разрабатывать, сложнее масштабировать)
  • Микросервисы - много маленьких сервисов (сложнее разрабатывать, проще масштабировать)
  • Serverless - функции в облаке (дешево для малых нагрузок)

Эволюция архитектуры: ВКонтакте

• Начинали с монолита на PHP
• При росте до миллионов пользователей начали выделять сервисы
• Сначала поиск, потом мессенджер, потом рекомендации
• Сейчас - сотни микросервисов

Итерация

Повторяющийся цикл разработки фиксированной длительности

Что происходит в итерации:

  1. Планирование - что будем делать
  2. Разработка - пишем код
  3. Тестирование - проверяем работу
  4. Демо - показываем результат
  5. Ретроспектива - что можно улучшить

Типы релизов

  • Alpha - для внутреннего тестирования ("работает на моей машине")
  • Beta - для ограниченной группы ("найдите баги за нас")
  • RC - почти готовый продукт ("вроде все работает")
  • GA - финальная версия ("теперь точно работает")
  • Hotfix - срочное исправление ("все сломалось, чините быстро")

MVP - Minimum Viable Product

Самая важная концепция в современной разработке

Заказчик хочет автомобиль:
• Плохой MVP: колесо → ось → рама...
• Хороший MVP: самокат → велосипед → мотоцикл → автомобиль

Каждая версия решает задачу передвижения!

Technical Debt - Технический долг

Накопленные проблемы в коде, которые замедляют разработку

Примеры:

  • Код без тестов ("потом напишем")
  • Устаревшие библиотеки ("работает же")
  • Костыли и хаки ("временное решение")
  • Отсутствие документации ("все в голове у Васи")

Как кредит: сначала ускоряет, потом замедляет

Definition of Done (DoD)

Критерии готовности задачи

Пример DoD:

  • Код написан и работает
  • Покрыт unit-тестами
  • Прошел code review
  • Задеплоен на тестовый стенд
  • Протестирован QA
  • Документация обновлена
  • Product Owner принял задачу

Творческий характер программирования

Одну задачу можно решить десятками способов

Задача: поиск по миллиону товаров

Над этим работает команда из 50+ разработчиков

Почему "90% готово" - это мем

Разработка авторизации:

  • 50% - базовая форма (1 день)
  • 70% - проверка пароля (1 день)
  • 80% - сессии (1 день)
  • 85% - восстановление пароля (2 дня)
  • 90% - двухфакторная аутентификация (3 дня)
  • 95% - OAuth (5 дней)
  • 98% - edge cases (7 дней)
  • 99% - security audit (10 дней)
  • 100% - финальное тестирование (5 дней)

Итого: месяц вместо недели

Зависимость от человеческого фактора

  • Разница в продуктивности между разработчиками - в 10-100 раз
  • Senior может решить за час задачу, над которой junior мучается неделю
  • 10x developer - миф или реальность? Реальность, но редкость
Mail.ru Group: Ушел ключевой разработчик поиска. Критические алгоритмы были только в его голове. Полгода восстанавливали знания.

Почему программисты "врут" с оценками?

Они не врут - они плохо оценивают время

Пример: "Добавить авторизацию"

  • Оптимистично: 2 дня ("просто форма")
  • Реалистично: 2 недели (с нюансами)
  • Пессимистично: 1 месяц (если все пойдет не так)

Что может пойти не так с авторизацией?

  • Нужна двухфакторная аутентификация
  • Интеграция с Active Directory
  • Поддержка SSO
  • Соблюдение GDPR
  • Старая база не поддерживает новые требования
  • Баг в сторонней библиотеке
  • Поменялся API внешнего сервиса

Методы оценки в IT

  • Story Points - относительная сложность
  • Planning Poker - командная оценка
  • T-Shirt Sizing - S, M, L, XL
  • Правило 3-х точек - оптимистичная, реалистичная, пессимистичная

"Перевод с технического"

Бизнес: "Почему нужно переписывать работающий код?"
Разработчик: "Legacy, спагетти-код, нет тестов, технический долг!"
Бизнес: "Не понимаю. Клиенты могут покупать, что еще нужно?"

Как объяснять бизнесу технические потребности

  • Технический долг = кредит (проценты надо платить)
  • Рефакторинг = ремонт дома (рано или поздно рухнет)
  • Тесты = страховка (кажется лишними до аварии)
  • Мониторинг = медицинские анализы (лучше узнать раньше)

Классическая дилемма

Быстро, качественно, дешево
Выберите два
  • "Надо быстро!" → костыли, хаки, медленное развитие
  • "Надо качественно!" → идеальный код, медленный старт

Как найти баланс качества и скорости

  • Definition of Done - четкие критерии готовности
  • Правило 80/20 - 80% результата за 20% усилий
  • MVP-подход - сначала минимально работающий продукт
  • Технический долг - планировать время на "выплату"
  • Непрерывное улучшение - постоянный рефакторинг
